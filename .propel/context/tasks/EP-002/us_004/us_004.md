## Story ID
   * ID: US_004

## Story Title
   * Per-IP Rate-Limiting Middleware

## Description
  * As an API platform engineer, I want a middleware that enforces configurable per-IP request limits with standard HTTP 429 and Retry-After semantics so that the platform can mitigate abusive traffic (DDoS, credential stuffing, brute-force) while providing clear feedback to legitimate clients and observability for operations.

## Acceptance Criteria
1. Given a running service with the rate-limiting middleware enabled and the per-IP limit configured to 100 requests per minute, When an IP issues fewer than or equal to 100 requests in a rolling 60 second window, Then the middleware allows requests through normally and increments the per-IP counter without returning 429.

2. Given a running service with the rate-limiting middleware enabled and the per-IP limit configured to 100 requests per minute, When an IP exceeds the configured limit within the configured window, Then the middleware responds with HTTP 429 Too Many Requests, includes a Retry-After header indicating seconds until the next allowed request (or window reset), and returns a JSON body containing a machine-readable error code (e.g., "RATE_LIMIT_EXCEEDED") and a human-friendly message. Response headers and body format must be documented and stable.

3. Given the service is deployed behind proxies and the trusted-proxy list is configured, When a request arrives with X-Forwarded-For from a trusted proxy, Then the middleware resolves the client IP according to the trusted-proxy rules (configurable selection of the correct FFW entry) and applies the per-IP limit against that resolved client IP; otherwise it uses the connection's remote address.

4. Given per-IP limit and window values are configured via environment variables or central configuration, When the service starts, Then the middleware reads those values and enforces them. The active limits must be exposed via a diagnostics endpoint or metrics for verification (metric example: rate_limit.config.per_ip_limit and rate_limit.config.window_seconds).

5. Given the middleware uses an external backing store (e.g., Redis) for counters, When the backing store is unavailable, Then the middleware enters a documented degraded mode: uses a local in-memory best-effort counter with TTL, emits a metric (rate_limit.store.unavailable), logs the event with context, and allows requests (fail-open) while operations resolves the outage. The degraded mode and its behavior are documented.

## Edge Cases
* What happens when a client forges X-Forwarded-For to bypass limits?
  - Handling: The middleware honors X-Forwarded-For only if the immediate upstream peer is in the configured trusted proxy list. The trusted proxy configuration determines which part of the X-Forwarded-For chain is considered the client IP. If the upstream is untrusted, the middleware uses the remote socket IP. Tests must cover spoofed headers against untrusted sources.

* How does the system handle IPv6 and IPv4 normalization and CIDR-based rules?
  - Handling: IP addresses are canonicalized (strip scope identifiers, normalize IPv4-mapped IPv6), and consistent key formatting is used for storage. Whitelist/blacklist and CIDR rules are applied against normalized addresses. Unit tests cover address variants and canonicalization.
