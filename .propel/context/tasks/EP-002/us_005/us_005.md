## Story ID: us_005

## Story Title: Per-Account Exponential Backoff

## Description
* As a security-minded platform, I want exponential backoff applied to repeated failed login attempts on a per-account basis so that credential-stuffing and brute-force attacks are slowed, legitimate users are guided about wait times, and system load and fraud risk are reduced.

## Acceptance Criteria
1. Given an account has X failed login attempts within the configured window (default: 5 attempts in 15 minutes), When another failed login occurs for that account, Then the authentication endpoint responds with HTTP 429 Too Many Requests, includes a Retry-After header (in seconds) computed by the exponential backoff formula (RetryAfter = min(baseDelaySeconds * 2^(failedCount-1), maxDelaySeconds); defaults: baseDelaySeconds = 30s, maxDelaySeconds = 3600s), and returns a JSON body containing a human-friendly message and the retry_after value in seconds.

2. Given a client retries a login for an account while an active backoff is in effect, When the Retry-After period has not yet expired, Then the server must not perform password verification, must respond with HTTP 429 and a Retry-After header reflecting remaining wait time (rounded up to nearest second), and must increment or maintain a persisted failed-attempt counter atomically so backoff progression is consistent across concurrent requests.

3. Given a successful authentication for an account under backoff, When authentication succeeds, Then the system resets and clears the account's failed-attempt counter and any backoff state immediately (persisted store cleared), and subsequent login attempts are treated as new (no enforced delay).

4. Given the system is horizontally scaled (multiple app nodes), When failed attempts and backoff are evaluated, Then the backoff state (failed count, first-failure timestamp, and last-failure timestamp) is read from and written to a central, highly-available store (e.g., Redis) with atomic ops and TTLs so all nodes compute the same Retry-After and state remains consistent after node restarts.

5. Given a configured hard lock threshold is reached (configurable; default: 10 failed attempts in 15 minutes), When that threshold is met, Then the account is marked as locked (immutable flag persisted), the authentication endpoint responds with HTTP 423 Locked (or agreed project standard), includes a Retry-After header or lock-expiry TTL if applicable, and UI surfaces the locked state with next steps (password reset / contact support) per SCR-LOCKED-INFO.

## Edge Cases
* What happens when concurrent failed attempts are submitted in parallel for the same account?
  - Handle by using atomic increment operations in the central store (e.g., Redis INCR with expiry or a Lua script) to avoid race conditions. If the central store shows the threshold crossed as a result of concurrent increments, ensure only the computed backoff/lock state is used and the API returns 429/423 as appropriate. Log the concurrency event for monitoring.

* How does the system handle persistent store (Redis/DB) unavailability?
  - If the central persistence is unavailable, fall back to a safe degraded behavior: (1) apply a conservative in-memory short backoff per node (e.g., 60s) to avoid allowing unlimited attempts, (2) apply/match global per-IP rate limiting as a protective fallback, (3) surface an internal alert and metric (error rate on persistence), and (4) do NOT silently disable throttling. Document in runbook and surface error in monitoring.

* What happens when server clocks are skewed?
  - Use server (cluster) authoritative timestamps for TTL and remaining Retry-After calculations. Persist last-failure timestamps in UTC epoch seconds and compute remaining time from persisted timestamp; if timestamps are inconsistent, prefer computed TTL based on stored expiry/TTL values rather than client-supplied times. If skew causes a negative remaining time, treat as expired and allow next attempt.
