## Story ID
   * ID: us_002

## Story Title
   * Enforce Per-Account Lockout

## Description
  * As a platform security engineer, I want the authentication system to lock accounts after a configurable number of failed login attempts and mark the lock state in the persistent store, so that we can prevent credential-stuffing and brute-force attacks and protect user accounts and platform reputation.

## Acceptance Criteria
1. Given an active user account and the configured failed-attempt threshold (e.g., 5 attempts in 15 minutes), When the number of consecutive failed authentication attempts for that account reaches the threshold, Then the system sets the account state to "locked" in the persistent store, records lock metadata (timestamp, source IPs summary, reason), and subsequent authentication attempts for that account return a standardized "account locked" error response with guidance for recovery.

2. Given an account that has been locked due to failed attempts, When the configured automatic lockout duration expires, Then the system automatically clears the locked state and failed-attempt counter in the persistent store, records the unlock timestamp, and permits authentication attempts; the automatic-unlock event is auditable.

3. Given a successful authentication for an account before reaching the threshold, When the user authenticates successfully, Then the system resets the failed-attempt counter for that account in the persistent store to zero and no lock is applied.

4. Given concurrent failed authentication requests for the same account across multiple application nodes, When those requests are processed, Then the failed-attempt counter is incremented atomically in the shared persistent store (or using an atomic backing mechanism) and the lock transition occurs exactly when the configured threshold is reached; only one lock event is persisted and observable.

5. Given the persistent store used for counters or lock state is temporarily unavailable, When the authentication flow attempts to record a failed attempt or lock, Then the system follows the fail-safe policy: (a) apply conservative access control (deny authentication if account state cannot be determined safely), (b) persist events to a durable retry queue for reconciliation when the store returns, and (c) emit alerts/metrics indicating persistence failures for ops action.

## Edge Cases
* What happens when a user is attacked from many IPs rapidly (credential stuffing across distributed sources)?
  - Handling: Counters are maintained per-account (and per-IP in related stories). The lock decision must consider per-account counts regardless of originating IP. Monitoring/alerts and rate-limiting middleware (US_011) should detect distributed attack patterns; CAPTCHA insertion (US_012) may be triggered earlier as configured.

* How does the system handle persistent-store write failures during the exact request that would trigger a lock?
  - Handling: Treat as a safety-critical condition: do not allow a write failure to silently bypass lock enforcement. Apply conservative behavior (deny authentication for that request if state cannot be verified), record the attempt to a durable queue for reconciliation, and surface an immediate alert to SRE/ops for remediation. The behavior and SLAs for reconciliation must be documented and tested.

* What happens when a locked user completes a verified password reset flow?
  - Handling: Allow verified recovery flows to complete; upon successful, verified password reset the system will clear the locked state and reset failed-attempt counters, and an audit record must be written documenting the unlock reason (password reset) and initiator.
