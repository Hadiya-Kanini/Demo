## Story ID
   * ID: us_007

## Story Title
   * Lockout Messaging UI

## Description
  * "As an account holder who cannot access my account because it has been locked, I want a clear lockout information screen (SCR-LOCKED-INFO) that explains why my account is locked, how long the lockout will last (or the recovery steps), and offers direct links to account recovery and support, so that I can recover access with minimal confusion and reduce unnecessary support contacts."

## Acceptance Criteria
1. Given a user attempts to sign in and the account is locked, When the authentication system detects the locked state, Then the UI displays SCR-LOCKED-INFO with:
   - a clear headline ("Account temporarily locked"),
   - a concise, non-sensitive explanation,
   - either a lockout duration OR clear recovery steps,
   - prominent actions: "Account Recovery" and "Contact Support".

2. Given the SCR-LOCKED-INFO screen is presented, When the user clicks "Account Recovery", Then the user is routed to the configured recovery flow (US_010) and a telemetry event lockout_recovery_initiated is generated (anonymized, no sensitive details).

3. Given the SCR-LOCKED-INFO screen is presented for an admin-locked account, When the screen renders, Then it uses wording that indicates administrative lock and shows only the "Contact Support" action (no automated reset link), per policy to avoid self-serve in admin interventions.

4. Given the SCR-LOCKED-INFO screen must be accessible and localized, When rendered in any supported locale, Then all text is translated, elements meet WCAG AA contrast and semantics, and buttons/links have proper ARIA labels and keyboard focus order.

5. Given potential abuse of recovery endpoints, When the user requests recovery repeatedly, Then rate-limiting is enforced (per-account and per-IP) and friendly messaging is shown ("Too many requests, try again in X minutes") without exposing internal thresholds or counts.

## Edge Cases
* What happens when the user's lock state changes while the SCR-LOCKED-INFO screen is open?
  - Handling: On navigation focus or periodic (short) refresh, client calls lock state endpoint; if unlocked, show confirmation toast "Your account is now unlocked" and navigate to login; if still locked, update message accordingly.

* How does the system handle multiple simultaneous recovery attempts or abuse from a malicious actor?
  - Handling: Enforce server-side rate limits and CAPTCHAs as configured by FR-010; show generic throttled messaging and require CAPTCHA or escalation if abuse thresholds hit. Log suspicious patterns to security telemetry.

* What happens when recovery tokens expire between issuance and use?
  - Handling: When user follows an expired token, display "Link expired" with a one-click option to resend a new recovery link and instructions to contact support if repeated failures occur.
