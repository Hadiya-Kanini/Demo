## Story ID
   * ID: US_us_001

## Story Title
   * Record Failed Authentication Attempts

## Description
  * As a platform security subsystem owner, I want the authentication service to persist per-account and per-IP failed login attempts and emit telemetry metrics for each failure, so that we can detect credential-stuffing and brute-force attacks, drive rate-limiting and lockout decisions, and monitor attack trends for alerting and forensic analysis.

## Acceptance Criteria
1. Given the authentication API receives a failed login attempt for account_id = A from source_ip = I, When the authentication module finishes processing the failure, Then a failed-attempt record SHALL be persisted within 2 seconds containing: account_id, timestamp (UTC), source_ip, authentication_method, failure_reason, request_id (if provided), and a TTL or sliding window marker (e.g., expiry for 15-minute window).

2. Given a failed login attempt occurs, When the failure is processed, Then the system SHALL increment telemetry counters within 5 seconds:
   - auth_failed_total{account_id=A, source_ip=I, method=M, result="failure"}
   - auth_failed_by_ip_total{source_ip=I, result="failure"}
   The metric increments must be monotonic counters compatible with the monitoring stack (Prometheus-style or equivalent).

3. Given a successful login for account_id = A, When the success is processed, Then the persisted per-account failed-attempt counter/state for account A SHALL be cleared (or a clear marker persisted) and a metric auth_failed_reset_total{account_id=A} SHALL be emitted so downstream lockout systems can reset enforcement state.

4. Given duplicate or retried failure events with the same request_id or within deduplication window, When the persistence/telemetry handler receives them, Then the system SHALL ensure idempotent recording (no duplicate persisted logical records) and SHALL avoid double-incrementing telemetry counters for the same logical failure.

5. Given the primary persistence store is unavailable, When a failed-attempt must be recorded, Then the system SHALL enqueue the event in a durable retry queue (or in-process with persistence fallback) and attempt up to 3 retries with exponential backoff; if all retries fail, the system SHALL:
   - respond to the authentication flow without blocking user response,
   - emit a telemetry event auth_failed_persistence_error with diagnostic labels,
   - ensure metrics eventual consistency by retrying export/backfill when persistence recovers.

## Edge Cases
* What happens when a high volume of failures from the same NATed IP occurs?
  - Persist per-IP counters with TTL and record per-account counters independently. Add IP aggregation metadata to the record. Note that downstream decision logic must consider shared-IP heuristics (handled in rate-limiter story). Ensure metrics include both per-account and per-IP labels for flexible querying.

* How does the system handle missing, forged, or multiple X-Forwarded-For headers?
  - Use the configured trusted-proxy header policy to select the source IP. Persist a field ip_source indicating where the IP was derived from (e.g., 'trusted_xff', 'connection', 'untrusted_xff', 'unknown'). If IP is unverifiable, persist source_ip='unknown' and emit metrics with source_ip='unknown' so monitoring and downstream logic can treat these events conservatively.

* What happens if telemetry or persistence export repeatedly fails?
  - Buffer events locally with bounded capacity and persist to a durable backlog for later backfill. Emit health/error telemetry (auth_failed_persistence_error, auth_telemetry_export_error). Implement alerts when error metrics cross configured thresholds. Do not block authentication responses while ensuring eventual consistency/backfill.
