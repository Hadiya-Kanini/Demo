## Story ID
   * ID: US_us_011

## Story Title
   * Monitoring Metrics & Alerts for Attack Indicators

## Description
  * As a security engineer, I want metrics instrumented for failed authentication attempts, account lockouts, and IP blocks and dashboards plus alert rules for suspected attacks so that operations and security teams can detect, investigate, and respond to credential-stuffing and brute-force attacks quickly and reduce account compromise and downtime.

## Acceptance Criteria
1. Given the authentication system is active, When any login fails, Then the system emits a metric "auth.failed_attempts" with tags account_id, ip, outcome_reason, and service_name to the observability pipeline within 5 seconds.

2. Given repeated failed attempts occur, When per-account or per-IP thresholds are breached (configurable defaults: 5 failed attempts / 15 minutes per account; 100 req/min per IP), Then the system emits "auth.lockout" or "auth.ip_block" events, updates lock-state in the persistence layer, and increments corresponding aggregated metrics so dashboards reflect the event within 30 seconds.

3. Given metrics and events are available in the observability backend, When defined alert rules are evaluated (examples: sudden 5x increase in failed_attempts rate sustained for 2 minutes; >10 distinct accounts targeted by same IP within 10 minutes), Then alerts are created and routed to PagerDuty, the #security-alerts Slack channel, and email distribution list, and alerts include contextual metadata (top affected accounts, source IPs, recent auth log links).

4. Given the dashboards are provisioned, When an analyst loads the "Attack Indicators" dashboard, Then they can view real-time graphs and historical trends (1h, 24h, 7d, 90d) for failed attempts, lockouts, ip_blocks, unique attacking IPs, and top-targeted accounts; filters for time window, account, and IP must work and charts must update within 5 seconds for live data.

5. Given a controlled load test or simulated attack run by QA, When the test executes against staging, Then instrumentation, dashboards, and alerting produce expected metrics/events/alerts per the test plan and the results are captured in a validation report confirming correct end-to-end behavior.

## Edge Cases
* What happens when there is a legitimate mass action causing spikes (boundary condition)?
  - Handling: Correlate failed attempts with recent password-reset or bulk email notifications. Alert severity is reduced (INFO -> WARNING) if >=50% failed attempts correlate with completed password-reset events or known maintenance windows. Add quick suppression control in dashboard for known scheduled events with audit log.

* How does the system handle metrics pipeline outages (error scenario)?
  - Handling: Implement an in-process bounded buffer (configurable size, default 10k events) and local short-term persistence for failed metrics with TTL 10 minutes. Emit a health metric "auth.metrics_buffer_utilization" and trigger an operational alert if utilization >70%. Authentication flow remains unaffected (do not block auth flows), and buffered metrics are flushed when pipeline recovers; any unrecoverable buffered data is logged to durable storage for post-mortem.

* What happens with IP churn / large NAT pools causing noisy IP indicators?
  - Handling: Enrich IPs with ASN, geo, and proxy detection; apply adaptive thresholds for known NAT/cloud provider ranges and include proxy/ASN tag in metrics. Present "likely-proxy" classification on dashboard and route those alerts to lower-severity channels unless additional signals (credential-stuffing across accounts) are present.
