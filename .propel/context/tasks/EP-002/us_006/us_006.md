## Story Title
   * CAPTCHA Trigger Integration

## Description
  * As a security engineer or backend developer, I want configurable CAPTCHA challenge insertion points in the login flow and backend hooks for the UI, so that we can mitigate credential-stuffing and brute-force attacks while providing the frontend with reliable data to render and verify CAPTCHA challenges.

## Acceptance Criteria
1. Given failed-login tracking and rate-limit counters are operational, When an account or IP exceeds the configured CAPTCHA threshold (e.g., 5 failed attempts in 15 minutes by default), Then the authentication API must:
   - return an HTTP 200/401 with a JSON payload containing captcha_required: true,
   - include provider metadata (provider: "recaptcha_v2" | "recaptcha_v3" | "hcaptcha"), site_key/client_token, challenge_endpoint (or challenge_id), and ttl_seconds,
   - emit an audit event (auth.captcha.challenge.created) with username, ip, reason, and correlation_id for monitoring.

2. Given the frontend needs pre-flight information to render a CAPTCHA, When it calls the backend hook /auth/captcha/challenge with attempt context, Then the backend returns a documented challenge payload: { challenge_id, provider, site_key_or_client_token, ttl_seconds, correlation_id } and sets appropriate cache-control headers; the challenge must be single-use or have a short TTL as configured.

3. Given the team updates CAPTCHA thresholds and windows in configuration and restarts the auth service, When new login attempts are processed, Then the system enforces the updated values (tests validate: lowering threshold immediately increases captcha_required responses; raising threshold reduces them) and configuration keys are documented (e.g., AUTH_CAPTCHA_THRESHOLD_ACCOUNT, AUTH_CAPTCHA_THRESHOLD_IP, AUTH_CAPTCHA_WINDOW_SECONDS).

4. Given both IP-based and account-based thresholds are exceeded for a single request, When processing the request, Then the service must:
   - require a CAPTCHA,
   - return response metadata indicating which threshold(s) triggered the challenge and the precedence used (priority: account|ip|both),
   - not immediately lock the account solely because thresholds were exceeded; account lockout must follow configured post-CAPTCHA failure rules (e.g., lock after N captcha failures).

## Edge Cases
* What happens when the CAPTCHA provider is unreachable?
  - System handling: Fail safe by rejecting the login attempt with a clear transient error message ("Authentication temporarily unavailable — please try again later"), increment internal backoff counters for the actor/IP, emit high-severity alert to monitoring, and (optionally configurable) escalate to stricter rate-limiting/temporary lock for that actor if attack indicators persist.

* How does the system handle repeated CAPTCHA verification failures?
  - System handling: Maintain captcha_failure_count per account/IP; if count exceeds configured captcha_failure_lock_threshold within window, apply account lockout policy (with unlock flow). Log all verification attempts and provide the UI with remaining attempts before lockout.

* What about users behind shared NAT/large CDNs causing false positives?
  - System handling: Use heuristic: if many distinct accounts from same IP exceed low thresholds, prefer inserting CAPTCHA for those accounts rather than locking; include the heuristic decision in observability fields; allow operator override via config to tune IP vs account sensitivity.
