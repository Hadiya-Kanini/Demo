## Story ID
   * ID: us_012

## Story Title
   * Rate-Limiter Fallback & Error Handling

## Description
  * As a Platform Operator, I want configurable rate-limiter fallback behavior (fail-open / fail-closed) and automated alerts when the rate-limit store is unavailable, so that the system maintains predictable, auditable behavior during outages and operators are notified to investigate and remediate.

## Acceptance Criteria
1. Given the rate-limit store is healthy, When requests are processed, Then rate limiting behaves according to configured quotas and no fallback behavior is triggered.

2. Given the rate-limit store is unreachable and the system configuration is set to "fail-open", When an incoming request arrives, Then the request is allowed to proceed; the system emits a metric named rate_limit_store_unavailable with severity=warning, logs an event reason="store_unreachable:fail-open", increments a local emergency-allow counter, and triggers alert rule RL_STORE_DOWN in the observability system within 60 seconds.

3. Given the rate-limit store is unreachable and the system configuration is set to "fail-closed", When an incoming request arrives, Then the system responds with HTTP 503 Service Unavailable and a Retry-After header set to fallback.retry_after_seconds, returns a standardized JSON error body { "error":"rate-limiter-unavailable", "mode":"fail-closed" }, logs an event reason="store_unreachable:fail-closed", emits metric rate_limit_store_unavailable with severity=critical, and the request is blocked.

4. Given the rate-limit store becomes reachable again after an outage, When connectivity is restored, Then the system stops emitting fallback-mode metrics, runs an automated reconciliation routine that syncs local fallback counters to the central store within 5 minutes, records a reconciliation-complete metric, and produces an audit log entry summarizing actions taken and any requests that were allowed/blocked during the outage.

5. Given a configuration change to the fallback mode via central config or admin API, When the change is validated and applied, Then the new mode takes effect across service instances within 60 seconds, invalid configurations are rejected with a clear error and do not change behavior, and an audit entry is created with who changed the setting, previous value, new value, and timestamp.

## Edge Cases
* What happens when the rate-limit store intermittently flaps (connectivity alternates rapidly)?
  - Handling: The system detects flapping using a 30s sliding window and a configurable flap threshold. If the failure rate exceeds the threshold, the node enters fallback mode and emits "rate_limit_store_flapping". Re-evaluation occurs every 15s and normal operation resumes only when success rate stabilizes above the healthy threshold for 60s. Operators receive an escalation if flapping persists beyond a configurable duration (default 5 minutes).

* How does the system handle partial failures where only some shards/replicas are unavailable?
  - Handling: The system requires a quorum (configurable, default = majority) to consider the store healthy. If quorum is not met, the node applies fallback behavior locally, tags emitted metrics with replica/shard IDs, and exposes diagnostic endpoints (health/replica-status) for operators to identify affected shards.

* What happens when a prolonged outage causes local fallback counters to grow large and risk incorrect throttling after reconciliation?
  - Handling: Local counters have a TTL (configurable, default 30 minutes). For outages longer than TTL, the system transitions to a conservative degraded mode (configurable policy; default: escalate and prefer fail-closed if security critical). On recovery, reconciliation uses exponential smoothing and rate-limited backfill to avoid sudden behavior changes. A reconciliation report is generated and stored for audit.
