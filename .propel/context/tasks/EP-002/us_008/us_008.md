## Story ID
* ID: us_008

## Story Title
* Configurable Protection Thresholds

## Description
* As a security administrator, I want to configure runtime thresholds for per-IP and per-account rate limits, lock durations, CAPTCHA insertion rules, and perform temporary admin overrides so that we can rapidly tune protection against credential-stuffing and brute-force attacks without code deploys and minimize false positives for legitimate users.

## Acceptance Criteria
1. Given default protection settings are in place and I am an authenticated admin, When I update threshold values (per-IP requests/minute, per-account failed attempts threshold, lock duration, CAPTCHA insertion parameters) via the admin configuration UI or API, Then:
   - Inputs are validated (type, min/max ranges), 
   - Changes are persisted to the central configuration store, 
   - Changes propagate to all active application instances within 60 seconds (or documented SLA), 
   - An auditable entry is created containing admin identity, timestamp, previous values, new values, and optional rationale.

2. Given a configured per-account lock threshold (for example: 5 failed attempts within 15 minutes), When a user exceeds that threshold, Then:
   - The user's account is marked "locked" for the configured lock duration,
   - Login attempts return the standardized lock message (SCR-LOCKED-INFO) and prevent authentication,
   - An alert/metric increments (for monitoring dashboards) indicating a lockout event.

3. Given configured per-IP rate-limiting and CAPTCHA rules, When an IP or account meets the CAPTCHA insertion criteria (e.g., failed attempts exceed CAPTCHA threshold or rapid requests from same IP), Then:
   - The login UI (SCR-LOGIN) prompts for CAPTCHA verification before credential validation proceeds,
   - CAPTCHA verification success allows the normal auth flow to continue; failure enforces backoff or rejection per configuration,
   - CAPTCHA behavior can be disabled for users with accessibility exemptions via an approved override path (logged and auditable).

4. Given an admin applies a temporary override (scope: single account, IP, or global) via the admin UI/API, When the override is active, Then:
   - The override takes immediate precedence over global thresholds for its scope,
   - Overrides support optional expiry and manual revoke,
   - All override actions are logged with admin identity, scope, start/expiry times, and reason,
   - Overrides are visible in admin UI with status (active/expired/revoked).

5. Given invalid configuration input (negative numbers, non-integer, values exceeding safe maximum), When the admin attempts to save the configuration, Then:
   - The system rejects the change with clear validation messages and recommended safe ranges,
   - No partial or invalid configuration is applied,
   - The attempted change is recorded in the admin audit log as a failed update attempt.

## Edge Cases
* What happens when propagation to all application instances fails (partial rollout or network partition)?
  - Handling: The central config store marks change as "pending" until confirmed by instances. Instances that can't hot-reload continue to use last-known-good settings. The admin UI displays per-instance propagation status and provides retry/backoff controls. Monitoring emits an alert if >10% of instances did not apply change within SLA.

* How does the system handle extreme boundary inputs (very large thresholds, zero, or negative)?
  - Handling: Server-side validation enforces min/max safe values. Values below min or non-sensical inputs are rejected. Values above max are either rejected or capped depending on policy; admin receives a confirmation/warning when submitting values near or above recommended limits.

* How does the system behave if the configuration persistence layer (EP-DATA) is unavailable during save?
  - Handling: Saves fail atomically with a clear error message. The system writes a local audit/trouble record and queues the change for retry (if configured). Admins are prevented from assuming active changes until persistence succeeds; monitoring/ops are alerted.
