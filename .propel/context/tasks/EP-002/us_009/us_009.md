## Story ID
   * ID: US_009

## Story Title
   * Integration Tests for Throttling & Lockouts

## Description
  * As a Security QA engineer, I want integration tests that simulate realistic attack patterns (credential stuffing, rapid IP bursts, distributed attempts) so that account lockouts, per-IP and per-account rate-limiting, exponential backoff, and CAPTCHA insertion points are validated and regressions prevented.

## Acceptance Criteria
1. Given the system is deployed with the configured protection defaults (example defaults: lock threshold = 5 failed attempts in 15 minutes; per-IP = 100 requests/min), When an automated test performs 6 consecutive failed login attempts for the same account within 15 minutes, Then the system marks the account as locked, returns the configured locked response (UI message or API code), persists lock state, and the test validates the persisted state.

2. Given the system is deployed, When an automated test issues 200 authentication-related requests within one minute from the same IP, Then the per-IP rate limiter rejects requests above the permitted threshold with 429 (or equivalent), the test confirms throttled response codes, and emitted metrics show rate-limit events.

3. Given protection mechanisms are active, When an automated test simulates credential-stuffing (many distinct IPs attempting logins against one account), Then account-level exponential backoff is applied, CAPTCHA insertion is triggered at the configured threshold, and tests verify that subsequent requests require CAPTCHA resolution or are delayed/blocked as specified.

4. Given concurrent high-volume mixed success/failure attempts across multiple accounts and IPs, When integration load tests run with concurrency (e.g., 500 concurrent virtual users), Then lockout and rate-limiting decisions remain deterministic (no lost increments or duplicate locks), persisted counters reflect accurate totals within acceptable variance, and no race conditions allow bypass of protections.

5. Given a simulated persistence/cache outage (e.g., Redis down), When attack-pattern tests run, Then the system performs the documented failover behavior (stricter temporary limits or safe-deny), logs an operational alert, and tests assert that either protections continue in degraded mode or the system fails safely with clear operational signals.

## Edge Cases
* What happens when attackers rotate IPs rapidly (distributed attack)?  
  - Tests will simulate a botnet pattern (many IPs with low-rate attempts) and assert that account-level protections (lockout, backoff, CAPTCHA) are triggered based on aggregated per-account signals even when per-IP thresholds are not exceeded; expected behavior documented and validated.

* How does the system handle persistence store unavailability (Redis/cache outage)?  
  - Tests will simulate store failures and verify fallback behavior: either an in-memory short window counter with stricter thresholds or default-deny posture. If no fallback is implemented, the test must surface the failure and mark the behavior as a blocker.

* What happens when legitimate users hit thresholds (false positives) at boundary conditions?  
  - Tests include recovery flows: successful CAPTCHA solve or correct credential after cooldown should clear backoff and unlock the account per policy; tests validate that recovery works within the configured cooldown window and that user-visible messaging is accurate.
