## Story ID
   * ID: US_003

## Story Title
   * Admin Unlock & Forensics Endpoint

## Description
  * As an administrator, I want APIs to view failed-attempt history, unlock accounts, and export forensic data with audit logs so that I can investigate security incidents, restore legitimate user access quickly, and produce tamper-evident evidence for compliance and incident response.

## Acceptance Criteria
1. Given an authenticated administrator with the required RBAC scope, When they request failed-attempt history for accountId with filters and pagination, Then the system returns HTTP 200 with a paginated payload that includes: items (timestamp, sourceIP, userAgent, authMethod, outcome, reason), totalCount, page, pageSize, and a request-id header; results respect time-range and IP filters and are returned within 2 seconds for typical queries.

2. Given an authenticated administrator with unlock privileges, When they POST /admin/accounts/{accountId}/unlock with { reason, override=false }, Then the system atomically clears the account lock (if locked), resets or adjusts per-account backoff counters per configured policy, returns HTTP 200 with updated account status and timestamp, and writes an immutable audit log entry containing actorId, actorRole, timestamp, accountId, action=UNLOCK, reason, requestId, and outcome.

3. Given an authenticated administrator with forensics/export privileges, When they POST /admin/forensics/export with validated filters (accounts[], startTime, endTime, eventTypes[], format), Then the system validates against retention/PII policies, enqueues an export job and returns HTTP 202 with exportJobId and statusEndpoint, and when the job completes places a signed, time-limited download URL in secure storage (default TTL 24 hours), includes a manifest (recordCount, checksum, appliedMasks, omittedRecordsNote) and creates audit log entries for requested, completed, and downloaded events.

4. Given any call to these admin endpoints by an unauthorized user, When the call is received, Then the system returns HTTP 401 or 403 (as appropriate), exposes no sensitive payload data, increments an internal denied-admin-attempt metric, writes a denied-attempt audit record (actor if available, endpoint, source IP, timestamp), and triggers an alert if denial rate exceeds configured threshold.

5. Given a request that would include purged records due to data retention, When the export is processed, Then the resulting export excludes purged records, the manifest clearly states the count of omitted records with reason "purged_by_retention_policy", and an audit log entry records that the export was partial due to retention constraints.

## Edge Cases
* What happens when exporting a dataset larger than configured synchronous threshold?
  - Handling: The API validates export size estimate and, if above threshold, rejects synchronous processing and enqueues an asynchronous streaming job. The response is HTTP 202 with exportJobId and a status endpoint. The job writes chunked files to secure object storage with checksums; the UI/API provides a manifest and a signed download link(s). Documentation specifies chunk size, TTL, and recommended filters.

* How does the system handle an unlock request for an account that is not locked or already unlocked?
  - Handling: If account does not exist -> HTTP 404. If account exists but not locked -> HTTP 409 with a clear business message and no state-change; an audit record is still written indicating attempted unlock and outcome=NO_OP. If caller included override=true and policy allows, respond per policy and record override in audit.

* What happens if the audit logging subsystem (write to audit store) fails during a state-changing operation (unlock/export lifecycle)?
  - Handling: System must follow a documented fail-safe policy:
    - Preferred option A (recommended): Use transactional outbox or write-ahead audit append before state-change; if audit write fails, reject operation with HTTP 503 and do not change state. If not possible, then:
    - Option B: Perform state change, mark audit status = PENDING, schedule guaranteed retries, and return HTTP 202 with note "action applied; audit logging delayed". In all cases emit immediate alert to security operations and include requestId for reconciliation.
