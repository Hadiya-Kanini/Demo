## Story ID
   * ID Format: US_<unique seq num>
   * Story ID: us_028

## Story Title
   * Implement Login Redirects for Roles

## Description
  * As an authenticated user, I want to be redirected after login to a role-specific dashboard/URI based on role claims contained in my issued token, so that I land immediately on the most relevant experience and reduce manual navigation.

## Acceptance Criteria
  * Given a successful authentication that issues a valid token containing a single known role claim, When the login completes, Then the response MUST include the role-specific redirect URI (JSON field "post_login_redirect") that matches the configured mapping for that role, and the system MUST set the token (Set-Cookie or response body) so clients can use it to navigate.

  * Given a successful authentication where the issued token contains multiple role claims, When the login completes, Then the system MUST resolve a single redirect target using the configured role-priority list (highest priority wins) and return that role's redirect URI in "post_login_redirect".

  * Given a login request that includes a client-supplied return_to (deep-link) parameter, When return_to is an internal path and the authenticated user's roles permit access to that path, Then the system MUST return return_to as the "post_login_redirect". If return_to points to an external host or the user lacks permission, Then the system MUST ignore return_to and fall back to the role-mapped redirect URI.

  * Given a successful authentication where the role claim maps to an external absolute URL, When the login completes, Then the system MUST only return that external redirect if it is present on the configured external-redirect whitelist; otherwise the system MUST return a configured safe default internal dashboard URI and create a security audit log entry recording the blocked redirect attempt.

  * Given an authentication attempt that fails or the token is invalid/expired, When the login is not successful, Then the system MUST NOT include a "post_login_redirect" field in the response, MUST return the appropriate error status (401/400/403/423 as applicable), and MUST NOT reveal role-specific or URI-mapping details in the error payload.

## Edge Cases
   * What happens when token role claim is missing or malformed?
     - Handling: System MUST treat this as "no role" and return the configured default dashboard URI (e.g., /dashboard) in "post_login_redirect". An audit event with debug-level context should record the missing/malformed claim for investigation; metrics should increment an "auth.unmapped_role" counter.

   * How does system handle an unrecognized role value or a role that has no configured mapping?
     - Handling: The system MUST fall back to the default dashboard URI, increment an internal metric for unmapped roles, and create a traceable log entry for product/ops to update role-to-URI mappings. No descriptive error should reach the client.

   * What happens when return_to is present but points to an external host (potential open-redirect)?
     - Handling: The system MUST validate return_to against an internal-only allowlist and an approved external whitelist. If return_to fails validation, ignore the parameter and return the role-mapped redirect; log the attempt as a security event and increment a "security.blocked_open_redirect" metric.

Notes / Implementation Guidance (non-mandatory, for developer handoff)
- API contract: POST /auth/login returns 200 with JSON { token: "<jwt or session>", expires_at: "<iso>", post_login_redirect: "<internal path or allowed external URL>" } for API clients. For browser-form flows, implement an HTTP 302 to the resolved URI if the request indicates a browser session (e.g., Accept: text/html) while still setting the authentication cookie.
- Role-to-URI mapping and role-priority list MUST be configurable via environment/feature config (do not hard-code in app code) and should be backed by a strongly-typed config object and automated unit tests.
- Security: enforce open-redirect checks (whitelist), validate that return_to is internal or approved, and ensure the redirect URIs are not user-controlled without validation.
- Observability: emit metrics for redirects by role, unmapped roles, blocked external redirect attempts, and include structured audit logs for blocked/ignored return_to occurrences.
- Tests: include unit tests for mapping resolution, integration tests for API response shape, and end-to-end tests (happy path + edge cases).
