## Story ID
   * ID Format: US_<unique seq num>
   * Value: us_029

## Story Title
   * Test Role Enforcement on Protected Endpoints

## Description
  * As a security/test engineer, I want automated tests that validate RBAC enforcement across sample protected API endpoints (including expected 403 responses and role changes mid-session), so that we can detect regressions in authorization, confirm audit/logging of denials, and ensure role changes are enforced consistently in production.

## Acceptance Criteria
  * Given a user with role 'viewer' and a valid access token, When they call GET /admin/resource (requires 'admin'), Then the API responds 403 Forbidden with a non-enumerating error message, and an audit log entry is created recording the denied access event (user id, endpoint, required role, timestamp).
  * Given a user with role 'editor' and a valid access token, When they call POST /content (requires 'editor'), Then the API responds 200 OK (or 201 Created as applicable) and no authorization denial is logged.
  * Given a user who authenticates as 'editor' and receives an access token, And an administrator removes the 'editor' role from that user (role change applied server-side) before the next request, When the user retries the same editor-only request using the same token, Then the API responds 403 Forbidden and an audit log entry records the role-change enforcement (includes prior token id or jti if available).
  * Given the system supports both token-embedded roles and server-side role lookups, When automated tests run, Then they include two explicit test variants: (a) verify current production policy (server-side lookup => immediate enforcement after role change), and (b) a documented compatibility test that asserts the alternative behavior (token-embedded-only => access persists until token expiry). The test run must clearly mark which behavior is expected for the current environment.
  * Given authorization checks are performed, When an access denial occurs, Then the response must not leak user-existence information (non-enumerating message) and the API must return consistent HTTP status codes (403 for role/permission denial, 401 for authentication failure).

## Edge Cases
   * What happens when the role/authorization service is temporarily unavailable?
     - Handling: Tests must verify fail-closed behavior per product policy. If the policy is "deny on uncertainty", the automated test asserts the API returns 503 Service Unavailable or 403 (as specified by configuration) and that an incident/audit log entry is created. The test should also assert that the response includes a Retry-After header if applicable.
   * How does system handle a token lacking roles claim or containing malformed role data?
     - Handling: Tests must verify the system falls back to server-side role lookup/introspection. If no valid role mapping exists, the system must respond 403 and create an audit log entry indicating missing/invalid role claim. The test should assert no sensitive info is returned in the error body.
   * What happens when a role change occurs concurrently with an in-flight request?
     - Handling: Tests simulate a race condition and must assert the system's deterministic policy (defined in project security design): either allow the in-flight request to complete with prior privileges or enforce the change immediately. The chosen policy must be codified in the test assertions and verified for consistency across repeated runs.
