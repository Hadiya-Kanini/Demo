## Story Title
   * Load & Performance Tests for Rate-Limiting

## Description
  * As a Performance/QA Engineer, I want to run load and performance test scenarios against the rate-limiting middleware so that we can validate correctness, measure latency and failure behavior under high concurrency, and ensure the system degrades gracefully under component failure.

## Acceptance Criteria
1. Given the rate-limiting middleware deployed with default configured thresholds (per-IP = 100 requests/min; account lock threshold = 5 failed attempts in 15 minutes), When a controlled test generates 200 req/min from a single IP and 10 failed login attempts for the same account within 5 minutes, Then:
   - Requests exceeding per-IP allowance receive HTTP 429 with Retry-After header.
   - Account is marked locked after 5 failed attempts and subsequent auth attempts return HTTP 423 Locked.
   - Legitimate requests within limits continue to succeed.
   - Test report includes raw request traces, success/429/423 counts, and evidence of lock state persistence.

2. Given the application is horizontally scaled (multiple instances) and rate-limits use Redis as central store, When an incremental ramp test reaches 5,000 concurrent connections and sustains peak load for 10 minutes, Then:
   - End-to-end p95 latency <= 500ms.
   - Middleware-added p95 latency <= 200ms.
   - Error rate for allowed requests <= 0.5%.
   - Load test scripts, aggregated percentile metrics, and raw logs are stored in the test artifact bundle.

3. Given observability (Prometheus metrics & Grafana dashboards) is configured, When rate-limit events occur during tests, Then:
   - Metrics rate_limit.hits, rate_limit.throttled, account.locked_count, and rate_limit.fallback_active are emitted and reflect observed events within ±5% accuracy.
   - Dashboards update within 60 seconds of metric emission.
   - Alerts are triggered for fallback activation and high throttling rates per configured alerting rules.

4. Given Redis becomes unavailable during a test window, When middleware loses connection to Redis, Then:
   - Middleware switches to an in-memory fallback with conservative per-instance limits (documented behavior).
   - Requests continue to be processed (no crashes); when local limits are exceeded middleware returns HTTP 429.
   - rate_limit.fallback_active metric = 1 and an alert is emitted; fallback behavior is logged with explanatory context.
   - Test report documents differences in throttling behavior compared to normal operation.

5. Given CAPTCHA insertion thresholds are configured, When simulated credential-stuffing meets CAPTCHA criteria, Then:
   - Middleware responds with a challenge indicator (HTTP 403 or 429 with header X-CAPTCHA-Required: true and X-CAPTCHA-Token).
   - captcha.challenge_count metric increments.
   - Test trace shows request -> challenge response and includes the challenge token for verification.
