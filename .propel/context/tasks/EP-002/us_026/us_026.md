## Story ID
   * ID Format: US_<unique seq num>
   * ID: us_026

## Story Title
   * Include Roles in Token Claims

## Description
  * As a system administrator/developer, I want tokens issued by the authentication service to include a canonical "roles" claim mapped from the user's assigned roles, so that downstream services can perform authorization checks without an additional user lookup.

## Acceptance Criteria
  * Given an authenticated user with one or more roles persisted in the user data store, When the authentication service issues an access token (login, refresh), Then the returned JWT MUST include a "roles" claim that is a JSON array of canonical role identifiers (strings) matching the published mapping document; the API response MUST include the token and expiry metadata as existing contract requires.
  * Given an authenticated user with no roles assigned, When a token is issued, Then the token MUST include "roles": [] (an empty array) in the claim set rather than omitting the claim.
  * Given a service account or non-human identity mapped to service-level roles, When a token is issued, Then the "roles" claim MUST include the mapped service role identifiers as defined in the mapping document (docs/auth/roles-to-claim.md) and the API spec MUST show examples.
  * Given existing tokens that do not contain a "roles" claim, When token verification is performed after this change is deployed, Then verification MUST continue to accept those pre-existing tokens until their configured expiry to preserve backwards compatibility; newly issued tokens MUST include the "roles" claim.
  * Given the change is implemented, When final artifacts are published, Then there MUST be a committed Roles-to-Claim mapping document at docs/auth/roles-to-claim.md, an updated OpenAPI /auth/login response schema showing the "roles" claim, and unit/integration tests that assert token contents for representative user and service identities.

## Edge Cases
   * What happens when a user has an extremely large number of roles (boundary condition)?
     - Handling: Enforce a configurable max roles-per-token (default 50). If the number of canonicalized roles exceeds the limit, the token issuance SHOULD fail with a 500 and log the incident for remediation; alternatively the service may prioritize and include the top-N roles (document choice), but the chosen behavior MUST be documented in docs/auth/roles-to-claim.md and approved by product/security.
   * How does the system handle role names with special or non-ASCII characters (error scenario)?
     - Handling: During mapping, role names MUST be converted to canonical role identifiers (ASCII, lower-case, alphanumeric with dash/underscore). Any role assignment that cannot be canonicalized MUST be omitted from the token claim and recorded in an audit log; administrators MUST be notified per audit retention policy.
   * What happens when a user's roles change immediately after token issuance (timing/consistency)?
     - Handling: Tokens are immutable once issued and remain valid until expiry. Immediate revocation is out-of-scope for this story; the mapping document MUST note recommended mitigation strategies (short token lifetimes, revocation list / token blacklist, and session invalidation endpoints) and reference dependent stories that implement revocation.

Notes / Implementation Guidance (non-normative)
- Tokens MUST include only canonical role identifiers (no display names, no PII).
- The mapping document (docs/auth/roles-to-claim.md) MUST include:
  - table of internal role (DB) -> canonical claim value,
  - examples of token payloads,
  - maximum roles-per-token and max claim size limits,
  - recommended approach for immediate role revocation.
- Tests required: unit tests for mapping logic, integration tests asserting JWT payloads for:
  - user with single role,
  - user with multiple roles,
  - user with no roles,
  - service account mapping,
  - backward-compat verification of old tokens without "roles".
- Keep story scoped to issuance and documentation; revocation mechanics and UI/admin consoles are out-of-scope and should be separate stories.
