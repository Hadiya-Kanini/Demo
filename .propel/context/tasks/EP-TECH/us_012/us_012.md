## Story Title
   * Implement backups and PITR for primary database

## Description
  * "As a platform engineer, I want automated backups and point-in-time recovery configured so that data can be restored within defined RTO/RPO targets."

## Acceptance Criteria
  * Given the production primary database is provisioned and reachable, When the backup IaC and automation are applied, Then automated full and incremental/PITR backups must run according to the configured schedule and show successful completion in backup logs for the past 7 days.
  * Given PITR is enabled and retention window configured, When a requested restore to a point-in-time within that window is executed to a non-production staging environment, Then the restore completes within the agreed RTO (<= 1 hour) and the recovered data is validated to the expected point-in-time (RPO <= 15 minutes) using checksum and sample-record verification.
  * Given backup artifacts are produced, When they are persisted, Then all backup files are encrypted at rest and in transit (KMS-managed keys), are replicated to a secondary region/storage class for DR, and access to those artifacts is restricted by least-privilege IAM roles and audited.
  * Given backup automation is in place, When a backup or PITR operation fails, exceeds expected duration, or a retention policy action runs, Then the monitoring system raises alerts to the on-call channel, creates a ticket in the incident tracker, records retriable failure attempts, and documents the failure in an audit trail.
  * Given the retention lifecycle policy is configured, When backups age beyond their retention thresholds, Then the system archives or deletes backups per policy (e.g., daily: 30 days; monthly: 12 months) and writes deletion/archive events to the audit log for compliance review.
  * Given runbooks and DR playbooks are required, When a restore test is performed, Then the runbook is followed end-to-end, the restore is validated by the QA/ops owner, and any runbook gaps discovered are updated in the operational runbook repository.

## Edge Cases
   * What happens when the database is larger than estimated and a full snapshot exceeds the backup window?
     - Handling: Backup automation should support incremental/PITR fallback, schedule full snapshot in a low-traffic window, and surface a capacity warning. If snapshot still cannot complete, mark the operation failed, trigger escalation alerts, and record mitigation steps in the incident ticket as per runbook.
   * How does the system handle KMS key unavailability, rotation errors, or revoked keys during backup/restore?
     - Handling: Backups must not be stored unencrypted. On KMS errors the pipeline fails with a high-severity alert, uses the configured secondary key only if pre-authorized, and includes step-by-step procedures in the runbook for emergency key recovery or rotation rollback.
   * What happens when the restore target environment has schema drift, missing extensions, or incompatible configuration?
     - Handling: Pre-restore validation detects schema/extension mismatches. If mismatch found, pause restore, produce a validation report, and either (a) apply controlled migration plan (if authorized), or (b) restore into an isolated sandbox for forensic analysis. All decisions and actions are logged.

Notes/Implementation Guidance (non-UI):
- Implementation should be IaC-driven (Terraform/CloudFormation/ARM) with pipeline automation to apply backup configuration and PITR settings across environments.
- Use the platform KMS (referenced in us_009) for encryption and ensure IAM policies restrict access to backup artifacts and restore operations.
- Implement monitoring dashboards and alerting (e.g., metrics for backup duration, success/failure rate, restore durations) and hook alerts into PagerDuty/Slack and the incident management system.
- Deliverables include: IaC modules for backup & PITR, automated test demonstrating a successful restore to staging, runbook with step-by-step restore and escalation procedures, and retention lifecycle automation with audit logs.
