## Story Title
   * Operational Runbooks: Key Rotation, Hashing, Rollback

## Description
  * As a Platform/SRE engineer, I want published and verified operational runbooks for key rotation, hashing tuning, and rollback procedures with verification steps, so that rotations and tuning can be performed safely, reproducibly, and rolled back quickly with measurable verification and minimal customer impact.

## Acceptance Criteria
  * Given repository write access, a staging environment, and CI that can publish docs, When a PR containing the runbooks is submitted and approved, Then the runbooks are published to the central documentation location (/docs/runbooks/us_012-ops-runbooks), include metadata (owner, contact, version, change-log), and have an attached verification checklist that can be signed off.

  * Given an active signing key in the secret manager and staging traffic, When the Key Rotation runbook is followed (including pre-checks, rotation steps, verification commands, and post-checks), Then the rotation completes without downtime: new key used for signing, verification succeeds across services, the old key is disabled/marked deprecated, monitoring shows no more than 0.1% increase in authentication errors over a 10 minute window, and the runbook checklist is completed and recorded.

  * Given a reproducible benchmark harness and baseline metrics captured, When the Hashing Tuning runbook is executed (apply new hashing param, run benchmark suite, collect metrics), Then the runbook records pre/post metrics for p50/p95 latency, throughput, and CPU utilization and ensures p95 latency regression is <=20% and throughput degradation is <=10% on the test harness; if thresholds are exceeded, the runbook prescribes rollback steps which are documented and executable.

  * Given a simulated failure or detected degradation in staging, When the Rollback runbook is executed, Then services are restored to the prior configuration or key within the documented rollback window, smoke tests pass, the error rate returns to baseline within 2 minutes, and the rollback is recorded with responsible operator and timestamps.

  * Given completed runbook executions, When verification steps are satisfied, Then an evidence artifact (runbook checklist with timestamps, CI run logs, monitoring snapshots, and PR references) is stored alongside the runbook and linked from the ops index for auditability.

## Edge Cases
   * What happens when the secret manager is unavailable during rotation?
     - Handling: The runbook includes a pre-rotation availability check. If unavailable, the runbook instructs to abort the rotation, capture diagnostic logs, alert the on-call SRE, and schedule a retry. No partial key updates are attempted. The abort path and required notifications are documented.

   * How does the system handle a partial rotation where some services have the new key and others do not?
     - Handling: The runbook mandates a canary-first rollout and defines verification thresholds for canary success. If canary verification fails, the runbook prescribes immediate rollback on canary nodes and halts propagation. It includes commands to reconcile key state across services and steps to re-run rotation after remediation.

   * What if hashing tuning causes CPU or memory exhaustion on production-like nodes?
     - Handling: The runbook requires that tuning be performed on dedicated test nodes with resource limits. It defines abort thresholds (e.g., CPU >85% for >2 minutes) that trigger automatic stop of the tuning workflow and an immediate rollback to previous hashing parameters. It instructs to open an incident, collect heap/cpu profiles, and update the runbook with observed limits.
