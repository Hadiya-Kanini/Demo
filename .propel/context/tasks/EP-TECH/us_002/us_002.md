## Story ID
   * ID Format: us_002

## Story Title
   * Provide local dev environment with containers

## Description
  * As a developer, I want reproducible local development containers and a quickstart script so that I can run and test the application locally with minimal setup.

## Acceptance Criteria
  * Given a developer has cloned the repository and has Docker Engine (and Docker Compose or Docker CLI with compose support) available on their machine, When they run ./scripts/dev-quickstart.sh (or the documented quickstart command), Then the script pulls/builds required container images, starts all required services (application, primary DB, cache, and mocked external adapters) and reports all service health endpoints as healthy within 10 minutes.
  * Given the repository provides environment variable templates (.env.example or .env.local.example), When the quickstart script runs and needs secrets, Then it creates a .env.local from the template with developer-safe placeholders, attempts to fetch development secrets from the configured dev secret provider (if configured), and fails with an actionable, non-sensitive error message if secret retrieval is not possible — no secrets are committed to source control.
  * Given containers are running, When the quickstart performs database setup, Then database migrations complete successfully, a smoke-test query returns expected results, and a minimal set of seed data required for local feature testing is present.
  * Given the local dev environment is operational, When ./scripts/run-local-tests.sh (or documented test command) is executed, Then unit and integration tests that depend on local containers run and generate test reports (JUnit or similar) under ./reports/tests-local/ and exit with pass/fail status suitable for CI.
  * Given potential environment problems (missing Docker, port conflict, low memory), When the quickstart script detects such conditions, Then it prints clear remediation steps, references the developer README/troubleshooting runbook, and exits with a non-zero status to indicate failure.

## Edge Cases
   * What happens when Docker is not installed or running?
     - Handling: The quickstart script performs a pre-flight check for Docker and Compose. If missing or Docker daemon is not reachable, it prints a clear error with step-by-step remediation (install links, start Docker instructions) and a pointer to an alternative remote dev environment if available. It exits non-zero without making partial changes.
   * How does system handle port conflicts (e.g., 5432 already in use)?
     - Handling: The quickstart detects port binding failures during startup. It reports the conflict clearly, suggests an alternative port, provides instructions to update .env.local or the compose override file, and offers to automatically pick and apply an ephemeral port for the session while warning how to persist changes.
   * What happens when secret retrieval from the dev secret provider fails (network/auth error)?
     - Handling: The script logs a non-sensitive error (no secret values), falls back to writing safe placeholder values to .env.local, instructs the developer how to obtain real dev secrets (link to runbook), and ensures the missing secret state is communicated and does not leave secrets in repo or logs.

Notes:
- Deliverables for this story include: dev-quickstart script(s) in ./scripts/, Dockerfile(s) and docker-compose.override.yml (or compose v2 equivalents) for local development, .env.example and .env.local.example templates, DB migration/seed commands integrated into quickstart, a local test runner command, pre-flight checks with clear error messaging, and updates to the developer README with a Quickstart section and troubleshooting runbook links.
- This story focuses on developer experience and reproducibility; it intentionally uses local container orchestration suitable for macOS/Linux/Windows WSL. Production tooling (Kubernetes manifests, infra IaC) remain in EP-TECH but separate stories will map to those deliverables.
