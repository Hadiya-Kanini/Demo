## Story Title
   * Local Development Environment Setup

## Description
  * As a developer (new or existing), I want a documented and automated local development environment bootstrap procedure and scripts, so that I can run the service and its required dependencies locally reliably, validate changes quickly, and onboard without manual environment configuration (Estimate: 2 pts).

## Acceptance Criteria
  * Given a developer has cloned the project repository and has Docker (or Podman), Docker Compose (or equivalent), and git installed, When they run ./scripts/bootstrap-dev.sh, Then the script:
    - pulls or builds required images,
    - starts containers for the app, local DB, and cache,
    - waits for each service health endpoint to return healthy,
    - prints connection details (host/port) and instructions to run integration/smoke tests, and
    - completes successfully within 5 minutes on a typical developer machine.

  * Given the repository lacks a .env.local file, When the bootstrap script runs, Then it creates .env.local from .env.example (without secrets), prints commands and links to the README for adding secrets/keys, and exits with a success code after starting services.

  * Given services have already been started by a prior bootstrap run, When the developer runs the bootstrap script again, Then the script performs idempotent operations: reuses healthy containers, restarts unhealthy ones, does not create duplicate containers, and logs an explicit message about reused containers.

  * Given a missing prerequisite (e.g., Docker not installed) or an unsupported tool version, When the developer runs the bootstrap script, Then the script fails fast with an informative error message, non-zero exit code, and direct link to the developer README with installation/upgrade steps.

  * Given the developer wants to stop and optionally clean up the local environment, When they run ./scripts/teardown-dev.sh (or make dev-down [--clean]), Then containers started by the bootstrap are stopped; with --clean ephemeral volumes and networks created by the script are removed, while persistent developer data is preserved unless explicitly requested to be deleted.

## Edge Cases
   * What happens when required local ports are already in use?
     - The bootstrap script checks for port availability before starting containers. If a conflict is detected, it lists the conflicting ports and processes, suggests alternative port mappings (documented in README), and either prompts the user to continue with alternate ports or exits with an actionable error. The script must not silently remap ports without developer confirmation.

   * How does the system handle long or failed dependency startups (e.g., DB schema migration fails)?
     - The bootstrap implements health-check retries with exponential backoff and a configurable timeout (default 300s). If a dependency fails to become healthy within the timeout, the script collects logs for the failed container(s), writes a diagnostics file to ./tmp/dev-diagnostics.tar.gz, prints remediation steps (e.g., run migration manually), and exits with a non-zero code.

   * What if the developer has no network access to Docker registries?
     - The bootstrap script detects image pull failures and surfaces a clear message, listing the failed images and registry endpoints. It then suggests offline options documented in the README (pre-pull images, use a provided local image archive, or start in a mocked-only mode). The script provides a --offline flag to run a minimal mock-mode that starts only lightweight local mocks for development without external pulls.
