## Story ID

   * Story ID: us_003

## Story Title
   * CI: Build and Unit Test Pipeline

## Description
  * As a developer on the authentication service team, I want a CI stage that builds the service and runs unit tests automatically for every pull request, so that regressions are prevented and PRs are gated by automated verification before code review and merge.

## Acceptance Criteria
  * Given a pull request targeting any branch in the repository, When the PR is created or updated, Then the CI pipeline triggers a "build" job and a "unit test" job and reports pass/fail status back to the PR.
  * Given that the repository contains the CI pipeline configuration file (.ci/config.yml or equivalent), When the pipeline runs the build job, Then a reproducible build artifact (binary, packaged artifact, or other appropriate deliverable) is produced and stored in the CI artifact store with a unique build identifier and accessible link from the CI run.
  * Given the unit tests are defined in the repository, When the unit test job executes, Then all tests run, test results are published in JUnit-compatible format (or other CI-recognized test-report format), and any test failures cause the pipeline and PR status check to fail (preventing merge).
  * Given the CI runner baseline, When a typical change triggers the pipeline, Then the build + unit test stage completes within 15 minutes under normal CI runner capacity, and both logs and test reports are retrievable from the CI UI/storage for debugging.
  * Given repeated pushes to the same PR (commits or branch updates), When the PR is updated, Then the pipeline re-runs automatically and the latest run's status is reflected on the PR.

## Edge Cases
   * What happens when transient external dependency fetches fail?
     - The pipeline will retry the dependency-fetch step automatically up to 2 times. If retries fail, the pipeline fails with clear, actionable logs indicating external dependency outage and the failure reason is surfaced in the PR status details so maintainers can triage.
   * How does the system handle flaky tests or intermittent test failures?
     - The pipeline must preserve test run history and artifacts (both passing and failing runs) so flakiness can be diagnosed. A single automated rerun may be allowed for suspected transient failures, but any pass after a rerun must record that a rerun occurred and create an annotation or ticket recommending remediation. Flaky tests must not be silently treated as passing.
   * What happens when the CI configuration file is missing or malformed?
     - The system must surface a blocked CI check on the PR with a clear error message and a link to the developer README and a sample CI configuration. The pipeline must not proceed silently and must provide guidance to add or fix the CI config.
