## Story Title
   * Provision IaC for dev environment

## Description
  * As an SRE/DevOps engineer, I want Infrastructure-as-Code (Terraform + Kubernetes manifests and provider-specific modules) to provision the development environment so that CI/CD can deploy repeatably, environments are consistent across developers, and platform-level baseline controls (networking, namespaces, RBAC, secrets integration) are enforced.

## Acceptance Criteria
  * Given a fresh developer repository clone with infra/ directory and CI credentials configured, When the CI pipeline executes the IaC job (terraform init/plan/apply and kubectl apply for k8s artifacts), Then the dev environment resources (network, cluster, dev namespace, minimal ingress, sample deployment) are created successfully and reachable from the CI validation step (sample app responds to health-check endpoint).
  * Given the IaC code in infra/dev is applied once successfully, When the same IaC apply is executed again without changes, Then the second apply is idempotent (terraform plan reports no changes and kubectl apply reports 0 changes) and exit code indicates success.
  * Given secrets are required by workloads in the dev environment, When the IaC provisions secret integration, Then secrets are provided via the configured secrets manager (e.g., AWS Secrets Manager or HashiCorp Vault), no plaintext secrets exist in the repo or pipeline logs, and a workload can retrieve the secret during CI validation.
  * Given credentials or permissions are missing/insufficient in the CI service account, When the IaC plan/apply runs, Then the pipeline fails with a clear, actionable error referencing the missing permission(s) and links to the runbook that documents required IAM roles/permissions.
  * Given the dev environment exists, When an automated destroy/teardown job is executed, Then the environment is removed cleanly (terraform destroy completes, cluster namespaces removed) and cloud resources with the dev tag are no longer present; the destroy step completes within the agreed dev timebox.

## Edge Cases
   * What happens when cloud credentials are missing or invalid?
     - The IaC job must fail fast with provider-auth errors captured. The pipeline must surface a concise message indicating which credentials/permissions are missing, link to the IAM/runbook for remediation, and optionally open a ticket for the platform team.
   * How does system handle partial apply failures (e.g., resource quota or API rate limits)?
     - The pipeline must capture state and logs, mark the run as failed, trigger the remediation playbook: run terraform refresh and plan to determine drift, attempt safe rollback/destroy for partially created resources, and notify the on-call SRE with the failure details and suggested remediation steps in the runbook.
   * What happens when the external secrets manager is unreachable during provisioning or at runtime?
     - IaC should allow creation of infra resources that do not strictly require immediate secret provisioning (idempotent); workloads that require secrets must fail CI validation with explicit error messaging. The pipeline should implement retry logic with exponential backoff; after retries are exhausted, the system should alert and create an incident per the runbook.
