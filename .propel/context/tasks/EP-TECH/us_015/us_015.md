## Story ID
   * ID Format: US_<unique seq num>
   * ID: us_015

## Story Title
   * Prototype WAF and API protection rules

## Description
  * As a security engineer, I want a WAF baseline and test rules deployed to staging, so that common web threats are mitigated and rules can be tuned before production

## Acceptance Criteria
  * Given a staging environment provisioned and reachable, When the WAF baseline configuration is applied via the IaC/CI pipeline, Then the WAF is active for staging and returns 200/403 as configured for allowed/blocked requests and the deployment is logged in CI with a successful status
  * Given an automated attack test harness containing OWASP Top 10 and common API abuse patterns, When the harness runs against staging, Then the WAF blocks or sanitizes malicious requests, logs events with rule IDs and request metadata, and test results are stored in the artifact store for review
  * Given the system's functional test suite for core APIs, When the suite runs against staging with the WAF enabled, Then legitimate flows succeed and false-positive rate remains <= 1% of test cases; any failed legitimate case produces a documented alert and a ticket for rule tuning
  * Given telemetry and alerting are configured, When a high-severity WAF rule triggers, Then an immediate alert is sent to the security/ops channel containing rule ID, sample request context, timestamp and link to the runbook/playbook for triage
  * Given performance baselines for staging traffic, When the WAF processes representative load, Then the measured median added latency per request is <= 100ms and 95th percentile latency remains within agreed NFR limits; any violation creates an escalation notification and a performance investigation ticket

## Edge Cases
   * What happens when a rule causes false positives?
     - Detect: automated functional tests and synthetic monitors flag regressions.
     - Handle: switch offending rule to "monitor only" or disable via CI-backed config roll (audit log), raise ticket for rule refinement, run targeted synthetic tests, then re-enable with adjusted rule conditions.
   * How does system handle WAF misconfiguration or deployment failure?
     - Detect: CI pipeline plan/apply validation fails or health checks fail post-deploy.
     - Handle: pipeline auto-rolls back to previously validated configuration, notifies on-call engineer, and attaches plan/apply failure logs. Manual remediation steps and rollback playbook included in runbook.
   * What if API contract changes cause rule mismatches (legit requests flagged)?
     - Detect: CI gating includes a WAF impact dry-run for API contract change PRs; staging smoke tests detect mismatches.
     - Handle: require PR to include WAF rule updates or exemptions; if missed, automated rollback and security review are triggered; create a temporary exemption policy with expiry and work item to refactor rule.
