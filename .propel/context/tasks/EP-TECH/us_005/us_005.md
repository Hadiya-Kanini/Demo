## Story ID: us_005

## Story Title: Implement CI pipeline: build, unit tests, artifact storage

## Description:
  * As a CI owner, I want a pipeline that builds code, runs unit tests, and stores artifacts so that changes are validated and reproducible artifacts are available.

## Acceptance Criteria (Given/When/Then format, 3-5 criteria)
  1. Given a new commit or pull request into the repository, When the CI pipeline is triggered, Then the pipeline completes a build stage that compiles the project or produces the expected build artifacts without compilation errors and records build logs.
  2. Given the repository contains unit tests, When the CI pipeline runs the test stage, Then all unit tests are executed and results are recorded; the pipeline enforces a configurable minimum code coverage threshold and fails if the threshold is not met.
  3. Given a successful build, When the artifact publishing stage runs, Then artifacts are uploaded to the configured artifact repository (e.g., Artifactory/Nexus/S3), are immutably versioned and tagged with commit SHA and pipeline metadata, and a checksum is stored and verified post-upload.
  4. Given a failing build or failing unit tests, When the pipeline detects the failure, Then the pipeline fails immediately, posts clear failure reasons and relevant logs in the CI UI, blocks merge of the PR, and does not publish artifacts.
  5. Given artifacts are published, When a subsequent job or user attempts to retrieve an artifact by tag (commit SHA) or pipeline metadata, Then the artifact is retrievable and integrity-verified within defined retention and access-control policies.

## Edge Cases
   * What happens when intermittent/flaky tests cause non-deterministic failures?
     - The pipeline implements a flakiness detection pattern: provide an option to automatically re-run failing test stage once (configurable); log re-run attempts and label tests that frequently flip; surface flakiness metrics to the team so flaky tests are fixed. Even with re-run, any failing tests after allowed re-runs cause pipeline failure.
   * How does system handle artifact repository outages or transient network errors during publish?
     - The publish step performs a limited series of retries with exponential backoff. If uploads ultimately fail, the pipeline marks the run as failed, archives the build output in the CI runner workspace with a retention window for manual retrieval, and sends an automated notification to the CI owners with details and remediation steps.
   * What happens when an uploaded artifact fails checksum verification or exceeds size limits?
     - The publish step computes and verifies checksums (e.g., SHA256). On mismatch, the artifact is rejected and the pipeline fails with diagnostics. For artifacts exceeding configured size limits, the publish step fails with a clear error and guidance to split artifacts or increase allowed limits via change control.
