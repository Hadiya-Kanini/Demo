## Story Title
   * Deploy-to-Staging Pipeline with Rollback

## Description
  * As a Release Engineer, I want a deploy-to-staging continuous delivery pipeline that deploys build artifacts to the staging environment, runs automated smoke tests, and performs an automated rollback on failures so that the staging environment remains healthy, developers receive fast feedback, and releases do not leave staging in a partially-broken state.

## Acceptance Criteria
  * Given an approved build artifact exists in the artifact registry, When the deploy-to-staging pipeline is triggered (on merge to main or manual promotion), Then the pipeline deploys the artifact to the configured staging namespace/cluster and completes a deployment readiness check within 10 minutes.
  * Given the deployment readiness checks succeed, When post-deploy smoke tests execute, Then all defined critical smoke tests pass and the pipeline marks the deployment as successful and records deployment metadata (artifact id, commit sha, deployment id, timestamp).
  * Given any smoke test or readiness check fails, When the pipeline detects the failure, Then it automatically initiates a rollback to the previous stable release, verifies the rollback succeeded, and posts a rollback and failure summary to the configured alerting channel and deployment logs.
  * Given the automated rollback cannot be performed (e.g., irreversible DB migration or infra errors), When the rollback step fails, Then the pipeline marks the deployment FAILED, creates an incident ticket with logs and context, notifies the on-call team, and surfaces the runbook for manual remediation.
  * Given any pipeline execution, When steps run (deploy, smoke tests, rollback), Then the pipeline emits structured logs and metrics (step status, durations, error codes) to centralized observability and exposes a retriable run/execution id for troubleshooting and reruns.

## Edge Cases
   * What happens when a smoke test is flaky (intermittent failure)?
     - Handling: Implement a retry policy (2 retries with exponential backoff). If retries produce inconsistent results, mark the test as "flaky" in the test report and fail the deployment only if retries ultimately fail. Record full test histories for triage.
   * How does the system handle irreversible database migrations included in the deployment?
     - Handling: The pipeline must detect migration manifests flagged as irreversible and, in such cases, block automated rollback. The pipeline will abort automated promotion, create an incident, and notify on-call with explicit instructions to follow the manual migration rollback/runbook. The pipeline will also require a manual approval gate for deployments containing irreversible migrations.
   * What happens when the artifact is missing or corrupted in the artifact registry at deployment time?
     - Handling: The pipeline aborts before touching staging, marks the run FAILED, emits diagnostics (missing checksum/error), notifies the responsible team, and logs remediation steps. A manual redeploy after artifact re-publish is required.
