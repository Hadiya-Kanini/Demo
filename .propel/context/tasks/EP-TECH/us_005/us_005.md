## Story Title
   * CI: SAST and Dependency Scanning Integration

## Description
  * As a Platform Engineer, I want static application security testing (SAST) and dependency vulnerability scanning integrated into the CI pipeline with gating rules for high-severity findings, so that code merged to protected branches meets the project's security baseline and vulnerabilities are detected and managed early.

## Acceptance Criteria
  * Given a merge request (MR) targeting a protected branch, When the CI pipeline runs, Then the pipeline executes both SAST and dependency scanning stages and attaches machine-readable scan reports (e.g., SARIF) and a human-readable summary to the MR; artifacts are retained for at least 30 days.
  * Given scan results contain at least one vulnerability classified as High or Critical (or CVSS >= 7.0), When the gating rules evaluate the results, Then the pipeline fails the MR build, the MR is blocked from merge, an automated comment is posted summarizing failing findings with remediation references, and a tracking ticket is created and assigned to the MR author or security owner.
  * Given scan results contain only Medium or Low findings (no High/Critical), When gating rules evaluate, Then the pipeline continues to success, a summary comment is posted to the MR with findings and recommended remediation timeline, and an aggregated tracking issue is created for Medium findings if count >= 3 or if any Medium has a known exploit.
  * Given the scheduled weekly dependency-scan job executes, When the scan completes, Then a full dependency vulnerability report and an SBOM are produced and stored as CI artifacts, and any High/Critical findings trigger an immediate alert/ticket to security and the responsible team.
  * Given scan results are provided, When developers or security reviewers view results, Then results must be available in a standard machine-readable format and a human-readable summary; the scan system must support a documented suppression/false-positive workflow with audit-trail entries showing who approved and expiry dates.

## Edge Cases
   * What happens when the external scanner service is unavailable?
     - The pipeline marks the scan as "ERROR - scanner unavailable", attaches diagnostics, notifies on-call, and requires explicit security approver override for merge. The pipeline must not auto-pass security gates when scan status cannot be validated.
   * How does system handle CI timeouts or very large repos that cause scans to exceed job limits?
     - The pipeline attempts an incremental or module-scoped scan; if timeouts persist, the job fails with a clear error, artifacts/logs are attached, and a ticket is created for platform to tune scan configuration or increase CI timeouts.
   * What happens when a finding is disputed as a false positive?
     - A documented suppression workflow must allow developers to request suppression with evidence; suppression requires security approver sign-off, is recorded with justification and expiry, and suppressed items remain visible in reports and audit logs.
