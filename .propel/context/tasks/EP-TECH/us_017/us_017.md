## Story ID
   * ID Format: US_<unique seq num>
   * ID: us_017

## Story Title
   * Add CI/CD rollback automation and SRE tests

## Description
  * "As an SRE, I want deployment rollback automation and validation tests so that unhealthy deployments are automatically rolled back and availability is preserved."

## Acceptance Criteria
  * Given a new deployment to an environment (dev/stage/prod) with automated health checks configured, When the deployment health SLI falls below the configured threshold (e.g., 3 consecutive failed readiness/liveness probes within 2 minutes OR a predefined error-rate threshold), Then the CI/CD system automatically triggers a rollback to the last known-good revision, records a rollback event in the deployment history, and stops the failing rollout.
  * Given an automatic rollback is initiated, When the rollback completes, Then an automated post-rollback validation suite (smoke tests + critical integration tests) is executed automatically; these tests must pass and results must be recorded in CI logs; the environment must reach healthy state within RTO targets (<= 10 minutes for staging, <= 15 minutes for production).
  * Given an automatic rollback attempt fails due to orchestration/apply errors or missing artifacts, When the failure occurs, Then the system opens an incident in the designated incident system, notifies the on-call SREs via configured alert channels (pager/Slack/email), and places the deployment in a quarantine mode (traffic drained or scaled down) with forensic logs and retry metadata preserved.
  * Given pipeline-level changes are proposed (PR modifying deployment/rollback logic), When the pipeline runs, Then the SRE validation tests (unit + integration + smoke for rollback behavior) are executed in a non-production environment and the PR is blocked from merging if critical SRE tests fail; test artifacts must be archived and linked to the PR.
  * Given runbooks are updated, When on-call invokes the manual rollback playbook, Then the manual procedure reverts to the same last-known-good state, invokes the same post-rollback validation tests, and produces an incident timeline consistent with automated rollback (including timestamps, operator IDs, and test results).

## Edge Cases
   * What happens when rollback target revision is missing?
     - Handling: Abort automatic rollback, create incident/ticket with priority escalation, notify on-call SREs, preserve current state and logs, and present remediation options in incident (e.g., use immutable snapshot restore or follow manual restore runbook). Document the missing-artifact condition in CI logs for audit.
   * How does system handle deployment flapping (rapid healthy/unhealthy toggling)?
     - Handling: Implement debounce/aggregation logic in health check evaluation (e.g., require sustained failure condition over configured window), do not rollback on transient failures; if oscillation continues, route traffic away using circuit-breaker or scaled replica adjustment and escalate to SREs for investigation.
   * How to respond when CI/CD control plane is degraded during rollback?
     - Handling: Attempt secondary control plane or failover path if configured; if none available, create an immediate incident with failover instructions, block further deploys, and instruct on-call to run manual rollback using documented safe steps; ensure all attempts and errors are logged for postmortem.
