## Story ID
   * ID Format: US_<unique seq num>
   * ID: us_004

## Story Title
   * CI Security Gates (SAST & Dependency Scans)

## Description
  * As a DevOps Engineer, I want SAST and dependency vulnerability scanning integrated into the CI PR pipeline so that pull requests are automatically blocked for critical findings and developers receive actionable feedback early in the development lifecycle.

## Acceptance Criteria
  * Given a feature branch with a PR opened, When the CI pipeline executes, Then SAST and dependency vulnerability scans run automatically and a summary report (with counts by severity, top findings, and links to full scan artifacts) is posted to the PR and to CI logs.
  * Given a PR with at least one critical severity finding (scanner "Critical" label or CVSS >= 9.0), When the scans complete, Then the PR status is set to failing and the merge is blocked until findings are remediated or an approved exception exists; the PR comment must include remediation guidance and the reference to the failing job.
  * Given non-critical findings (high/medium/low), When scans complete, Then findings are reported in the PR and CI artifacts but the PR is not blocked by default; a configuration option must exist to promote selected severities to fail for specific repos or branches.
  * Given a registered exception previously approved by security owners, When the same finding is detected in a PR, Then the pipeline recognizes the exception id and does not block the PR for that specific finding while still reporting it and linking to the approval record.
  * Given scanner service outages, timeouts, or errors during a PR pipeline run, When the CI step fails, Then the pipeline follows the defined fail-mode policy (default: fail-open with a clear PR annotation and an automated alert to platform/security owners); the policy must be configurable per-repo or per-branch to allow fail-closed behavior for high-compliance branches.

## Edge Cases
   * What happens when scanner service is down or returns intermittent errors?
     - Handling: The pipeline must detect the outage and annotate the PR with the error. Default platform policy: fail-open and create an alert/ticket for platform owners; configuration must allow opting into fail-closed for protected branches. Retries and exponential backoff are attempted within the CI step timeout window.
   * How does system handle confirmed false positives or third-party legacy code flagged as critical?
     - Handling: Provide an auditable exception workflow: developer submits an exception request (template + justification), security reviewer approves/denies, approved exceptions are stored in a central registry referenced by the pipeline, exceptions are timeboxed and require re-validation on expiry.
   * How are performance/time constraints in large or monorepos handled?
     - Handling: Support path-filtered scans and incremental scanning (only changed modules), allow background/async full scans on main/nightly builds, enforce per-job timeouts, and surface partial results with clear indicators. If incremental scanning misses cross-module issues, schedule a full scan job and warn the PR.

Notes:
- Definition of "critical" : scanner-reported "Critical" or CVSS score >= 9.0. This mapping must be implemented and documented so scanners with different severity taxonomies normalize to the platform policy.
- Implementation considerations: include selection/evaluation of SAST (e.g., semgrep/ShiftLeft/CodeQL) and dependency scanners (e.g., OWASP Dependency-Check, Snyk, GitHub Dependabot) as part of the story or reference a preceding spike (if required). The pipeline must support configurable scanners, store artifacts securely, and integrate with the central exceptions registry and alerting channels. The feature must be testable via automated tests that simulate scanner outputs for pass/fail, exception recognition, and outage conditions.
