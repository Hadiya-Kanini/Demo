## Story ID: us_004

## Story Title: CI: Integration Tests and Artifact Publishing

## Description:
  * As a CI/CD engineer, I want an integration test stage and automated publishing of build artifacts and container images to the artifact registry, so that downstream environments (staging/production) can reliably consume tested, versioned artifacts for deployment.

## Acceptance Criteria:
  * Given the build and unit test stages have completed successfully, When the CI pipeline reaches the integration stage, Then the integration test stage runs integration tests against an ephemeral test environment and produces JUnit-style test reports and logs accessible from the pipeline run.
  * Given integration tests complete successfully, When the pipeline reaches the publish step, Then the pipeline tags artifacts and container images with the git commit SHA and pipeline run ID, publishes them to the configured artifact registry (container registry and artifact storage), and returns the published artifact URIs in the pipeline summary.
  * Given integration tests fail, When the pipeline stops at the failed integration stage, Then artifacts and images are not published, the ephemeral test environment is torn down (unless DEBUG_KEEP_ENV flag is set), the pipeline run is marked failed, and links to the test reports and logs are included in the failure message.
  * Given the publish step requires credentials, When the publish step executes, Then credentials are fetched securely from the configured secret manager (no plaintext secrets in pipeline config), and on authentication/authorization failure the pipeline fails with an explicit error and retains build artifacts in CI storage for debugging.
  * Given pipeline run completes (success or failure), When ephemeral resources were provisioned for integration tests, Then resources are cleaned up automatically within the post-job cleanup step unless a debug flag is set, and cleanup status (success/failure) is recorded in the pipeline logs.

## Edge Cases:
  * What happens when intermittent/flaky tests cause non-deterministic failures?
    - The pipeline will support a single automatic rerun of the integration stage for runs that fail with pre-defined flaky/test-transient signatures. If the rerun passes, the pipeline proceeds to publish. If it fails again, treat as a hard failure and do not publish. Flaky tests must be surfaced in test reports and flagged for engineering review.
  * How does the system handle registry authentication/authorization failures or network timeouts during publish?
    - On auth/authorization errors, the pipeline fails fast with an explicit error message including registry API response. Build artifacts remain in CI temporary storage for debugging and a link is included in the failure message. Network timeouts/partial uploads cause the pipeline to fail and preserve logs and partial upload metadata; retries follow pipeline retry policy and must be explicitly visible in pipeline UI.
  * What if a required external dependency (e.g., downstream service or test DB) is unavailable during integration tests?
    - The integration stage should attempt to initialize with local mocks/stubs for missing services where possible. If core dependencies are missing and tests cannot proceed, fail fast with a clear reason, tear down ephemeral resources, and mark pipeline as failed. Test reports should indicate which dependency was missing and include health-check failure logs.

Notes/Implementation Guidance (non-normative):
- Use existing CI provider conventions (GitHub Actions/GitLab/GCP Cloud Build/etc.) as agreed by platform team; implementation detail is negotiable as long as acceptance criteria are met.
- Artifacts to publish include: built binaries (e.g., JAR/WAR), build manifests, and container images. Use immutable tags: <repo>:<commit-sha>-<build-number>.
- Secret retrieval should use configured secret manager (EP-TECH will provide integration story for secret manager retrieval) and not store credentials in pipeline YAML.
- Integration ephemeral environment should be lightweight (containerized compose, ephemeral k8s namespace, or test harness) and must include automated teardown in post-job steps.
- Provide clear documentation in repo README describing how to run integration stage locally and how to validate published artifacts via registry API.
