## Story ID

## Story Title
   * Provision IaC baseline for staging and production

## Description
  * "As an SRE, I want reusable IaC modules and resource quotas for staging and production, so that environments are provisioned consistently to meet sizing, autoscaling, security and governance standards and are repeatable across teams."

## Acceptance Criteria
  * Given the project repository scaffold and IaC module layout exist (dependency: us_001), When a developer runs the IaC "plan" target for the staging environment with environment variables populated, Then Terraform/CloudFormation/Helm plan completes without errors, outputs a manifest showing resources that conform to the agreed sizing defaults (cpu/memory limits, replica counts), and the plan contains ResourceQuota/LimitRange (K8s) or equivalent cloud quota resources for staging.

  * Given the IaC modules are run via CI pipeline (dependency: us_002), When the pipeline executes the "apply" job for the staging environment on an authorized branch, Then infrastructure is created/updated successfully, Kubernetes namespaces (or cloud equivalents) are tagged with environment and cost-center metadata, autoscaling policies (HPA/ClusterAutoscaler or cloud autoscale groups) are configured to the documented baseline, and automated smoke checks (pod readiness, service endpoint health) pass within the accepted time window.

  * Given the production IaC workspace and policy-as-code checks are enabled, When a production apply is attempted, Then pre-apply policy checks (OPA/Gatekeeper or cloud policy engine) validate IAM least-privilege, required tags, and TLS/encryption settings and block the apply if any high-severity policy violations are detected, producing actionable failure messages.

  * Given resource quota/limit defaults are applied to staging and production, When a workload exceeds the defined resource quota in staging, Then the system enforces quota behavior (pod admission blocked or evicted per policy), generates an alert to the monitoring channel, and a documented runbook provides steps to remediate (request quota increase or optimize workload).

  * Given the IaC modules for staging and production are merged to main and associated docs updated, When an on-call SRE follows the operational runbook to perform a controlled scale verification, Then autoscaling exercises (scale up/down) succeed, monitoring metrics reflect expected CPU/memory/replica changes, and the runbook is validated with a recorded checklist entry.

## Edge Cases
   * What happens when the IaC plan fails due to missing environment variables or secrets?
     - The CI pipeline must fail the plan step with clear error messages identifying missing variables/secrets. The pipeline must not attempt apply. A remediation note including how to provision missing secrets (reference to us_003 KMS/secrets story) should be surfaced in the pipeline logs.

   * How does system handle organization quota limits exceeded (cloud provider hard limit) during apply?
     - The apply must fail safely and trigger an automated rollback where applicable. The failure should create a ticket (or notify the platform channel) with cloud provider error details, and the runbook/incident playbook must include next steps to request quota increase or reduce requested capacity. No partial resources should be left in an inconsistent state; IaC modules should include idempotent cleanup steps.

   * What happens when resource quota is too low causing frequent pod evictions in staging?
     - Monitoring detects elevated OOMKilled/eviction rates and triggers an alert. The runbook indicates either temporarily increasing namespace quota via a controlled change or adjusting workload resource requests; the IaC includes parameters (with safe guardrails) to make that change and a post-change verification checklist.

Notes/Implementation Guidance (non-UI):
- Provide modular, documented Terraform modules (or equivalent) for: networking, IAM, Kubernetes cluster config (namespaces, resource quotas, limit ranges), autoscaling (HPA/cluster autoscaler), and monitoring bootstrap. Modules must accept environment-specific variables and have sensible defaults with overrides.
- Policy-as-code (OPA/Gatekeeper or cloud-native policy engine) checks must be integrated into the CI pipeline as blocking gates for production applies.
- Include automated smoke tests (HTTP/S readiness, liveness, DB connectivity) executed post-apply by the pipeline.
- Ensure tagging and metadata strategies are enforced via IaC modules and validated by policy checks.
- Deliverables include README, parameter docs, runbook for quota/scale incidents, and a checklist to validate staging→production parity.
- Acceptance criteria are written to be testable by automated pipeline runs and manual runbook verification.
