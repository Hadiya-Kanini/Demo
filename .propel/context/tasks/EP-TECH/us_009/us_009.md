## Story Title
   * Monitoring & Observability Bootstrap

## Description
  * "As a Platform Engineer, I want to deploy telemetry collectors, baseline Prometheus/Grafana dashboards and alert rules, and validate alert delivery to configured channels, so that the platform has reliable, testable observability from day one and on-call teams receive timely notifications of critical incidents."

## Acceptance Criteria (Given/When/Then format, 3-5 criteria)
  * Given a provisioned cluster and network (infrastructure task US_F complete), When Helm manifests/Helmfile/HelmRelease for Prometheus stack (prometheus-server, node-exporter, kube-state-metrics) and Grafana are applied, Then Prometheus, node-exporter, kube-state-metrics and Grafana pods are Running, Prometheus shows healthy scrape targets for cluster components, and there are no critical scrape errors in the last 10 minutes.
  * Given metrics are being scraped, When baseline Grafana dashboards and Prometheus alert rules are imported/applied, Then Grafana displays dashboards for cluster CPU, memory, pod restarts, API latency and application HTTP error rates with data populated within 5 minutes and Prometheus shows alerting rules in a 'firing' / 'pending' state when conditions are met.
  * Given alerting integrations are configured (Slack webhook and PagerDuty integration), When an alert condition is simulated (e.g., forced CPU saturation or firing a synthetic alert), Then an alert notification is delivered to the configured Slack channel and a PagerDuty incident is created within 2 minutes, and the alert includes the alert name, affected resource, severity, timestamp, and a link to the Grafana dashboard.
  * Given repository contains monitoring IaC and automation, When the monitoring bootstrap verification script/CI job runs, Then it executes an end-to-end smoke test: validates Prometheus scrape targets, confirms dashboards are loaded and show recent data, triggers a synthetic alert, verifies delivery to notification channels, and returns pass/fail with logs/artifacts for debugging.
  * Given alert rules exist, When duplicate rapid-fire alert triggers occur from the same root cause, Then deduplication/suppression settings or alertmanager routing prevents flood to notification channels (rate-limit or group by labels), and the behavior is documented in the runbook.

## Edge Cases
   * What happens when network partition prevents Prometheus from scraping targets?
     - Detection/Handling: Prometheus marks targets as down; verification script flags prolonged scrape failures (>5 minutes) as a failed validation and attaches diagnostics (scrape logs, endpoints, service IPs) to the CI artifact. Runbook includes steps to check network policies, service endpoints, CNI, and restart exporters if needed.
   * How does system handle alert delivery failures to external channels (Slack/PagerDuty)?
     - Detection/Handling: Alertmanager configured with retry/backoff and secondary route (email/fallback on-call). Verification script simulates invalid webhook and checks that Alertmanager logs retries and routes to fallback. Runbook contains rotation steps for credentials, verification checklist, and contact escalation.
   * What happens when high-cardinality metrics cause Prometheus memory/OOM issues?
     - Detection/Handling: Configure relabeling to drop high-cardinality labels at scrape time, set resource requests/limits for Prometheus, and create alerts for Prometheus memory/cpu pressure. Verification script includes a cardinality check and flags sources exceeding thresholds; runbook advises on cardinality reduction and sharding/remote_write to long-term storage.
