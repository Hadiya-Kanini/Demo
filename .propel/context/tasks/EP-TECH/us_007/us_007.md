## Story ID
   * ID Format: US_<unique seq num>
   * Story ID: us_007

## Story Title
   * Integrate SAST and dependency scanning into CI

## Description
  * "As a security engineer, I want static analysis and dependency vulnerability scans in the pipeline so that security issues are detected early and gated appropriately."

## Acceptance Criteria
  * Given a pull request to any protected branch, When the CI pipeline runs, Then a SAST scan and a dependency vulnerability scan are executed, results are attached as machine-readable artifacts (e.g., SARIF/JSON) and printable reports, and inline annotations are created on the PR for findings so developers can triage quickly.
  * Given scan results that include SAST findings classified as High or Critical severity or dependency vulnerabilities with CVSS >= 7.0 or with known exploit metadata, When the pipeline completes, Then the pipeline fails/gates the merge, a standard security ticket is automatically created (or linked) containing file/line references, CWE/CVE identifiers, and suggested remediation, and a notification is sent to the owning team and security inbox.
  * Given a merge to the main branch, When the post-merge pipeline executes, Then the same SAST and dependency scans run as part of the promotion checks; if the severity threshold is exceeded, deployment to downstream (staging/prod) environments is blocked until remediation or security approved exception is granted; scan artifacts are persisted in the CI artifact store for at least 90 days.
  * Given typical repository size (as defined in the repository README sizing guidance), When SAST and dependency scans run, Then the combined scan runtime is <= 15 minutes in the CI environment for baseline builds; if runtime exceeds this threshold, the pipeline marks the scan as timed out, logs diagnostic telemetry, and creates a remediation ticket indicating the need for scan optimization (e.g., incremental scan or path filters).
  * Given a developer adds a suppression annotation or a repository suppression policy file with justification, When the pipeline processes results, Then suppressed findings are excluded from gating decisions (per suppression policy), the suppression and justification are recorded in scan metadata for audit, and suppressions older than 30 days require explicit Security Team re-approval to remain excluded.

## Edge Cases
   * What happens when external vulnerability DB or hosted scanning service is unavailable?
     - Handling: CI marks the scan step as errored and, by default, fails the build (fail-closed). An automated alert is raised to the Security and Platform teams (PagerDuty/Slack), the pipeline emits a standardized outage message with timestamp and tool version, and an auditable manual override process exists for Security Engineers to temporarily allow merges with recorded justification.
   * How does system handle extremely large monorepos or long-running scans?
     - Handling: Implement incremental scanning and caching (using CI cache keys) and path-based filters. If incremental scanning cannot be applied and runtime > 15 minutes, the pipeline fails the scan step with a "scan-timeout" status and generates a tech-debt ticket detailing candidate optimizations (split scans, incremental, containerizing scanner). The failure is visible to the PR author with diagnostic logs.
   * How are false positives or scan regressions handled?
     - Handling: Developers can submit a suppression with justification via a defined repository suppression file or inline annotation. Suppressions are recorded in scan metadata and routed to Security for review within SLA (5 business days). If a tool regression is suspected (e.g., after an upgrade), the pipeline captures the tool version and fails the build until a compatibility investigation and a mitigation plan are in place.
