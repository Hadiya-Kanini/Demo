## Story ID: us_007

## Story Title: Deployment Templates (K8s/Helm) with Resource & Health Policies

## Description
  * As a Platform Engineer, I want Kubernetes manifests and a parameterized Helm chart that include resource requests/limits and liveness/readiness probes for staging and production, so that deployments are predictable, schedulable, observable, and validated by CI/CD pipelines to reduce runtime incidents and ease operational handoffs.

## Acceptance Criteria
  * Given a built container image and access to a conformant Kubernetes cluster (staging), When I install the provided Helm chart with staging values, Then:
    - helm lint passes,
    - helm template produces valid manifests,
    - pods reach Running and Ready within 2 minutes,
    - readiness and liveness probes report success.
  * Given the chart installed with production values, When deployed to a production-like cluster, Then:
    - pods are scheduled (no Pending due to resource constraints),
    - resource requests and limits are present and match the values.yaml provided for production,
    - PodDisruptionBudget (if included) and probe behavior meet production defaults.
  * Given the chart templates are run through CI validation (helm lint && kubeval or equivalent), When CI runs these checks, Then:
    - the templates pass schema validation and linting,
    - health probes and resource fields are present in the generated manifests,
    - any violations are surfaced as CI failures with actionable messages.
  * Given a developer overrides probe endpoints or resource quantities via values.yaml or --set, When the chart is rendered, Then:
    - the rendered manifests reflect the overrides,
    - the deployment behaves accordingly (probes checked against the provided endpoints).
  * Given an unhealthy container (readiness probe fails), When Kubernetes lifecycle events occur, Then:
    - pod readiness status is NotReady,
    - liveness probe behavior triggers restarts per liveness settings,
    - pod events and container logs contain probe failure information for debugging.

## Edge Cases
   * What happens when requested resources exceed node capacity?
     - The helm chart README will include guidance and recommended maximums for both staging and production.
     - CI will run a validation step that compares requested resources to a provided cluster-capacity profile and fails with a clear message if requests are likely unschedulable.
     - The values.yaml provides conservative defaults to avoid accidental oversubscription.
   * How does system handle invalid resource units or malformed probe configuration?
     - Helm lint and kubeval (or JSON schema validation) are enforced in CI; invalid units or malformed probe config produce deterministic, descriptive failures that block merge until corrected.
     - Chart templates include robust template guards with sane defaults and explicit validation errors for misconfigurations.
   * What happens when probes are intentionally disabled?
     - Disabling probes must be explicit via a values flag (e.g., probes.enabled=false).
     - CI flags disabling probes as a policy deviation; team owners decide whether it's blocking. Documentation must explain operational risk and recommended alternatives (e.g., sidecar health check).

Notes / Implementation Guidance (non-normative):
- Provide both plain Kubernetes manifests (kustomize-friendly or templated) and a Helm 3 chart with values files for staging and production (values.staging.yaml, values.production.yaml).
- Include default health probes (HTTP GET or TCP as appropriate), configurable timeouts, initialDelaySeconds, periodSeconds and failureThreshold suitable for both environments.
- Provide recommended resource requests/limits for CPU and memory in values files and document rationale (expected memory/CPU footprints, safety margins).
- Add CI validation steps: helm lint, helm template | kubeval, and an optional scheduler-check script to detect unschedulable resource requests against provided cluster profiles.
- Include examples in README for deploying to staging and production clusters and a section on how to override resources/probes during testing.
- Ensure Helm templates include template guards and schema validation (values.schema.json) to catch misconfigurations early.
