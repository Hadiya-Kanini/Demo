## Story ID: us_016

## Story Title: Provide adapter stubs for email, MFA, and CAPTCHA

## Description
  * As a backend developer, I want adapter interfaces and in-memory stub implementations for email, MFA, and CAPTCHA, so that feature teams can integrate third-party services via well-defined contracts and run local and CI tests without calling external providers.

## Acceptance Criteria
  * Given the project scaffolding and module layout are available, When I open the repository, Then I see three interface contracts:
    - EmailAdapter with methods such as SendEmail(ctx, EmailMessage) (returns SendResult / error)
    - MfaAdapter with methods such as SendMfaChallenge(ctx, MfaRequest) and VerifyMfaCode(ctx, MfaVerifyRequest)
    - CaptchaAdapter with methods such as ValidateCaptchaToken(ctx, token) (returns boolean / error)
    All interfaces include comments describing expected semantics, error types, retry expectations, and idempotency characteristics.

  * Given a feature team runs unit tests or local development, When they import the adapter package, Then they can instantiate EmailStub, MfaStub, and CaptchaStub implementations that:
    - Are in-memory and do not make network calls
    - Support configuration for deterministic success/failure, injected latency, and pre-seeded responses
    - Are usable as drop-in replacements for production adapters in DI/factory patterns

  * Given the CI pipeline executes for PRs touching adapter code, When the pipeline runs,
    Then:
    - Adapter package compiles and unit tests for adapters run and pass
    - CI includes a scan that ensures no API keys/secrets are present in the repository (common secret patterns are flagged)
    - Tests exercise concurrent access to stubs to verify thread-safety (test asserts no race conditions)

  * Given a downstream test wants to simulate failure scenarios, When the test configures the stub via constructor params or environment variables, Then the stub produces:
    - Configurable transient errors (e.g., 50% failure rate), permanent errors, and timeout errors
    - Simulated rate-limit responses with a documented, typed error to allow caller backoff logic testing

  * Given a developer wants to implement a production adapter, When they follow the provided documentation,
    Then README contains:
    - Clear instructions for implementing the real adapter (required interfaces and example method implementations)
    - How to retrieve credentials from the platform KMS/secret store (reference to recommended story/us_010)
    - Security guidance (no secrets in repo, logging best practices, scrubbing PII)
    - Example config and CI validation checks expected for production adapter pull requests

## Edge Cases
   * What happens when a third-party provider is unavailable?
     - The stub must be able to simulate unavailability by returning a documented timeout/ServiceUnavailable error. Downstream code should receive a clear, typed error so retries/backoff can be exercised in tests. The interface includes context for cancellation/timeouts.

   * How does system handle missing or malformed configuration when initializing an adapter?
     - Adapter constructors/factory functions must perform validation and return a typed ConfigError. The CI/unit tests must include negative tests that assert initialization fails cleanly (no panics) and provide actionable error messages.

   * What happens under concurrent load or when rate limits are exceeded?
     - Stubs must be concurrency-safe and optionally enforce a configurable rate-limit simulator to emulate production constraints; when limits are hit, stubs return a documented RateLimitError so feature teams can validate retry/backoff behavior.
