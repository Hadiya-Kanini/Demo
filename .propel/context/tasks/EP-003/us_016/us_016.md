## Story Title
   * Track Failed Logins and Lockout Flag

## Description
  * As a product owner / authentication service, I want failed_attempts to increment on failed login attempts and locked_at to be set when a configurable threshold is reached so that user accounts are protected from brute-force attacks, lockout state is persisted for audit and admin recovery, and legitimate users are given predictable unlock behavior.

## Acceptance Criteria
  * Given a valid existing user who is not currently locked, When a login attempt fails due to invalid credentials, Then the user's failed_attempts value in the persistent user record is incremented by 1 atomically, persisted to the database, and the response is 401 Unauthorized with a non-enumerating generic message (e.g., "Invalid credentials").
  * Given a valid existing user whose failed_attempts equals (threshold - 1), When a login attempt fails due to invalid credentials, Then the system sets failed_attempts to the threshold value, writes locked_at = current server timestamp to the persistent user record, persists both atomically, and returns 423 Locked with a non-enumerating message (e.g., "Account temporarily locked"), and if a finite lockout duration is configured the response includes a Retry-After header with the remaining seconds.
  * Given a user whose locked_at is set and the lockout period has not expired (or is configured as admin-only unlock), When any login attempt (correct or incorrect credentials) is made, Then the system responds with 423 Locked, does not disclose whether the credentials were correct (non-enumerating), does not reset failed_attempts, and does not attempt token issuance.
  * Given a user whose locked_at was set but the configured lockout duration has expired, When a login attempt with valid credentials is made, Then authentication succeeds, failed_attempts is reset to 0, locked_at is cleared (set to NULL), and a normal successful login response with tokens is returned.
  * Given multiple concurrent failed login attempts against the same account, When those attempts are processed, Then updates to failed_attempts and locked_at are applied atomically such that the final persisted state reflects the correct incremented counter and exactly one request triggers the transition to locked_at when the threshold is reached (no lost updates or double-lock race conditions).

## Edge Cases
   * What happens when a login is attempted for a non-existent account?
     - Handling: Treat identically to invalid credentials. Do NOT create or persist failed_attempts for non-existent identities. Return 401 Unauthorized with a generic, non-enumerating message. Ensure timing differences are minimized to avoid user enumeration via response latency.
   * How does the system handle database write failures when incrementing failed_attempts or setting locked_at?
     - Handling: Log and surface the error to observability (error tracking + audit log). Return 500 Internal Server Error for the login request. Use transactional writes / retries for transient errors to avoid partial updates; ensure no inconsistent lock state (no locked_at set without corresponding failed_attempts update).
   * How are different lockout policies handled (time-based expiry vs admin-only)?
     - Handling: Support configuration flag:
       - If lockout_duration > 0: compute expiry = locked_at + duration; include Retry-After header with remaining seconds and automatically allow login post-expiry (per AC above).
       - If lockout_duration = 0 or policy = admin_only: require explicit admin unlock via admin API; do not include Retry-After header.

Notes for implementation and QA (non-normative):
- Implement persistent fields: failed_attempts (integer, default 0), locked_at (timestamp nullable). Ensure DB-level default and index on locked_at if used for queries.
- Make threshold and lockout_duration configurable via service config (env + secrets) and documented in runbook. Default threshold and durations must be approved by security owners.
- Use DB atomic update statement (e.g., UPDATE ... SET failed_attempts = failed_attempts + 1 RETURNING failed_attempts) or optimistic locking with retry to avoid race conditions in concurrent requests.
- Ensure audit logs capture lock/unlock events (who/what triggered, timestamp, source IP) for security investigations (tagged AIR/DR).
- Provide unit, integration, and concurrency tests (including simulated concurrent failures) and manual QA checklist for non-enumeration checks and header correctness.
