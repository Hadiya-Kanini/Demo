## Story ID
   * ID Format: US_<unique seq num>
   * Story ID: us_018

## Story Title
   * Implement Per-IP and Per-Account Rate Limiting

## Description
  * As a platform security engineer, I want a rate-limiter middleware that enforces per-IP and per-account policies with Retry-After headers and progressive backoff, so that the authentication endpoints are protected from brute-force and abuse while providing predictable retry guidance and observability.

## Acceptance Criteria
  * Given the auth service is running and Redis (shared counters store) is available, When a client sends requests to POST /auth/login and the client's IP exceeds the configured per-IP threshold (default: 100 requests per 10 minutes), Then the middleware returns HTTP 429 Too Many Requests with a Retry-After header (seconds) indicating the remaining window and emits a metric rate_limit.exceeded with tags {policy: per-ip, ip: <client-ip>}.

  * Given a valid account identifier (email or account id) is present in the login payload, When requests for that account exceed the configured per-account threshold (default: 5 requests per minute), Then the middleware returns HTTP 429 Too Many Requests with a Retry-After header and applies progressive backoff for subsequent violations (Retry-After doubles each violation, starting at 30 seconds, max 86400 seconds), and the middleware emits metric rate_limit.exceeded with tags {policy: per-account, account: <account-id>}.

  * Given both per-IP and per-account limits are exceeded simultaneously, When the middleware evaluates policies, Then it enforces the most restrictive policy (the one with the longer Retry-After), returns the corresponding Retry-After header and 429 response, and logs a single aggregated rate_limit.exceeded event referencing both policies.

  * Given an account is currently locked due to failed-attempt lockout (locked_at set and lock window active), When a login attempt occurs, Then the middleware short-circuits and returns HTTP 423 Locked with a standard non-enumerative body and a Retry-After header indicating remaining lock duration (in seconds) and does NOT leak which policy (rate-limit vs lockout) caused the response; metrics emitted must record lockout.hit with account tag.

  * Given the service is deployed in multiple replicas, When concurrent requests arrive that should increment counters, Then counter updates are atomic/consistent (Redis INCR/EXPIRE or Lua script used), the shared store maintains correct counts across instances, and tests verify counter behavior under concurrency (no lost increments, and Retry-After calculations are deterministic).

## Edge Cases
   * What happens when Redis (shared counters store) is unavailable?
     - Handling: Provide two configurable modes:
       1) fail-closed (default): return HTTP 503 Service Unavailable for authentication requests with Retry-After: 60 and emit rate_limit.dependency_failure metric and alert. This prevents uncontrolled load spikes.
       2) fail-open (opt-in for emergency): allow auth request to proceed, emit rate_limit.dependency_failure metric and high-severity alert, and record a best-effort local in-memory counter (non-durable) to reduce burstiness.
     - Tests: simulate Redis down and verify both modes behave as configured.

   * How does the system handle forged or proxied IPs (X-Forwarded-For)?
     - Handling: Resolve client IP using the trusted proxies list from configuration. If X-Forwarded-For exists and the immediate peer is trusted, use the left-most non-trusted IP in the header. If ambiguous or untrusted, fall back to immediate peer IP. Map unresolved/unknown IPs to a conservative 'unknown' bucket with stricter default limits to reduce abuse risk. Document configuration and test with common proxy headers.

   * How are Retry-After values calculated for progressive backoff and when they would exceed maximum?
     - Handling: Progressive backoff formula:
         retry_after_seconds = min(initial_backoff_seconds * 2^(violation_count - 1), max_backoff_seconds)
         where initial_backoff_seconds = 30, max_backoff_seconds = 86400 (24 hours).
       If retry_after_seconds equals max_backoff_seconds, emit metric rate_limit.backoff_maxed. Return Retry-After as integer seconds. Ensure that violation_count increments are stored atomically and include TTL aligned with the backoff window.

Notes and Implementation Guidance (non-normative)
- Storage/consistency: Use a single shared store (Redis) with atomic ops (INCR with EXPIRE or Lua scripts) to implement counters and backoff state to ensure correctness across replicas.
- Configurability: Expose thresholds, window durations, initial_backoff_seconds, max_backoff_seconds, trusted_proxies list, and fail-open/fail-closed behavior via configuration (env/config service).
- Observability: Emit metrics (rate_limit.hit, rate_limit.exceeded, rate_limit.backoff_maxed, rate_limit.dependency_failure, lockout.hit) with tags for policy, account, ip and sample traces. Log aggregated events without PII.
- Security & Privacy: Do not include raw credentials or PII in logs/metrics. Ensure Retry-After and response bodies do not enable user enumeration.
- Testing: Include unit tests for calculation logic, integration tests with Redis (including failure modes), concurrency tests, and contract tests validating response codes/header semantics.
