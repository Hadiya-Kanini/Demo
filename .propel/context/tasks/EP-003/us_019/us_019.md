## Story ID: us_019

## Story Title: Build Admin Unlock API with Audit Logging

## Description:
  * As an admin user (role=admin), I want a POST /admin/users/{id}/unlock endpoint that clears locked_at and failed_attempts for a user and emits an immutable audit event, so that administrators can safely restore legitimate users who were locked out while maintaining a tamper-evident trail for compliance, forensics, and monitoring.

## Acceptance Criteria:
  * Given a valid admin auth token and an existing user that is locked, When the admin sends POST /admin/users/{id}/unlock, Then the response is 200 OK, the user's locked_at is set to NULL, failed_attempts set to 0, and the response body contains { target_user_id, status: "unlocked", audit_event_id }.

  * Given a valid admin auth token and an existing user that is not locked, When the admin sends POST /admin/users/{id}/unlock, Then the response is 200 OK, the user's record is left unchanged (no negative counters or duplicate resets), and an audit event is emitted recording this no-op unlock with prior state values.

  * Given a request with missing or invalid authentication, When the caller sends POST /admin/users/{id}/unlock, Then the response is 401 Unauthorized and no changes or audit events occur; Given a request authenticated as a non-admin user, When they call the endpoint, Then the response is 403 Forbidden and no changes or audit events occur.

  * Given an unlock request for a user id that does not exist, When the admin calls POST /admin/users/{id}/unlock, Then the service responds 404 Not Found and does not emit an audit event.

  * Given any unlock request, When the operation proceeds, Then an audit record must be created containing: admin_id (actor), target_user_id, timestamp, previous_locked_at, previous_failed_attempts, action:"admin_unlock", request_id/correlation_id, and outcome; and the update + audit creation must be atomic (either both succeed or none persist). If atomicity cannot be guaranteed synchronously, API behavior and response must document eventual delivery and provide an outbox_id.

## Edge Cases:
  * What happens when two admins concurrently call unlock for the same user?
    - The system must prevent lost updates by acquiring a lock (pessimistic) or using optimistic concurrency control (version column / compare-and-swap). Both requests should succeed or one should detect no-op and still emit an audit event. The final persisted state must be locked_at = NULL and failed_attempts = 0 and failed_attempts must never become negative.

  * How does system handle audit storage failure (e.g., audit DB or event bus unreachable) during unlock?
    - Preferred approach: implement transactional outbox or an atomic DB transaction that includes an audit row. If audit persist cannot be guaranteed immediately, the API must fail and return 503 Service Unavailable to avoid silent loss of audit trail. If asynchronous outbox is implemented, respond with 202 Accepted and include outbox_id and documented SLA for delivery; system must surface unresolved outbox entries to monitoring/alerts.

  * What happens when the provided user id is malformed or an invalid identifier?
    - System returns 400 Bad Request for invalid id format and does not attempt user lookup or emit audit event.

- Negotiable: Error codes, response payload shape (audit_event_id vs outbox_id), and atomicity strategy are negotiable during grooming.

- Estimable & Small: Scope limited to one endpoint, DB update, audit write, auth check, tests; should be <= 5 story points per decomposition guidance.
- Testable: Acceptance criteria are concrete (state change, audit record, auth behavior, 4xx/5xx cases).
