## Story Title
   * Test Rate-limiter and Lockout Scenarios

## Description
  * As a QA Automation Engineer, I want automated integration tests that validate per-IP vs per-account limits, lockout thresholds, expiry, Retry-After behavior, and non-enumeration guarantees, so that we can detect regressions in abuse-mitigation logic, ensure consistent secure responses, and provide confidence before release.

## Acceptance Criteria
  * Given the authentication service configured for tests with:
      - account_failed_threshold = 5 (lock on 5th failed attempt),
      - account_lock_duration = 15 minutes,
      - ip_rate_limit = 20 requests per minute,
    When an existing account receives 5 consecutive failed login attempts from any single IP,
    Then the 6th login attempt returns HTTP 423 Locked, includes a Retry-After header indicating remaining lock duration, the user's locked_at flag is set in the datastore with expiry approximately now + 15 minutes, and subsequent successful-login attempts are rejected until expiry or an admin unlock.

  * Given the authentication service configured for tests with ip_rate_limit = 20/min,
    When a test client issues 21 failed login requests from the same IP within one minute targeting multiple accounts (including invalid emails),
    Then requests after the 20th are rejected with HTTP 429 Too Many Requests containing a Retry-After header, per-IP counters are incremented appropriately, and existing accounts targeted in the window still have their per-account failed_attempt counters incremented only for attempts that match that account.

  * Given the system's non-enumeration requirement and test config normalizing timing,
    When a test issues identical wrong-password login attempts against (A) a non-existent email and (B) an existing account with an incorrect password,
    Then both responses must be indistinguishable to an external caller: same HTTP status (401 Unauthorized), identical generic response body ("Invalid email or password" or configured message), and response times within a configured tolerance (e.g., ±100ms). No 423 Locked must be returned for the non-existent account scenario.

  * Given a locked account with locked_at set and near-expiry,
    When a login attempt is made after lock expiry (advance test clock or wait),
    Then the account is treated as unlocked: locked_at is considered expired (or cleared), failed_attempt counters are reset per policy, and a subsequent valid credential attempt succeeds (returns 200 with token), while an invalid attempt increments counters as normal.

  * Given a simulated rate-limiter datastore outage (test harness toggles availability),
    When login requests are made,
    Then the service honors the configured fallback (explicit in test config — either fail-open or fail-closed), the test asserts the fallback behavior, and the service emits an observable metric/alert; user data (failed_attempt, locked_at) is not corrupted.

## Edge Cases
   * What happens when [boundary condition]?  
     - Exactly-on-threshold behavior: The system must lock on the configured Nth failed attempt. Tests will assert the Nth attempt triggers lock state and that the response for the Nth attempt reflects the failed credential (and subsequent attempts reflect lock state).

   * How does system handle [error scenario]?  
     - Rate-limiter backend/store unavailable: Tests will simulate the outage and must validate the documented fallback behavior (configurable) is followed. Tests will also validate that an outage does not silently corrupt per-account counters and that metrics/alerts are emitted.

   * What happens when [race/concurrency] and counters update concurrently?  
     - Concurrent failed attempts from multiple clients/nodes should result in atomic increments. Tests will issue concurrent requests and assert that final failed_attempt count equals the number of attempts recorded (within tolerance for in-flight operations) and that exactly one lock event occurs once threshold reached.
