## Story ID
* ID: us_011

## Story Title
* Add Observability and Security Logging for Auth Events

## Description
* As a platform/SRE engineer, I want the authentication service to emit structured logs and metrics for authentication success, failure, and revocation events so that we can monitor health and performance, detect suspicious activity, alert on security incidents, and perform reliable forensics and audits.

## Acceptance Criteria
1. Given the authentication service receives a successful authentication request, When the authentication completes, Then the system emits a structured JSON log entry (event=auth_success) with fields: timestamp, request_id, service_name, tenant_id, hashed_user_id, client_ip, auth_method, latency_ms, token_id_hash (if token issued), and log level INFO; AND the metric auth_success_count is incremented with tags [service_name, tenant_id, auth_method], the log is delivered to central logging within 30 seconds, and the metric is visible in monitoring within 60 seconds.

2. Given the authentication service processes a failed authentication attempt, When the attempt is rejected, Then the system emits a structured JSON log entry (event=auth_failure) with fields: timestamp, request_id, service_name, tenant_id, hashed_user_id_or_anonymous, client_ip, failure_reason (enum), auth_method, and log level WARN/ERROR; AND the metric auth_failure_count is incremented and failure logs are never sampled (always sent in full).

3. Given a logout or explicit token revocation occurs, When the revocation is executed, Then the system emits a structured JSON log (event=auth_revocation) with fields: timestamp, request_id, service_name, tenant_id, hashed_user_id, token_id_hash, revocation_reason, actor, and increments auth_revocation_count; AND the revocation is recorded in the token revocation store and the log appears in central logging and monitoring within the same time bounds as other events.

4. Given the environment experiences sustained high authentication throughput (>10k events/min), When the sampling configuration is active, Then metrics are emitted for every event without sampling, successful-event logs may be sampled at a configured rate (default 10%), but failure and revocation logs are never sampled; the sampling rate must be configurable and documented.

5. Given any attempt to send logs to the central logging pipeline fails, When the logging client detects delivery errors, Then authentication operations continue unblocked, events are buffered to a bounded local queue (configurable default 1000 events or 5 MB), retries use exponential backoff, an auth_logging_failed_count metric is emitted, a durable minimal audit fallback is written, and an alert is raised if failures persist >5 minutes.

## Edge Cases
* What happens when the logging backend is unreachable or too slow?
  - Buffer events to a bounded, durable queue (default size: 1000 events / 5 MB).
  - Continue authentication flows without blocking (non-blocking log writes).
  - Retry delivery with exponential backoff and jitter.
  - Emit auth_logging_failed_count and auth_log_dropped_count metrics.
  - If the buffer fills, drop oldest sampled success logs first; never drop failure or revocation logs.
  - Trigger alerting (on-call) if logging failures persist for more than 5 minutes.

* How does the system prevent PII leakage and metric cardinality explosion?
  - Never include plaintext passwords, full email addresses, or other sensitive PII in logs.
  - Hash/obfuscate user identifiers (e.g., SHA-256 truncated) before logging; store hashing scheme in docs.
