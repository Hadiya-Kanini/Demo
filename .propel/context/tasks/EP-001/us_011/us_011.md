## Story ID: us_011

## Story Title: Implement Login API Happy Path

## Description:
  * As a registered user, I want to POST my credentials to /auth/login and have the service validate them, verify the stored password hash (including migration-on-login if required), issue a signed token with required claims, and return an expires_at timestamp so that I can authenticate subsequent requests and the client can manage token lifecycle.

## Acceptance Criteria:
  1. Given a registered user with correct email and password, When the client POSTs valid credentials to POST /auth/login, Then the API responds 200 OK with JSON containing:
     - access_token (string, signed JWT or signed token),
     - token_type: "Bearer",
     - expires_at: ISO 8601 UTC timestamp equal to token.exp,
     - user: { id, email, roles } (no sensitive fields).
  2. Given valid credentials, When password verification occurs, Then the service verifies the password against the stored hash and algorithm; if verification passes, a signed token is issued containing the claims: sub (user_id), email, roles, iat, exp. The token must be verifiable using the platform's current signing key and standard alg header.
  3. Given a configured TTL (configuration value token_ttl_seconds), When the token is created, Then exp = iat + token_ttl_seconds and response.expires_at equals exp expressed as an ISO 8601 UTC string; tests must assert the difference equals token_ttl_seconds (tolerance ±1s for processing).
  4. Given a user record with failed_attempts > 0 or locked_at set, When login succeeds, Then failed_attempts is reset to 0, locked_at is cleared, and last_login_at is updated to the current UTC timestamp atomically with issuing the token.
  5. Given a user record indicating a legacy hashing algorithm eligible for migration, When authentication with the legacy hash succeeds, Then the service rehashes the plaintext password with the new recommended algorithm and updates the user record atomically (so the successful login and migration are both applied), and the response returns the issued token as above.

## Edge Cases:
  - Concurrent successful logins for the same user:
    - Handling: Allow both requests to complete and issue separate valid tokens. Use DB-level atomic updates or optimistic locking to ensure last_login_at reflects the latest timestamp and failed_attempts resets without lost updates.
  - Migration-on-login failure after successful verification:
    - Handling: Do not lose the user's ability to login. Either (a) proceed with issuing the token and mark the account for follow-up with an audit log and automated retry, or (b) if persistence is impossible due to a data integrity error, return 500 and log a remediation ticket. The preferred behavior is (a) to avoid user disruption while alerting ops.
  - Signing key / vault unavailability at token issuance:
    - Handling: Return 503 Service Unavailable with a non-sensitive error code; do not modify authentication-related counters (failed_attempts/last_login_at) and emit an alert to the platform/security on-call channel. Include a retry/backoff strategy at client or gateway if appropriate (documented separately).
