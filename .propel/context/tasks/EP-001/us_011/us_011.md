## Description:
  * As an authenticated user, I want to POST my current token to /auth/logout (or request revocation of all my sessions) and have the service revoke the token(s) (via a blacklist/revocation store or by invalidating a session record) so that revoked tokens are rejected for subsequent requests and session state follows product policy.

## Acceptance Criteria:
  1. Given a valid, non-expired access token presented in Authorization: Bearer <token>, When the client POSTs to POST /auth/logout, Then the API responds 200 OK and the presented token is marked revoked such that subsequent requests using that token are rejected with 401 Unauthorized (error code: "invalid_token").
  2. Given revocation-by-jti is enabled, When a token is issued (login flow), Then each issued token MUST include a unique identifier claim (jti) and the logout operation MUST use the token's jti as the revocation key. (Dependency: token creation story must add jti.)
  3. Given a token with exp > now, When that token is revoked, Then the revocation entry persisted to the revocation store has a TTL equal to (token.exp - current_time) (tolerance ±1s) so entries naturally expire when the token would otherwise expire.
  4. Given a revoked token, When a protected endpoint receives a request with that token, Then the token verification middleware checks the revocation store and rejects the token with 401 Unauthorized and a standard error body { "error": "invalid_token", "error_description": "token revoked" } (no sensitive debug details).
  5. Given the product supports "revoke all sessions" (logout all devices) and the authenticated user issues POST /auth/logout?all=true (or equivalent), When authorized, Then all active sessions/tokens for that user are invalidated (either by marking all session records revoked or inserting revocation keys for known jtis) and the response includes 200 OK with a count of revoked sessions (or a stable response if count is not available).
  6. Given a token missing a revocation identifier (no jti) and revocation strategy requires jti, When client calls POST /auth/logout with that token, Then the API responds 400 Bad Request with a clear machine-friendly error code "token_not_revocable" and logs the incident for follow-up. (Systems should avoid issuing non-revocable tokens if revocation is required.)
  7. Given the revocation store or session-store is unavailable, When a logout request is made, Then the API returns 503 Service Unavailable (do not claim the token revoked), emit observability/alerting events, and surface a non-sensitive error code to the caller so clients can retry later. The system must not return 200 OK unless revocation is persisted.
  8. Given the revocation store is used for token checks, When verifying tokens, Then the verification path includes a cache-friendly lookup (with a short TTL) to avoid excessive store load and must guarantee consistency for revoked tokens (no false-acceptance window beyond allowed clock skew).
  9. Given tokens are rotated/keys are rotated, When verifying revocation, Then the revocation check relies on token-specific identifiers (jti or session id) that remain valid across signing-key rotations so revoked tokens cannot be accepted simply because a new key was used.
  10. Given an administrative “revoke user” or incident flow (out-of-band), When an operator marks a user or specific session as revoked, Then the same revocation mechanism is used so middleware rejects those tokens the same way as user-initiated logout.

## Edge Cases:
  - What happens when a token is already expired at logout time?
    - Handling: If token is expired, API may return 200 OK (idempotent) or 400 with "token_expired" depending on product policy; preferred behavior is 200 OK (idempotent) since expired tokens are already invalid, and log a debug event. Revocation store should not be written for already-expired tokens.
  - What happens when a token lacks jti but revocation is required?
    - Handling: Return 400 "token_not_revocable" and log. Long-term mitigation: ensure token issuance includes jti (see dependency).
  - How does the system handle concurrent logout and request use (race condition)?
    - Handling: Token verification MUST check revocation store at time of request; the design should accept a small, bounded race window if using caches but ensure strong read-after-write semantics for immediate revocation where product requires it (e.g., bypass caches for revocation checks for a short period or implement cache invalidation).
  - What if the revocation store grows very large (scale / cost)?
    - Handling: Enforce TTL = token remaining lifetime so entries auto-expire; support sharding/partitioning and an eviction policy. Monitor size and provide alerts. Consider session-store approach (per-user active sessions index) to reduce per-token entries when refresh tokens or long-lived tokens are used.
  - What if the revocation store is unavailable during logout?
    - Handling: Return 503 as acceptance criteria requires; do not mark token revoked. Optionally provide a best-effort "soft logout" message to the client but do not misrepresent persistence.
  - How to handle tokens issued before revocation feature was enabled?
    - Handling: Provide a migration path: either accept non-revocability (documented risk), issue a forced rotation (require re-login), or seed the revocation store with active session identifiers if available. This must be decided in product policy and documented in implementation notes.
  - Key rotation interaction:
    - Handling: Revocation must be based on token identifiers (jti/session id), not on signing key values, so token revocation remains effective across signing-key rotations.
  - Replay / reuse of revoked token in rapid succession:
    - Handling: Middleware must consistently check revocation and return 401 for each request; logs should capture attempted reuse for security monitoring.
  - Large-scale "revoke all" operation impact:
    - Handling: Long-running revocation operations must be bounded (batch), report progress, and retry on transient failures. Clients should receive an accepted/processing response if operation cannot complete synchronously.

Notes:
- Implementation choices (in-memory blacklist vs persistent revocation store vs session store) are design decisions; acceptance criteria require the chosen approach to satisfy persistence, TTL alignment, and correctness guarantees. If a cache is used to accelerate checks, ensure cache coherence and a strategy to avoid a window where revoked tokens are accepted beyond allowed clock skew.
- Security: ensure revocation endpoints require authentication and appropriate CSRF protections where applicable for browser clients.
- Observability: emit audit logs for revoke actions (who, when, token jti or session id) and metrics for revocation store size, latency, and failure rates.
