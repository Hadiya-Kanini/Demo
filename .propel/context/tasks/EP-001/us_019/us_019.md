## Story ID
   * ID Format: US_<unique seq num>
   * ID: us_019

## Story Title
   * Developer Sandbox & Mock Vault for Local Development

## Description
  * As a developer, I want a turnkey developer sandbox and a local mock vault service that emulates the production secrets-vault interface, so that I can produce, sign, and verify authentication tokens locally and safely without ever using production secrets.

## Acceptance Criteria
  * Given a developer workstation with Docker and the repo checked out, When the developer runs the documented startup (e.g., ./scripts/start-dev-sandbox.sh or docker-compose -f docker-compose.dev.yml up), Then:
    - the mock vault service is available at the configured local URL (default: http://localhost:8200/mock-vault or a clearly documented URL),
    - the auth service is configured to point to that mock vault via an environment variable (e.g., DEV_VAULT_URL),
    - the README and startup script print the service endpoints and an example curl command,
    - and the provided smoke-test script exchanges credentials for a token and verifies the token signature against the mock vault public key, exiting 0 on success.

  * Given the mock vault is seeded using the included seeding script or fixture, When a developer performs a POST /auth/login (using the local sandbox auth service) with valid test credentials (provided in the sandbox seed data), Then:
    - the auth service fetches the signing key from the mock vault,
    - issues a signed token containing claims: sub (user id), roles (array), iat, exp,
    - responds with HTTP 200 and body that includes a token and expires_at in ISO8601 format,
    - and the token can be verified by retrieving the corresponding public key from the mock vault’s public key endpoint and validating signature, claims (sub, roles), and exp within the configured clock-skew tolerance.

  * Given the developer enables the sandbox, When the mock vault is configured or discovered with a URL or settings that match known production domains/hosts (configurable pattern: e.g., PRODUCTION_VAULT_DOMAINS), Then startup must refuse to run (exit non-zero) and print an explicit, actionable error that instructs the developer to use the mock vault and shows the expected mock vault URL; no production vault credentials are used in the sandbox environment by default.

  * Given a need to simulate key rotation and invalid signatures, When a developer triggers the provided key-rotation endpoint or uses the seed script to rotate the active signing key in the mock vault, Then:
    - the mock vault provides both the previous and new public keys via its keys listing endpoint for a configurable grace period,
    - tokens signed with the previous key remain verifiable until the configured grace period elapses,
    - and the mock vault exposes a clear admin endpoint for rotating keys during dev testing with a documented command and expected behaviors.

  * Given CI or automated local unit/integration tests run against the sandbox, When tests are executed with the sandbox fixtures, Then:
    - the mock vault must provide a pre-seeded deterministic test key pair (seeded from repo fixtures, not production data),
    - the smoke-test and integration tests must pass token-sign and verify flows,
    - and all test artifacts must not include or reference production secrets (the startup script validates and fails if production-like secrets are detected).

## Edge Cases
   * What happens when the mock vault is unreachable?
     - Behavior: The startup script and auth service should fail fast with an explicit, actionable error stating "Mock vault unreachable at {URL}. To continue, ensure the mock vault is running or set ALLOW_EPHEMERAL_KEY=true to run with a temporary in-memory signing key (NOT for production)." If ALLOW_EPHEMERAL_KEY=true is set intentionally, the auth service must:
       - generate an ephemeral key pair at startup,
       - log a strong WARNING that ephemeral keys are in use and cannot be used in CI or persisted sessions,
       - and write ephemeral key fingerprint to logs for reproducibility of local troubleshooting.
   * How does system handle accidental production configuration in a local sandbox?
     - Behavior: During sandbox start, config-check must detect if any vault URL, credentials, or vault token match configured production patterns (configurable via PRODUCTION_VAULT_DOMAINS and PRODUCTION_SECRET_PATTERNS). If a match is found, startup must abort with a clear error and remediation steps. If the developer intentionally overrides with an explicit allow flag (e.g., FORCE_USE_PROD_VAULT_IN_DEV=true), the script must require multi-step confirmation (an environment variable and an interactive prompt) and then log a conspicuous warning—this must be disabled in CI.
   * What happens during local key rotation mid-test?
     - Behavior: The mock vault supports rotating keys and exposing both current and previous public keys for a configured grace-window (default 5 minutes). Token verification accepts tokens signed by any key present in the vault's public-keys endpoint that is within its grace period; after the grace window, verification with old keys should fail, and tests that expect rotation behavior must be able to configure the grace-window via environment variable for deterministic test runs.

Notes (developer-facing):
- Deliverables for this story:
  - docker-compose.dev.yml (or similar) with a mock-vault service and seeded test keys (stored in repo fixtures; not production secrets).
  - scripts/start-dev-sandbox.sh that starts services and runs a smoke test.
  - mock-vault implementation that exposes a minimal subset of the production vault API needed by auth service:
    - GET /mock-vault/v1/keys -> list of public keys (kid, alg, public_jwk/pem, active flag, created_at)
    - GET /mock-vault/v1/keys/{kid} -> public key metadata
    - POST /mock-vault/v1/admin/rotate -> rotate signing key (dev-only)
    - GET /mock-vault/v1/signing-key -> returns private key for services configured to use the mock vault (auth service will call this in dev only)
  - README.md with step-by-step instructions, environment variable list, and troubleshooting steps for common failures (mock vault unreachable, production config detected).
  - Automated smoke-test script to exercise: start, seed, login, token verify, rotate key, re-verify behavior.
- Security and safety constraints:
  - The repo MUST NOT contain production key material.
  - Sandbox should clearly indicate in all logs that it is a development-only environment.
  - CI pipelines should disable fallback ephemeral key behavior and require sandbox fixtures to be used deterministically.

This story adheres to INVEST: independent (developer sandbox is a vertically useful capability), negotiable (implementation of mock API and runtime choices can vary), valuable (enables safe local development and reliable tests), estimable (well-scoped), small (can be completed within a single sprint-sized task), and testable (clear Given/When/Then acceptance criteria and smoke tests).
