## Story ID: us_003

## Story Title: Client Login Form & Validation

## Description: As a returning user, I want a client-side login form with accessible controls and validation, so that I can sign in quickly with clear guidance while the app enforces non-revealing error messaging and accessibility requirements.

## Acceptance Criteria (Given/When/Then format)
1. Given I am on the login screen, When the page loads, Then the form renders with semantic HTML (email input, password input, submit button, "Forgot password" link), visible labels, keyboard focus on the first input, and an aria-live region for error messages.

2. Given I enter an invalid email format or leave required fields empty, When I blur the field or attempt to submit, Then inline validation prevents submission, shows an accessible inline error (e.g., "Enter a valid email address" / "Password is required"), and the submit button remains disabled until client-side validation passes.

3. Given I submit valid credentials, When the client sends the request to the /auth/login endpoint and receives a successful response with authentication and role information, Then the UI shows a loading state (submit disabled, spinner), stores auth per security guidance (prefer server-set HttpOnly cookie), and redirects the user to the role-appropriate dashboard.

4. Given the server responds with authentication failure (invalid credentials), When the client receives a 401/403 response, Then display a non-revealing, accessible error message in the aria-live region: "Invalid email or password" (do not indicate which field is incorrect), and allow retry after clearing or editing inputs.

5. Given the server responds with rate-limit or temporary error (e.g., 429 or 5xx), When the client receives that response, Then present a user-friendly, non-technical message (e.g., "Too many attempts. Try again later." or "We're having trouble—please try again in a few minutes."), preserve input values (except password if security policy requires clearing), and surface links to help/forgot-password where applicable.
