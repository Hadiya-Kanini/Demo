## Story ID
   * ID: us_003

## Story Title
   * Issue Access Tokens with TTL and Claims

## Description
  * As a backend authentication service, I want to issue signed access tokens with configurable TTL and required claims so that clients can authenticate to protected APIs securely and tokens expire according to policy.

## Acceptance Criteria
1. Given valid user credentials and client authorization, When the client POSTs to the token issuance endpoint (/auth/token or /auth/login per API contract) with required parameters, Then the system responds with HTTP 200 and a JSON payload containing:
   - access_token: a signed token (JWT or agreed token format),
   - token_type: "Bearer",
   - expires_in: numeric seconds equal to configured TTL,
   - scope: granted scopes,
   and the token payload includes claims: sub, iss, aud, iat, exp, jti, roles (or scope) and any allowed custom claims as defined by policy.

2. Given an issued token and a protected API request, When the API validates the token, Then the token is accepted only if:
   - signature validates using the active JWK(s),
   - current_time + allowed_clock_skew < exp,
   - required claims (sub, aud, iss) are present and match expected values,
   - token is not listed as revoked;
   otherwise the API returns HTTP 401 Unauthorized with a non-revealing error (e.g., "Invalid or expired credentials").

3. Given the system uses key rotation and publishes a JWK Set endpoint, When a token is issued with the active key and later verified, Then verification succeeds for tokens signed by current or recently-rotated keys present in the JWK set until token exp, and verification fails for tokens signed with unknown or retired keys beyond the rollover/retention window.

4. Given a token issuance request includes requests for optional custom claims (e.g., department, tenant_id), When policy allows inclusion for that client and user, Then only permitted claims are included; otherwise requested claims are omitted and the response still returns a valid token without unauthorized attributes.

5. Given a token has been explicitly revoked (revocation record exists), When that token is presented or introspected, Then the introspection endpoint returns active=false and protected APIs reject the token with HTTP 401 and an audit event is recorded with minimal, non-sensitive details.

## Edge Cases
* What happens when system clock skew exists between issuer and consumer?
  - Handling: Implement a configurable clock-skew tolerance (default 60 seconds) applied during validation (allow token if current_time <= exp + skew and current_time >= nbf - skew when nbf is used). Log warnings when tolerance is exercised and alert if skew exceeds a threshold.

* How does the system handle malformed, tampered, or tokens signed with unsupported algorithms?
  - Handling: Reject with HTTP 401 Unauthorized, increment security metrics, return a standardized non-revealing error message, and emit a structured audit/log entry including the token jti (if parseable) or correlation id; do not include raw token or sensitive internals in logs.

* What happens when configured TTL is zero, negative, or extremely large?
  - Handling: Enforce server-side validation of TTL configuration; reject issuance if requested TTL is out of allowed bounds (min > 0, max <= configured_max_ttl). If configuration is invalid, fail issuing with HTTP 500 and create an operation alert for platform team; do not issue tokens with indefinite expiry.
