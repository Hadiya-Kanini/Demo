## Story ID: us_015

## Story Title: Configure Token Expiry and Clock Skew

## Description:
  * "As a platform security engineer, I want token expiry (TTL) and allowable clock skew to be configurable so that token issuance and verification honor operational policy and tolerances across distributed clients and servers."

## Acceptance Criteria:
  * Given system configuration contains token_ttl_seconds and allowable_clock_skew_seconds, When a user successfully logs in, Then the issued token (JWT) MUST include an exp claim set to (issued_at + token_ttl_seconds) and the login response MUST include an expires_at timestamp matching that exp value in ISO-8601 UTC.
  * Given an incoming token presented to the verification endpoint, When current_server_time <= exp + allowable_clock_skew_seconds, Then the token MUST be accepted as not-expired; otherwise verification MUST fail with a standard 401/invalid_token error and a machine-readable reason 'token_expired'.
  * Given configuration values are updated at runtime (e.g., via config service or environment reload), When a subsequent login request occurs, Then new tokens MUST be issued using the updated token_ttl_seconds immediately and verification MUST use the updated allowable_clock_skew_seconds for both new and existing tokens.
  * Given a token near expiry within the allowable_clock_skew_seconds window, When verification runs, Then authentication succeeds and metrics/logging MUST record that the token was accepted due to clock skew (for monitoring) without exposing timing/expiry details to clients.
  * Given automated test suites run, When unit, integration, and e2e tests execute, Then tests MUST cover: issuance TTL correctness, verification acceptance within skew, verification rejection beyond skew, dynamic configuration change effects, and the case of missing or malformed exp claims (which MUST be rejected).

## Edge Cases:
   * What happens when system clock jumps forward or backward?
     - Handling: Verification MUST prefer monotonic time where the runtime supports it for skew calculations. If monotonic time is unavailable, the verification logic MUST apply allowable_clock_skew_seconds conservatively and emit a WARN-level audit log. Tokens that appear expired because of a significant clock jump MUST be rejected and recorded for incident investigation.
   * How does system handle tokens missing an exp claim or with malformed exp?
     - Handling: Tokens missing an exp claim or having a non-numeric/malformed exp MUST be rejected with HTTP 401 and machine-readable error 'missing_or_malformed_exp'. Rejection SHOULD be logged with safe audit details (no token material).
   * What happens if allowable_clock_skew_seconds or token_ttl_seconds are misconfigured (negative or excessively large)?
     - Handling: On startup and on config reload, values MUST be validated. Negative values MUST be rejected and the configuration change must fail. Values above a configurable safe maximum (default cap = 24 hours) MUST be clamped and emit an admin alert and WARN log entry; the system MUST continue running with clamped values to avoid security regressions.

Notes/Implementation guidance (non-normative, for devs and QA):
- Config sources: support environment variables, centralized config service, and local config file; document precedence and provide automated reload tests.
- Logging/metrics: emit metric counters for tokens issued (tagged by TTL used), token_verification_accepted_due_to_skew, token_verification_expired, and config_reload_success/failure. Emit WARN/AUDIT on suspicious clock events.
- Tests: include unit tests for TTL computation, property-based tests around boundary times, integration tests using simulated clock (freeze/advance), and an e2e test that toggles config and verifies issuance/verification behavior.
- Security: never echo token contents in logs; log only token IDs or safe hashes when needed for correlation.
