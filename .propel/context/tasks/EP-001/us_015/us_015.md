## Story ID: us_015

## Story Title: Legacy Password Hash Migration Support

## Description:
  * "As a platform security engineer, I want the authentication system to verify legacy password hashes securely and transparently re-hash passwords to the new algorithm on successful login so that users are not forced to reset passwords and stored credentials are migrated to the stronger algorithm over time."

## Acceptance Criteria:
  * Given a user record whose password is stored using the new algorithm, When the user logs in with valid credentials, Then the system MUST verify with the new algorithm and must not alter the stored hash.
  * Given a user record whose password is stored using a recognized legacy algorithm, When the user logs in with valid credentials, Then the system MUST verify using the legacy algorithm in a timing-safe manner and, on success, MUST re-hash the plaintext password with the current configured algorithm and atomically update the stored user record to the new hash/version before finalizing login.
  * Given a re-hash occurs as part of login, When the login completes, Then the client response MUST be indistinguishable from a normal successful login (no algorithm/version details), and internal audit/metric MUST record the migration event (user_id or safe hash, previous_version, new_version, timestamp).
  * Given concurrent login attempts for the same account, When a re-hash migration is attempted, Then the system MUST preserve data integrity using optimistic locking or atomic conditional updates so only one migration-write persists and no corrupted hash states are produced.
  * Given a stored hash has an unknown, malformed, or corrupted format, When a login attempt occurs, Then verification MUST fail with HTTP 401/invalid_credentials and the system MUST emit an admin/audit event describing 'unknown_or_corrupted_hash' with only safe identifiers and remediation guidance.
  * Given legacy verification imposes significant CPU cost, When system load or rate limits are reached, Then the system MUST apply adaptive rate-limiting, queueing, or worker-offload strategies to protect availability and MUST emit metrics for legacy_verification_latency, legacy_verification_rejections_due_to_load, and rehash_update_failures.
  * Given runtime configuration for supported algorithms or migration feature flags change, When new logins occur, Then the authentication flow MUST honor the updated configuration immediately and record a config-change audit event.
  * Given automated test suites run, When unit, integration, and e2e tests execute, Then tests MUST cover legacy/new detection, successful legacy verification and on-login re-hash, unknown/corrupted hash handling, concurrency during re-hash update, timing-safe compare behavior, and that no plaintext passwords or full hash values are present in logs.

## Edge Cases:
   * What happens when the legacy hash is corrupted (truncated, malformed, missing salt)?
     - Handling: Reject login (401/invalid_credentials), emit a safe admin/audit event 'corrupted_password_hash', and route the account to standard password-reset remediation flow.
   * How does the system handle unknown/unrecognized hash formats or algorithm identifiers?
     - Handling: Do not attempt verification. Reject login with 401/invalid_credentials. Emit admin alert and create a remediation ticket. Consider bulk-migration tooling or manual reset.
   * What if the re-hash DB update fails after successful verification (transient DB error)?
     - Handling: Do not block user login; complete issuance of tokens. Enqueue an asynchronous retry with exponential backoff for the re-hash update, emit metric 'rehash_update_failed', and escalate to admin alerts if retries exceed threshold.
   * How are race conditions handled when multiple logins attempt migration at the same time?
     - Handling: Use DB conditional writes (compare-and-swap on stored hash/version) or row-version checks; failing conditional writes should cause the instance to re-read the latest record and proceed without corrupting data.
   * What if legacy verification depends on secrets (pepper) or parameters unavailable due to secret rotation?
     - Handling: Fail verification safely and return 401/invalid_credentials; create high-severity alert describing the missing dependency and provide remediation steps; do not expose secret details in logs.
   * How to protect service availability from expensive legacy verification workloads?
     - Handling: Monitor legacy verification latency, apply adaptive rate-limiting or offload to controlled worker pool, and provide metrics to ops. If necessary, reject excessive legacy verification attempts with a throttling response while ensuring operators are alerted.
   * How are migration/audit logs handled to avoid leaking sensitive data?
     - Handling: Logs MUST NOT contain plaintext passwords or full password hashes. Only safe identifiers (user_id or hashed_id), hash_version_from, hash_version_to, timestamp, and correlation IDs are allowed.

Notes/Implementation guidance (non-normative, for devs and QA):
- Hash metadata: store a hash_version or algorithm identifier alongside the stored hash so the system can deterministically detect legacy vs current format.
- Verification: always use constant-time comparisons and vetted crypto libraries for legacy and current algorithms. Avoid home-grown crypto.
- Atomic update: prefer DB conditional-update semantics (e.g., WHERE hash = :old_hash) or transactions with row-version to avoid lost updates or double-write races.
- Secrets: if legacy verification requires a global pepper or external secret, ensure secret retrieval is integrated with EP-006 secrets vault and handle missing secrets with clear admin alerts.
- Logging/metrics: emit metrics for migration_count, legacy_verification_latency, rehash_update_failures, legacy_verification_rejections_due_to_load. Audit events must include only safe identifiers and not reveal algorithm details to clients.
- Operational controls: provide feature-flag/config toggles to disable legacy verification, to enable bulk migration tooling, and to set thresholds for adaptive rate-limiting.
- Tests: unit tests for detection logic, property tests for timing-safety, integration tests exercising on-login re-hash and DB conditional updates, and chaos tests simulating DB failures and concurrent logins.
- Security: never log plaintext passwords or full hash texts. When correlating, use a one-way safe identifier (e.g., HMAC of user_id with rotation-safe key) for support traces.
