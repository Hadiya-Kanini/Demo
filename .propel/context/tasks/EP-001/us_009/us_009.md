## Story ID

   * ID: US_009

## Story Title
   * Automated Unit & Integration Tests for Login Flows

## Description
  * As a developer/QA engineer, I want automated unit and integration tests for the auth login flows so that correctness of login, token claims/expiry, and vault-backed signing behavior is validated and regressions are detected in CI.

## Acceptance Criteria
1. Given a deterministic unit-test environment with time control and test fixtures, When unit tests run for auth core modules, Then they must:
   - validate input validation logic (missing/invalid fields => 400 with documented error code),
   - assert password verification includes timing-safe compare and correct handling of matches/non-matches,
   - verify token creation produces tokens containing required claims (sub, roles, iat, exp) and the service response contains expires_at matching token exp,
   - cover error branches (user not found, locked account) with expected status codes and messages.

2. Given an isolated integration test environment (ephemeral DB, service under test, mocked vault), When integration tests run against /auth/login, Then they must include tests that:
   - perform a successful login and assert HTTP 200, presence of access token and expires_at, and that the token verifies using the service's signing key and contains correct claims,
   - attempt login with invalid credentials and assert HTTP 401 and no token returned,
   - assert token verification rejects expired tokens and returns the documented expiry error code,
   - assert clock-skew tolerance behavior (accept tokens within configured skew, reject tokens outside),
   - simulate vault-provided signing key and rotated key scenarios, asserting correct signing, verification, and rotation overlap behavior,
   - simulate vault-unreachable scenario and assert the service behaves according to contract (uses dev-fallback or surfaces clear error) and tests assert that behavior.

3. Given CI pipeline runs on PRs, When code is pushed, Then unit + integration test suites run automatically and:
   - all tests must pass to allow merge,
   - integration tests run in CI using mocked/emulated vault and seeded DB; they must not require external network services,
   - CI artifacts for failed tests (logs, failing requests/responses, JWT samples, and trace snippets if available) are attached to the PR.

4. Given test quality constraints, When tests are added/modified, Then:
   - critical auth modules have measurable unit coverage (recommendation: >= 80% for core modules; team to confirm exact threshold),
   - integration tests are deterministic (use time mocking or clock freeze) and idempotent (clean up seeded data),
   - the integration test suite executes within an agreed CI time budget (team-configurable threshold) to keep CI pipeline time reasonable.

## Edge Cases
* What happens when the secrets vault is unreachable during tests?
  - Tests must include both the configured fallback behavior and the explicit failure mode. Integration tests that mock the vault should assert the service either uses the dev/test fallback key (and tokens verify) or fails with a clear, documented error depending on configuration.

* How does the system handle clock skew and expiry boundary conditions?
  - Tests must simulate clock skew and validate that tokens within configured skew are accepted and those outside are rejected; tests must assert the exact error codes/messages for expired/invalid tokens.

* What happens when DB user record is missing or corrupted?
  - Integration tests must seed DB state for the missing/corrupted case and assert the login endpoint returns 401 (or documented error) and does not issue a token.

* How are signing key rotations handled mid-test?
  - Integration tests must simulate key rotation and verify that tokens signed with the previous key remain verifiable for the configured overlap window and new tokens are signed with the new key.

* How are timing-related flakiness and CI resource contention handled?
  - Tests must avoid real-time waits and use time mocking or deterministic waits. If retries are used, they need to be deterministic and limited; flaky tests must be fixed before being allowed in CI.
