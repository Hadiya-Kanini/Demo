## Story ID
* ID: US_005

## Story Title
* Implement Role-Based Redirects

## Description
* As an authenticated user, I want to be routed after login to my role-specific dashboard based on the role claim returned in my authentication token, so that I land directly on the interface tailored to my responsibilities and can begin relevant tasks immediately.

## Acceptance Criteria
1. Given a user has successfully authenticated and the issued token contains a validated role claim "Admin", When the authentication flow completes, Then the user is redirected to the Admin dashboard (SCR-DASHBOARD-ROLE - /dashboard/admin), the UI loads Admin-specific modules, and an audit event "redirect:admin" is recorded.

2. Given a user token contains multiple role claims (for example ["User","Manager"]), When the authentication flow completes, Then the system selects the effective role using the configured role-precedence list, redirects the user to the corresponding dashboard (e.g., /dashboard/manager), and logs which role was selected and the precedence reason.

3. Given a user token is missing a role claim or contains an unknown role value, When the authentication flow completes, Then the user is redirected to the generic/default dashboard (/dashboard), a non-blocking informational banner prompts the user to contact their administrator for role assignment, and an audit event "redirect:unknown-role" is created.

4. Given the authentication token is invalid or expired, When the post-login redirect is attempted, Then the system treats the session as unauthenticated, redirects the user to the login page (/login) with a non-revealing message ("Authentication required"), and no role-based redirect occurs.

5. Given successful authentication and a valid token, When the redirect decision is executed, Then the redirect decision and client navigation begin within 1 second (satisfies NFR-003 latency target) and the chosen dashboard's primary data fetch is initiated immediately.

## Edge Cases
* What happens when the token is missing the role claim?
  - Handle: Redirect to the generic/default dashboard, display a clear but non-revealing banner advising to contact the administrator, create an audit/log entry, and surface a support ticket indicator in telemetry.

* How does the system handle multiple roles returned in the token?
  - Handle: Resolve using a configurable role-precedence table (e.g., admin > manager > editor > user). The precedence configuration is stored in application config (backed by EP-DATA); the selected role and precedence rationale are logged for audit purposes.

* What happens when the role mapping service or configuration is unavailable?
  - Handle: Use a last-known-good in-memory cache of role mappings; if none exists, fallback to the generic dashboard and emit a critical telemetry event and an operational alert. Do not block login or expose sensitive debug information to the user.
