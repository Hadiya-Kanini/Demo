## Story ID
   * ID: US_005

## Story Title
   * Token Verification Middleware

## Description
  * As the API platform, I want middleware that validates incoming bearer tokens' signature and claims (including iat/exp) with configurable clock-skew tolerance, so that protected endpoints only receive requests authenticated with valid, non-expired tokens and operators can tune expiry/clock-skew behavior safely.

## Acceptance Criteria
1. Given an incoming HTTP request with Authorization: Bearer <token>, When the middleware processes the request, Then it:
   - Verifies the token signature against known signing key(s) (supporting key id/kid lookup).
   - Validates required claims (at minimum sub and exp; enforce aud/roles if configured).
   - Validates iat/exp with configured clock-skew tolerance (both forward and backward skew).
   - Attaches an authentication principal (sub, roles, claims) to request context for downstream handlers.
   - Allows the request to proceed when all validations pass.

2. Given a token with an invalid signature or unsupported algorithm (e.g., alg:none), When processed, Then middleware returns HTTP 401 Unauthorized, a generic error body (no sensitive details), and logs an audit/security event with a non-sensitive failure code.

3. Given a token expired outside the configured clock-skew tolerance, When processed, Then middleware returns HTTP 401 Unauthorized with an error code indicating token_expired, logs an audit event, and does not forward the request.

4. Given a token whose exp/i at are within the configured clock-skew tolerance (recently expired or issued slightly in the future within configured tolerance), When processed, Then middleware accepts the token subject to other checks.

5. Given a malformed token or malformed Authorization header, When processed, Then middleware returns HTTP 400 Bad Request (or 401 per platform policy), returns a generic error message, and logs the incident.

6. Given a token missing required claims (e.g., missing sub or required aud), When processed, Then middleware returns HTTP 401 Unauthorized and logs the event.

7. Given revocation-checking is enabled and the token is recorded as revoked, When processed, Then middleware returns HTTP 401 Unauthorized and logs an audit event indicating revocation detection.

8. Given revocation-checking is enabled and the revocation store is unreachable, When processed, Then middleware returns HTTP 503 Service Unavailable (fail-closed), logs with correlation id, and emits operator metrics/alerts. If configured as best-effort instead of required, middleware accepts tokens but emits an alert and metric (explicit configuration must control this behavior).

9. Given key retrieval (secrets vault or JWKS) is required and key discovery fails, When processing tokens, Then middleware returns HTTP 503 Service Unavailable (fail-closed), honors cached keys within configured TTL, logs details with correlation id, and emits metrics/alerts. Key cache must be configurable (TTL, size) and subject to single-flight discovery to avoid stampedes.

10. Given performance constraints, When validation is executed under normal operation, Then middleware should add no more than the configured NFR overhead to median request latency (default target <= 5ms median), and expose metrics for validation latency, cache hit/miss, success/failure rates.

## Edge Cases
* What happens when a token's iat is in the future beyond configured forward-skew?
  - Reject with HTTP 401 Unauthorized (token not yet valid). Log and emit a metric; if many requests show forward skew, surface a clock-sync alert.

* How does middleware handle alg mismatch or 'none' algorithm?
  - Reject with 401 and log as potential attack.

* What happens during key rotation when tokens are signed with new/old keys?
  - Middleware should verify with cached keys first; if kid unknown, attempt discovery. If discovery succeeds or cached previous key verifies, accept. If discovery fails and verification cannot be completed, fail closed (503) or 401 depending on key retrieval vs verification result.

* How does middleware handle revocation-checking unavailability?
  - If revocation-checking is required, return 503 and emit alerts. If configured best-effort, accept but emit high-severity metrics; behaviour must be explicitly configured.

* How are tokens that claim to be refresh tokens handled at resource endpoints?
  - Reject with 401 if refresh tokens are disallowed for resource endpoints. If allowed, validate same as access tokens and ensure scope/claims enforce allowed actions.

* How are large numbers of simultaneous key discoveries prevented?
  - Implement single-flight/backoff for discovery; on repeated failures return 503 and emit metrics.

* How does middleware mitigate DOS via very large or complex token payloads?
  - Enforce a maximum token size and parsing limits; reject over-size tokens with 400 and rate-limit offenders.
