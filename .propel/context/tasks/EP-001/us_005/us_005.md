## Story ID
   * ID: US_005

## Story Title
   * Add Logout Endpoint to Revoke Tokens

## Description
  * As an authenticated API client, I want a POST /auth/logout endpoint that revokes my active token so that my session is terminated immediately and the token cannot be reused.

## Acceptance Criteria
1. Given a valid, unrevoked bearer access token in the Authorization header, When the client POSTs to /auth/logout, Then the system marks the token as revoked in the token store and any caches, returns HTTP 200 with JSON { "success": true, "message": "Token revoked" } (non-revealing), and subsequent requests using the same token to protected endpoints result in HTTP 401 within 5 seconds.

2. Given a revoked or already-expired token, When the client POSTs to /auth/logout, Then the system responds idempotently with HTTP 200 and JSON { "success": true, "message": "Token not active" } and records an audit/log entry indicating an idempotent logout attempt.

3. Given no Authorization header or a malformed token, When the client POSTs to /auth/logout, Then the system returns HTTP 401 Unauthorized (or HTTP 400 Bad Request for clearly malformed tokens), with a generic error message (no sensitive details) and does not attempt to mark any token as revoked.

4. Given the token revocation persistence layer (e.g., database or Redis) is unavailable or returns errors, When the client POSTs to /auth/logout, Then the system returns HTTP 503 Service Unavailable, logs the failure with correlation id and details for operators, and does not return a success confirmation; where appropriate, it enqueues a retryable revocation task (if safe) and exposes an observability event.

5. Given a user provides an access token that is associated with an active refresh token (per system linkage), When the client POSTs to /auth/logout and policy requires cascade revocation, Then the system revokes both access and linked refresh tokens atomically (or marks both as revoked) and returns HTTP 200 confirming both were revoked.

## Edge Cases
* What happens when a token is already expired?
  - The system responds idempotently with HTTP 200 and message indicating the token is not active. Expired tokens still must not authenticate; any use after logout/expiry results in HTTP 401.

* How does the system handle logout when the revocation store is temporarily down?
  - The system returns HTTP 503 Service Unavailable and does not report success. It logs the event with a correlation id and, if architecture allows, enqueues a guaranteed-delivery retry (background job) for revocation; operations must monitor and surface retries to SRE.

* What happens when client supplies a refresh token instead of an access token?
  - If API accepts refresh-token-based logout per spec, validate token type; if allowed, revoke the refresh token (and optionally any associated access tokens) and return HTTP 200. If not allowed, return HTTP 400 with a generic message.
