## Story ID: us_007

## Story Title: Standardize Non-Revealing Error Responses

## Description: 
As a backend engineer / product owner, I want authentication and validation error responses standardized and non-revealing across client and API, so that we prevent information disclosure that could aid account enumeration or attacks while providing consistent, testable error handling for clients and maintainers.

## Acceptance Criteria (Given/When/Then format)
1. Given a login attempt with invalid credentials (unknown username OR wrong password), When the API responds, Then it returns HTTP 401 with a standardized JSON body containing a stable error code and a generic message (e.g., { "code": "AUTH_FAILURE", "message": "Authentication failed" }), includes a correlation_id header for traceability, and does not indicate whether the username exists or which part of the credentials was wrong.

2. Given a password-reset or "forgot password" request that references an email/username, When the endpoint processes the request, Then the response is identical for existing and non-existing accounts (e.g., HTTP 200 with { "code":"RESET_REQUEST_RECEIVED", "message":"If an account exists we will send instructions" }), and no timing differences or different messages indicate account existence.

3. Given a request with input validation errors for non-sensitive fields (e.g., malformed email format, missing required fields on registration), When the API responds, Then it returns HTTP 400 with a structured, non-sensitive error payload: { "code": "INVALID_INPUT", "message": "Invalid input", "errors": [ { "field": "email", "message": "Invalid format" } ] } — field-level messages allowed for format/length but must not include sensitive existence or internal diagnostics.

4. Given an unexpected server error or an exception during request processing, When the API responds in production, Then it returns HTTP 500 with a generic standardized payload (e.g., { "code":"SERVER_ERROR","message":"An unexpected error occurred"
