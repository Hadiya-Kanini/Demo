## Story ID
   * ID: US_007

## Story Title
   * Auth API Error Codes & Docs

## Description
  * As an API consumer and backend engineer, I want a consistent set of machine-readable error codes and non-revealing human-readable messages for authentication endpoints, with examples included in the API documentation and OpenAPI specification, so that clients can reliably handle error cases and we avoid leaking sensitive information.

## Acceptance Criteria
1. Given the authentication API endpoints exist, When the backend returns an error, Then the response body conforms to the standardized error schema:
   - HTTP status: appropriate standard code (see mappings below).
   - JSON body:
     {
       "error": {
         "code": "AUTH_<UPPER_SNAKE_CODE>",
         "message": "Human-facing non-revealing text.",
         "details": { /* optional machine-friendly object for field errors or vendor info */ },
         "request_id": "UUID or trace id"
       }
     }

2. Given malformed or invalid input to /auth/login, When validation fails, Then the backend returns 400 Bad Request with:
   - error.code = "AUTH_INVALID_INPUT"
   - error.details contains field_errors array: [{ "field": "<fieldName>", "code": "AUTH_<FIELD_SPECIFIC_CODE>", "message": "Brief description" }]
   - Example codes (must be present in docs): AUTH_INVALID_EMAIL, AUTH_PASSWORD_TOO_SHORT, AUTH_MISSING_FIELD

3. Given incorrect credentials are supplied, When authentication fails, Then the backend returns 401 Unauthorized with:
   - error.code = "AUTH_INVALID_CREDENTIALS"
   - error.message = a non-revealing message such as "The credentials provided are incorrect."
   - No field-level indication that the account or email exists

4. Given an account is locked/disabled due to policy or admin action, When a login attempt is made, Then the backend returns 403 Forbidden with:
   - error.code = "AUTH_ACCOUNT_LOCKED" (or "AUTH_ACCOUNT_DISABLED" where appropriate)
   - error.message = a non-revealing but actionable message such as "This account is temporarily unavailable. Contact support if you need help."

5. Given rate limiting thresholds are exceeded, When requests continue, Then the backend returns 429 Too Many Requests with:
   - error.code = "AUTH_RATE_LIMITED"
   - error.message = "Too many attempts. Please wait and try again later."
   - Response includes standard Retry-After header where applicable

6. Given unexpected internal failures, When the system cannot process the request, Then the backend returns 500 Internal Server Error with:
   - error.code = "AUTH_INTERNAL_ERROR" or "AUTH_UNKNOWN_ERROR"
   - error.message = generic "An unexpected error occurred. Reference: <request_id>."
   - request_id present to correlate logs for debugging

7. Given third-party identity provider errors (e.g., OAuth provider unavailable), When the provider returns an error, Then the backend maps to one of the documented codes (e.g., AUTH_IDP_ERROR) and includes minimal details in error.details for operator debugging, while error.message remains non-revealing to end users.

8. Given the API specification and docs repository, When this story is complete, Then:
   - The OpenAPI/Swagger spec includes the error schema as a reusable component and enumerates all AUTH_* codes for auth endpoints (POST /auth/login, POST /auth/refresh, etc.) with sample responses for each mapped HTTP status.
   - The API docs contain a table of error codes, canonical human message, recommended client handling, and sample JSON for each code.
   - A small examples folder contains curl/JSON examples demonstrating each error response for auth endpoints.
   - The error codes are defined as stable strings (UPPER_SNAKE) and the docs specify the compatibility policy for changes (e.g., adding new codes is minor release; renaming is breaking).

9. Given automated tests and CI, When server-side behavior is validated, Then integration tests assert that endpoints return the correct HTTP status, error.code, and that error.message does not include account-existence or other sensitive details for credential failures.

10. Given localization requirements, When the docs and API are reviewed, Then the spec documents that machine-readable error.code is the canonical value clients should use for logic and that error.message may be localized; the spec provides a recommended pattern for localization (e.g., Accept-Language and a fallback to an English message and request_id).

## Edge Cases
* What happens when the backend receives a malformed JSON payload?
  - Handling: Return 400 Bad Request with error.code = "AUTH_INVALID_INPUT" and a general message; if possible include a minimal detail "malformed_json": true in error.details. Do not include parsed content or sensitive data.

* How does the API handle extremely large request payloads or oversized fields?
  - Handling: Return 413 Payload Too Large or 400 with error.code = "AUTH_PAYLOAD_TOO_LARGE" with guidance in message and preserve no sensitive input in the response or logs.

* What happens when an unknown/unmapped error occurs (exception not covered by mapping)?
  - Handling: Return 500 with error.code = "AUTH_INTERNAL_ERROR" (or "AUTH_UNKNOWN_ERROR"), generic message referencing request_id. Ensure request_id is logged server-side to correlate and debug.

* How are third-party identity provider errors surfaced?
  - Handling: Map provider failures to AUTH_IDP_ERROR (or more specific AUTH_IDP_TIMEOUT / AUTH_IDP_INVALID_RESPONSE) with minimal details in error.details for operator debugging; messages must not reveal upstream provider-specific sensitive details to end users.

* How to handle account lockout vs rate-limit to avoid user enumeration?
  - Handling: Use distinct codes for internal telemetry (AUTH_ACCOUNT_LOCKED) but ensure public-facing messages are non-revealing and guidance is consistent; documentation must instruct clients to surface generic guidance rather than exposing whether a specific email exists.

* How are field-level validation errors represented?
  - Handling: In 400 responses include error.details.field_errors array with entries for each bad field: { field, code, message }. Field-specific codes must be prefixed with AUTH_ and documented.

* How to handle localization of human messages?
  - Handling: error.code remains constant; error.message may be localized per Accept-Language. Documentation must require clients to rely on error.code for logic and use message purely for display.

* How to handle deprecated or extended error codes?
  - Handling: Document deprecation policy. Deprecated codes remain supported for at least one major version; clients should rely on codes but be tolerant of unknown codes (treat unknown as generic error).
