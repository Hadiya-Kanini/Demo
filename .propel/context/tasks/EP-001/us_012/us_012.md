## Story Title
   * Configurable Token Expiry & Clock Skew

## Description
  * As a system administrator / backend engineer, I want configurable token expiry and clock-skew tolerance (via configuration defaults and runtime admin controls), so that token issuance and verification behave securely and robustly across heterogeneous client environments (handling clock drift, client TTL requirements) without code changes and with tests to prevent regressions.

## Acceptance Criteria
  1. Given the auth system has configured timing settings (default_ttl_seconds, max_ttl_seconds, clock_skew_seconds),
     When a successful POST /auth/login issues a token,
     Then the response includes a signed token (JWT or agreed transport) with iat and exp claims and a JSON response field expires_at (UTC ISO8601) equal to iat + effective_ttl; effective_ttl must be determined by per-client override if present, else global default, and must be <= max_ttl_seconds.

  2. Given a JWT token presented to protected endpoints,
     When the token verification middleware validates timing claims,
     Then it accepts a token if:
       - iat <= now + clock_skew_seconds (allow small future iat due to client clock),
       - exp >= now - clock_skew_seconds (allow recently expired tokens due to client clock),
     Otherwise it rejects the token with HTTP 401 and clear error_code values ("TOKEN_NOT_YET_VALID" for iat violations, "TOKEN_EXPIRED" for exp violations), and logs request_id and observed time differences.

  3. Given an admin with privileges,
     When the admin requests current auth timing settings via GET /admin/auth/settings,
     Then the system returns the current effective configuration (default_ttl_seconds, max_ttl_seconds, clock_skew_seconds, per_client_defaults) without exposing signing secrets and includes request_id for tracing.

  4. Given an admin with privileges and valid input,
     When the admin updates auth timing via PATCH /admin/auth/settings (e.g., change clock_skew_seconds from 60 to 120),
     Then the system validates inputs, persists them, applies them at runtime without restart (within a bounded propagation window), returns updated configuration and request_id, and rejects invalid updates with HTTP 400 and field-level error messages.

  5. Given a client requests a token with a requested TTL or a client-specific override exists,
     When the token is issued,
     Then effective_ttl = min(requested_ttl, per_client_max_ttl, global_max_ttl) and the token's exp reflects that; the server must never issue tokens with TTL > global max_ttl_seconds.

  6. Given configuration changes and a multi-instance deployment,
     When an admin updates settings,
     Then changes persist centrally and are observable by all instances within the defined propagation window (documented expectation e.g., 5s); instances that cannot observe the change must log a warning and retain last-known-good configuration.

  7. Given the test suites run,
     When unit and integration tests execute,
     Then they cover issuance (iat/exp/expires_at), verification acceptance and rejection with/without clock skew, admin GET/PATCH behavior, per-client override precedence, and malformed tokens (missing iat/exp -> rejection with "MALFORMED_TOKEN"); tests are included in CI.

## Edge Cases
   * What happens when clock drift exceeds configured tolerance?
     - Verification fails with HTTP 401 and error_code "TOKEN_CLOCK_DRIFT_TOO_LARGE" (or corresponding TOKEN_EXPIRED/TOKEN_NOT_YET_VALID), logs include request_id and measured drift, and the incident is surfaced for operators to decide on increasing clock_skew_seconds or addressing client/system clocks.

   * How does system handle tokens missing exp or iat claims?
     - Reject token verification with HTTP 401 and error_code "MALFORMED_TOKEN". Do not accept tokens without required timing claims.

   * How does system validate admin-supplied config values?
     - PATCH requests are validated: TTLs must be positive integers, max_ttl_seconds must be >= default_ttl_seconds, clock_skew_seconds must be a non-negative integer and within an acceptable ceiling (e.g., <= 86400). Invalid inputs return HTTP 400 with field-level errors.

   * What happens if per-client TTL would exceed global max?
     - Cap effective TTL at global max and record an audit log entry; do not return capped details to end-user responses.

   * What happens when config persistence (DB/kv) is unavailable?
     - Respond to admin update with HTTP 503 and error_code "CONFIG_PERSISTENCE_ERROR"; do not apply partial changes. System continues operating with in-memory config until persistence is restored or an operator resolves the issue.

   * How are changes propagated in multi-instance deployments?
     - Changes are written to the central store and instances subscribe/poll to pick up updates; if an instance misses updates it must log and keep last-known-good settings rather than crash.

   * How does token expiry interplay with key rotation and revocation?
     - Verification considers token exp/iat and clock_skew only; key rotation is handled by EP-006 (signing keys) via kid header and prior-valid keys; immediate invalidation requires revocation story dependency.
