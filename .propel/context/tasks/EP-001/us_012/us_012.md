## Story ID

## Story Title
   * Implement Field Validation and Error Contracts

## Description
  * As a backend/API developer, I want field-level input validation on the /auth/login endpoint and a standardized 400/401 error payload contract, so that client applications can reliably surface field-specific errors and consistently handle authentication failures with machine-readable error codes and human-friendly messages.

## Acceptance Criteria
  1. Given a POST /auth/login request with one or more invalid fields (examples: missing required field, email not RFC-compatible, password too short),
     When the API validates the request,
     Then the API returns HTTP 400 with the following JSON payload shape:
     {
       "error_code": "VALIDATION_ERROR",
       "message": "One or more fields are invalid.",
       "errors": [
         {
           "field": "email",
           "rejected_value": "<obfuscated or truncated value or null>",
           "message": "Email must be a valid email address.",
           "code": "INVALID_EMAIL"
         },
         {
           "field": "password",
           "rejected_value": null,
           "message": "Password is required.",
           "code": "REQUIRED"
         }
       ],
       "request_id": "<server-generated id>"
     }
     - The errors array must list every validation failure for predictable client mapping.
     - Sensitive values must not be returned verbatim (rejected_value may be omitted or truncated/obfuscated).

  2. Given a POST /auth/login request with well-formed JSON and valid-looking fields but incorrect credentials,
     When authentication fails,
     Then the API returns HTTP 401 with the following JSON payload shape:
     {
       "error_code": "INVALID_CREDENTIALS",
       "message": "Invalid email or password.",
       "request_id": "<server-generated id>"
     }
     - The message must be non-enumerating (do not indicate whether email exists).
     - Include request_id for client-side correlation and support for observability/logging.

  3. Given a POST /auth/login request with malformed JSON or unsupported Content-Type,
     When the server fails to parse the body,
     Then the API returns HTTP 400 with:
     {
       "error_code": "MALFORMED_REQUEST",
       "message": "Malformed request body. Expected application/json.",
       "request_id": "<server-generated id>"
     }

  4. Given a POST /auth/login request that fails schema constraints due to field length limits (e.g., email > 254 chars),
     When the API enforces configured max lengths,
     Then the API returns HTTP 400 using the validation contract with code "FIELD_TOO_LONG" and indicates the maximum allowed length in the human-readable message only (do not echo the entire rejected value).

  5. Given the validation component encounters an unexpected runtime error,
     When the error surfaces,
     Then the API returns HTTP 500 with:
     {
       "error_code": "INTERNAL_ERROR",
       "message": "An internal error occurred. Please retry later.",
       "request_id": "<server-generated id>"
     }
     - The server must log full diagnostic information tied to request_id for debugging.

## Edge Cases
   * What happens when input fields exceed expected boundaries?
     - System returns HTTP 400 with a validation entry: field name, code "FIELD_TOO_LONG" and a safe human-readable message showing the configured max (e.g., "Email must be 254 characters or fewer"). Do not echo full offending value; if rejected_value is included it must be truncated/obfuscated to a safe max length.
   * How does system handle malformed JSON or unsupported content-types?
     - Return HTTP 400 with error_code "MALFORMED_REQUEST" and a brief message instructing clients to use Content-Type: application/json with valid JSON. Add guidance in API docs and link via request_id where helpful.
   * How does system behave if the validation library/service throws an unexpected exception?
     - Return HTTP 500 with error_code "INTERNAL_ERROR" and a non-sensitive message. Log full error details and request_id to centralized logs/observability. Fail-safe: do not attempt to authenticate when validation pipeline crashed.

Notes on implementation and team guidance:
- Follow the standardized error contract above for all future auth endpoints to ensure client consistency.
- Keep error_code values stable and documented in API reference (e.g., VALIDATION_ERROR, INVALID_CREDENTIALS, MALFORMED_REQUEST, FIELD_TOO_LONG, INTERNAL_ERROR).
- Ensure request_id is included in every error response and correlated with server logs/traces.
- Validation must run before any authentication checks to avoid unnecessary credential lookups and to prevent timing differences that could leak account existence information.
- Tests: unit tests for each validation rule, contract tests asserting response shapes and HTTP status codes, and integration tests to confirm validation short-circuits authentication.
