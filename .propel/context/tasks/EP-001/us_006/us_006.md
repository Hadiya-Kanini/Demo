## Story ID
   * ID: US_006

## Story Title
   * Signing Keys: Secrets Vault Integration

## Description
  * As the authentication service, I want signing keys to be retrieved from a centralized secrets vault with a secure development fallback so that tokens are signed and verified using centrally-managed keys (supporting rotation, KID selection and JWKS exposure) and production secrets are never served from local/dev storage.

## Acceptance Criteria
1. Given the authentication service needs to sign tokens using a named key-id, When the signing component requests the private key from the configured secrets vault, Then the vault client successfully retrieves the private key material and associated metadata (kid, alg, version, activation flag), the service uses the retrieved private key to sign tokens, and an audit event is emitted (operation=key-retrieval, kid, caller-service-id, timestamp) while sensitive key material is never logged.

2. Given the authentication service needs to provide verification keys (public portion) for downstream services, When a JWKS endpoint or internal verification flow requests public keys, Then the service exposes the public JWKs (kid, alg, kty, use=sig) derived from vault keys or cached public material and responds within the configured latency SLA; audit/log events record JWKS-serving actions (kid, timestamp).

3. Given multiple key versions or keys exist in the vault, When issuing a token, Then the service deterministically selects the active signing key (per vault "active" flag or configured key-priority list), includes the selected key's kid in the token header, and stores/records the kid used in the signing audit event so verification can select the matching public key.

4. Given the vault is unreachable or returns transient errors, When the environment is non-production and dev-fallback is explicitly enabled in configuration, Then the service uses a locally stored development signing key (keystore protected on disk or in environment with restricted access), emits an auditable warning (dev-fallback-used, reason, timestamp), and increments a dev-fallback metric. If the environment is production (env=prod) or dev-fallback is disabled, Then the signing attempt fails safely (login/signing request returns HTTP 500 with generic message), an operational alert is raised, and no local key material is used.

5. Given a key has TTL or version lifecycle in vault, When a key is retrieved, Then the client caches the key material in-memory with a configurable TTL and proactively refreshes the key before expiry (refresh window configurable), exposes cache metrics (cache_hit, cache_miss, cache_eviction), and logs refresh/invalidation events.

6. Given a signing or verification operation is requested, When the vault returns malformed/unsupported key material, Then the service rejects the operation, logs an audit event (kid if available, failure-reason), returns an appropriate error (login: HTTP 500 generic "internal error"; verification endpoints: HTTP 503 or 500 as appropriate), and increments a vault_key_error metric.

7. Given vault access requires credentials (token, mTLS cert, cloud-managed identity), When the service is configured, Then the vault client authenticates using the configured secure auth method (no plaintext credentials in repo), and any authentication failures produce an auditable error and are surfaced to ops via alerts and metric (vault_auth_failure).

8. Given automated tests run in CI, When unit/integration tests require signing keys, Then tests may use a vault test stub or an encrypted dev key fixture; CI must not use production vault credentials and must assert that dev-fallback is not enabled for prod-like pipelines.

## Edge Cases
* What happens when the vault denies access to the requested key (403)?
  - Handling: Do not silently fallback to other keys. Fail signing with a safe error (login: HTTP 500 with generic message), emit "unauthorized-vault-access" audit event including caller-service-id and key-id, and raise an operational alert for immediate remediation.

* How does the system handle ambiguous/duplicate active keys?
  - Handling: Follow deterministic selection using explicit key-priority configuration. If ambiguous and no priority is configured, fail fast, emit config-alert and audit entry; avoid random selection.

* What happens if a key is rotated mid-operation (race condition)?
  - Handling: Either use the version read and record the kid/version used, or detect version inconsistency on use and retry fetch once with exponential backoff. Log both read and use versions for traceability and alert on repeated inconsistencies.

* What happens if dev-fallback is accidentally enabled in production?
  - Handling: Deployment validation must reject such config and fail startup. Production must never use local/dev keys. If detected at runtime, treat as high-severity incident: disable signing, emit alert, and fail safe.

* How are rate-limiting and transient vault errors handled?
  - Handling: Implement retry/backoff and a circuit-breaker. On sustained failures, return HTTP 503 for signing-reliant endpoints and emit metrics and alerts.

* What happens if the vault returns an unsupported key algorithm or malformed key?
  - Handling: Reject operation, log algorithm-mismatch audit event, return generic internal error to callers, and raise an alert for ops to remediate key metadata.

* How is caching invalidation handled when keys are rotated in the vault?
  - Handling: Cache TTL + proactive refresh. Invalidate cache on explicit vault notification (if vault supports), on key retrieval error indicating rotation, or when refresh fails; log invalidation event and attempt fresh retrieval.

* What happens if a verification request references a kid that is unknown in the current JWKS?
  - Handling: Attempt to fetch the public key from vault; if not found, return verification failure (token invalid), log an audit event (kid-not-found), and increment a verification-failure metric.
