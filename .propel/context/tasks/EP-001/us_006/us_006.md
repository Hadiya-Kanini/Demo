## Story ID
   * ID Format: US_006
   * ID: us_006

## Story Title
   * Add RBAC Middleware

## Description
  * As a backend developer, I want RBAC middleware that enforces role-based permission checks on protected endpoints so that unauthorized access is consistently prevented and clients receive clear, non-revealing error responses.

## Acceptance Criteria
1. Given a protected endpoint that requires permission "resource.read", When a request includes a valid bearer token whose user roles include that permission, Then the middleware allows the request to proceed to the handler, attaches the user's identity and roles to the request context, emits an access-granted audit log with correlation-id, and middleware processing adds less than 100ms latency for 95% of requests.

2. Given a protected endpoint that requires permission "resource.write", When a request includes a valid bearer token but the user's roles do NOT include the required permission, Then the middleware responds with HTTP 403 and a structured JSON response:
   { "error": "insufficient_permissions", "message": "Access denied", "requestId": "<correlation-id>" },
   and logs the denied attempt without revealing resource existence or internal policy logic.

3. Given a request to a protected endpoint with a missing, malformed, or expired token, When the middleware validates the request, Then it responds with HTTP 401, sets the appropriate WWW-Authenticate header (e.g., Bearer realm="api"), returns { "error": "unauthorized", "message": "Authentication required", "requestId": "<correlation-id>" }, and increments authentication-failure metrics.

4. Given that role/permission resolution depends on an external permission store or cache, When that store is unavailable or returns an error/timeout, Then the middleware denies access by default, responds with HTTP 503 and JSON:
   { "error": "service_unavailable", "message": "Authorization service unavailable", "requestId": "<correlation-id>" },
   logs the internal failure with diagnostic details (not exposed to client), and emits an alert to monitoring.

5. Given protected endpoints configured with granular permissions at route or action level, When middleware is applied to a route, Then it supports route-level configuration (required permission(s) specified via attribute/metadata) and enforces all listed permissions (logical AND where configured; logical OR supported via explicit policy definition), with clear failure reason recorded in logs.

## Edge Cases
* What happens when a user has multiple roles with overlapping and conflicting permissions?
  - Behavior: Compute the union of permissions across all assigned roles. If any assigned role grants the required permission, allow access. Log role combinations that create policy conflicts for audit.
* How does the system handle token expiry or revocation mid-session?
  - Behavior: Expired or revoked tokens are treated as invalid and return HTTP 401. Middleware must check token expiry locally (JWT exp) and consult revocation list or introspection where required. If revocation check is temporarily unavailable, deny-by-default and return HTTP 503.
* What happens when permission configuration for an endpoint is missing or malformed?
  - Behavior: Deny access by default (fail-closed), return HTTP 403 with generic message, emit high-severity log entry and generate an operational incident for platform team to correct configuration.
