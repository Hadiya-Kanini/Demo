## Story ID: us_006

## Story Title: Implement POST /auth/logout & Revocation

## Description: As a logged-in API client (authenticated user or client app), I want to call POST /auth/logout to revoke my active access and refresh tokens, so that my tokens cannot be used after logout and the system rejects revoked tokens for improved security and session control.

## Acceptance Criteria
1. Given a valid access token presented in the Authorization header, When the client calls POST /auth/logout, Then the system marks the access token as revoked (by jti or token identifier) and any associated refresh token(s) as revoked, persists revocation with TTL equal to remaining token lifetime, and returns HTTP 200 with a generic success message (no sensitive details).

2. Given a token that has been revoked via logout, When any request is made to a protected endpoint using that token, Then the authentication middleware rejects the request with HTTP 401 Unauthorized and a non-revealing error ("invalid or expired token"), and the request is logged for audit.

3. Given a logout request where the access token is expired, missing, or already revoked, When the client calls POST /auth/logout, Then the endpoint responds idempotently with HTTP 200 and the same generic success message; no internal error is exposed and no duplicate revocation records cause failure.

4. Given the client passes a refresh token in the logout request body or cookie (if refresh-token support is used), When POST /auth/logout is processed, Then the refresh token is also revoked (so future refresh attempts fail), and the response confirms revocation via generic success message.

## Edge Cases
- Token replay/race: If two concurrent logout requests attempt to revoke the same token, handle idempotently by upserting revocation entry (or using Redis SET with NX semantics + TTL). Do not return error on conflicts; return HTTP 200 and ensure a single revocation state is persisted.
- Stateless JWTs without server-side storage: If tokens are purely stateless JWTs, require jti (token id) claim and a revocation list (fast store, e.g., Redis) to record revoked jtis. If jti missing, reject logout with HTTP 400 and log for developer action (or require token rotation/refresh implementation).
- Revocation persistence vs token expiry: If revocation store entry TTL is shorter than remaining token lifetime due to misconfiguration, ensure fallback: compute TTL from token expiry (exp claim) and set revocation entry TTL accordingly; if expiry cannot be determined, set conservative TTL equal to configured max access token lifetime.
