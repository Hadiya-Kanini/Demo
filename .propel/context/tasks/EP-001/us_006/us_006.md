## Story ID
   * ID: US_006

## Story Title
   * Enforce RBAC and Role-Based Redirects

## Description
  * As an authenticated user, I want the system to enforce role-based access on protected endpoints and redirect me to a role-specific dashboard after login so that unauthorized access is prevented and users land directly on the interface relevant to their role.

## Acceptance Criteria
1. Given a request to a protected endpoint with a valid access token containing the user's role claim, When the middleware evaluates the token and the role matches the endpoint's required role, Then the request is allowed to proceed to the downstream handler and the middleware records an access-granted audit event (user id, endpoint, role, timestamp).

2. Given a request to a protected endpoint with a valid access token whose role does NOT meet the endpoint's required role, When the middleware evaluates the token, Then the middleware responds with HTTP 403 Forbidden with a non-revealing message ("Access denied"), writes an access-denied audit event (user id, attempted endpoint, required role, timestamp), and increments the RBAC denied metric.

3. Given a successful login with valid credentials, When the authentication service issues the session/token, Then the system resolves the user's effective role (using role claims and server-side precedence), issues a fresh session/token (rotated session id), and either issues an HTTP 302 redirect to /dashboard/{resolved-role} for browser flows or returns a login response payload containing nextUrl=/dashboard/{resolved-role} for API/SPA flows.

4. Given a request with an expired, revoked, or otherwise invalid token, When the middleware validates the token, Then the middleware responds with HTTP 401 Unauthorized with a generic message ("Authentication required"), does not reveal role or internal details, and emits a token-validation-failure audit event.

5. Given a user with multiple role claims, When the middleware must determine access or redirect target, Then it resolves the effective role using the configured role-precedence list (stored in config) and applies access/redirect behavior based on that resolved role; if precedence config is absent, the system uses a safe default role and raises a config alert.

## Edge Cases
* What happens when a token is missing or has a malformed role claim?
  - Handling: Treat as invalid for authorization. Middleware returns HTTP 401 Unauthorized, logs the incident (user id if available, token id, failure reason), and does not perform a role-based redirect. A user-facing message should be generic and non-revealing. Create follow-up task to expose a profile-completion path for users missing role assignments.

* How does the system handle users with multiple roles (role collision)?
  - Handling: Resolve effective role deterministically using a server-side configurable role-precedence list (e.g., Admin > Manager > Editor > User). Document precedence in configuration. If two roles map to different dashboards, choose dashboard of highest-precedence role. Log the resolution decision in audit logs for traceability.

* What happens if the role -> dashboard mapping is missing or the dashboard route returns 404?
  - Handling: Fall back to a safe default dashboard (/dashboard/user) and emit an operational alert/log indicating the missing mapping (role id, attempted redirect path). Monitor this metric and surface to SRE/ops for fix.
