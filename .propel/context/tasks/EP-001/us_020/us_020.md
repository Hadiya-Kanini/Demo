## Story Title
   * Security Hardening: Session/Cookie Best Practices

## Description
  * As a Security Engineer / Backend Developer, I want session and cookie handling to follow modern security best practices (HttpOnly, Secure, SameSite, expiration alignment) and to provide documented guidance for clients using cookies, so that authentication cookies are resistant to theft and CSRF, compatible with browser restrictions, and clearly understood by integrators.

## Acceptance Criteria
  * Given a successful browser login, When the server issues a session cookie, Then the response MUST include a Set-Cookie header with: HttpOnly; Path=/; SameSite default=Lax (configurable to Strict or None); Secure flag set when connection is TLS/production; Expires/Max-Age aligned with issued token exp claim.
  * Given a non-TLS (development) environment, When the server issues a cookie, Then the Secure flag MUST NOT be set and the system MUST emit a clear developer warning in logs and in a debug response header indicating cookies are insecure; production deployments MUST require TLS and refuse to issue non-Secure auth cookies.
  * Given a client that requests token-only delivery (e.g., API client or native/mobile), When the client indicates opt-out via a documented request parameter or header (e.g., Prefer: no-cookie or ?cookie=false), Then the server MUST NOT set an authentication cookie and MUST return the token in the response body (JSON) and document the exact response schema.
  * Given cookie-based authentication is provided, When engineers consult the API documentation, Then the documentation MUST include: cookie name and recommended entropy/length; explicit behavior of HttpOnly, Secure, SameSite in prod vs dev; CSRF mitigation guidance (anti-CSRF tokens, SameSite recommendations); subdomain and domain handling guidance (Domain attribute); guidance for SameSite=None use and requirement for Secure; mobile/native guidance to use Authorization header; examples and sample responses showing Set-Cookie header and JSON token responses.
  * Given a request arrives with an expired or invalid token stored in cookie, When server validates the cookie, Then server MUST return the standard token-expired error code and JSON body defined under EP-001, clear any short-lived cookie values as appropriate (Set-Cookie with Max-Age=0), and record an audit/log event for the invalid/expired cookie evaluation.

## Edge Cases
   * What happens when browser blocks third-party cookies or has tracking protection enabled?
     - Detect the login flow failure from known response patterns or client-side signals; document alternative flows: use Authorization Bearer header for API/native clients, or use SameSite=None with Secure only for explicit third-party embedding after evaluating CSRF risk. Include troubleshooting guidance in docs.
   * How does system handle insecure (HTTP) clients vs secure (HTTPS) clients?
     - In development (non-production) HTTP, allow cookies without Secure flag but log warnings and flag in test assertions. In production, refuse to issue authentication cookies without TLS; CI/CD checks should prevent deployment misconfiguration.
   * What happens when the cookie payload would exceed browser limits or token grows large?
     - Enforce a server-side maximum cookie payload (e.g., 4096 bytes); if exceeded, return 400 with a documented error code and recommend use of token-in-body or compressed token strategies. Document payload limits and mitigation steps.

Notes / Implementation Guidance (non-normative)
- Default SameSite value is Lax for a good balance between security and compatibility; teams may set to Strict for highest protection or None (with Secure) for cross-site integration — this must be configurable per-deployment.
- Ensure Set-Cookie header values conform to RFC6265bis recommendations and modern browser requirements (SameSite=None requires Secure).
- Provide an API-level toggle or request parameter to opt-out of cookie issuance for programmatic clients; do not rely on user-agent sniffing.
- Add automated tests that assert Set-Cookie header contains required attributes under TLS and that production configuration disallows issuing non-Secure auth cookies.
- Add a documentation page under API docs (dependency: us_007) that includes examples, debugging tips, and migration notes for integrators.
