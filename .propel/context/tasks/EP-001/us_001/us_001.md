## Story ID
   * ID: US_001

## Story Title
   * Implement Login API (POST /auth/login)

## Description
  * As a registered user, I want to POST my credentials to /auth/login and receive an access token so that I can authenticate to the API and access protected resources according to my role.

## Acceptance Criteria
1. Given a registered user with valid credentials, When the client sends a POST /auth/login with a JSON payload:
   {
     "username": "user@example.com",
     "password": "correct-password"
   }
   Then the server responds with HTTP 200 and a JSON body:
   {
     "access_token": "<JWT>",
     "token_type": "Bearer",
     "expires_in": 3600
   }
   - The token is signed with the system's configured signing key, contains standard claims (sub, iat, exp, roles), and expires after the configured TTL.
   - The response body contains no stack traces or internal error details.

2. Given a user supplies incorrect credentials, When the client POSTs to /auth/login, Then the server responds with HTTP 401 and a generic error message:
   {
     "error": "Invalid credentials"
   }
   - Failed login attempt is recorded (user identifier or IP) for rate-limiting and monitoring.
   - Response time and payload size remain within acceptable limits to avoid timing/side-channel leaks.

3. Given malformed input (missing fields, invalid JSON, invalid email format), When the client POSTs to /auth/login, Then the server responds with HTTP 400 with a validation error that identifies invalid fields (e.g., "password: required", "username: invalid email format") and does not proceed to password verification.

4. Given repeated failed attempts that exceed configured thresholds, When threshold is reached, Then subsequent requests return HTTP 429 (or 423 if account locked) with a generic message ("Too many attempts, try again later"), and the system enforces backoff/lockout per security policy. Alerts and audit logs must be created for suspected brute-force activity.

5. Given a successful login, When the server issues a token, Then an audit record is created (user id, timestamp, client IP, user-agent), and integration tests demonstrate median authentication latency meets NFR (auth median < 1s) under standard load.

## Edge Cases
* What happens when the database or user store is unavailable?
  - Handling: Return HTTP 503 Service Unavailable with message "Service temporarily unavailable, please try again later". Do not expose internal errors. Trigger an incident metric and circuit-breaker behavior to prevent cascading failures.

* How does the system handle legacy password hashes or algorithm upgrades?
  - Handling: Detect legacy hash on verification; if legacy verification succeeds, issue token and asynchronously replace stored password hash with the current recommended algorithm (e.g., Argon2/bcrypt) during the same request flow or a background job. Log the re-hash event. Maintain identical API responses to avoid revealing hash type.

* What happens when a user attempts login during/after account lockout or multi-factor enforcement?
  - Handling: If account is locked due to policy, return HTTP 423 with a generic message. If policy requires MFA and the user has not completed it, return HTTP 202 with a next-step payload describing required second factor step (only if MFA flow implemented in scope) or return HTTP 403 "Additional authentication required" if not in scope. (If MFA is [UNCLEAR], mark as dependency/clarification.)
