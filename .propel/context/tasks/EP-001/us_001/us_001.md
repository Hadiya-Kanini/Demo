## Story ID
   * ID: US_001

## Story Title
   * Login API (POST /auth/login)

## Description
  * As a registered user, I want to POST my credentials to /auth/login and receive a signed access token (JWT) with expiry metadata so that I can authenticate to the API and access protected resources according to my role.

## Acceptance Criteria
1. Given a registered user with valid credentials, When the client sends a POST /auth/login with a JSON payload:
   {
     "username": "user@example.com",
     "password": "correct-password"
   }
   Then the server responds with HTTP 200 and a JSON body:
   {
     "access_token": "<JWT>",
     "token_type": "Bearer",
     "expires_in": 3600,
     "expires_at": "2026-04-06T12:34:56Z"
   }
   - The token is signed with the system's configured signing key (retrieved from secrets management per EP-006).
   - The token contains standard claims: sub (user id), iat, exp, and authorization claims (roles). exp equals iat + configured TTL.
   - The response body contains no stack traces, internal error messages, or implementation details.

2. Given a user supplies incorrect credentials, When the client POSTs to /auth/login, Then the server responds with HTTP 401 and a generic error message:
   {
     "error": "Invalid credentials"
   }
   - The failed login attempt is recorded (user identifier and/or IP) for rate-limiting, monitoring, and potential lockout.
   - Response timing and payload size are controlled to reduce timing/side-channel information leakage.

3. Given malformed input (missing fields, invalid JSON, invalid email format), When the client POSTs to /auth/login, Then the server responds with HTTP 400 with a validation error that identifies invalid fields (e.g., "password: required", "username: invalid email format") and the server does not perform password verification.

4. Given repeated failed attempts that exceed configured thresholds, When threshold is reached, Then subsequent requests return HTTP 429 (or HTTP 423 if account locked) with a generic message ("Too many attempts, try again later"), and the system enforces backoff/lockout per configured security policy. Alerts and audit logs must be created for suspected brute-force activity.

5. Given a successful login, When the server issues a token, Then an audit record is created containing (user id, timestamp, client IP, user-agent, token id/nonce), structured metrics are emitted (auth.success, auth.latency), and integration/performance tests demonstrate median authentication latency meets NFR (auth median < 1s) under standard load.

6. Given the signing key or secrets vault is unavailable at token-issuance time, When a login request requires token creation, Then the server returns HTTP 503 Service Unavailable with a non-revealing retryable message and logs the event for incident response; the system must not fall back to insecure local signing.

7. Given a legacy password-hash record is verified successfully, When the login succeeds, Then the server issues the normal response and triggers a secure re-hash (in-band or background job) to migrate the stored hash to the current recommended algorithm, and records the re-hash event in audit logs without exposing hash details in responses.

## Edge Cases
* What happens when the database or user store is unavailable?
  - Handling: Return HTTP 503 Service Unavailable with message "Service temporarily unavailable, please try again later". Do not expose internal errors. Trigger an incident metric and circuit-breaker behavior to prevent cascading failures.

* How does the system handle legacy password hashes or algorithm upgrades?
  - Handling: Detect legacy hash on verification; if legacy verification succeeds, issue token and asynchronously replace stored password hash with the current recommended algorithm (e.g., Argon2/bcrypt) during the same request flow or a background job. Log the re-hash event. Keep API responses identical to avoid revealing hash type.

* What happens when signing key or secrets management service is unavailable?
  - Handling: Return HTTP 503 Service Unavailable with a non-revealing message. Do not use insecure fallbacks. Emit high-priority alert for on-call.

* What happens when a user attempts login during/after account lockout or multi-factor enforcement?
  - Handling: If account is locked due to policy, return HTTP 423 with a generic message. If policy requires MFA and the user has not completed it, return HTTP 202 with a next-step payload describing required second factor step (only if MFA flow implemented in scope) or return HTTP 403 "Additional authentication required" if MFA flow is out-of-scope. Mark MFA as dependency/clarification if not implemented.

* How are timing/side-channel attacks mitigated?
  - Handling: Use constant-time password comparison, normalize response timing/payload size where feasible, and do not reveal user existence in error messages.

* What about signing key rotation and verification of recently-rotated keys?
  - Handling: Token issuance uses current key; verification accepts keys within a configured rotation window (handled by verification middleware / separate story). If rotation prevents issuance, return HTTP 503 and log incident.

* High concurrency re-hash/update race conditions
  - Handling: Ensure re-hash migration is idempotent or guarded by a single-writer/update strategy to avoid corrupt or inconsistent password records.
