## Story Title
   * Implement Token Verification and Expiry Handling

## Description
  * As a backend authentication service developer, I want server-side token verification and expiry (exp/iat) handling implemented and covered by automated tests so that only valid, non-expired tokens are accepted, clients receive consistent error responses for invalid/expired tokens, and the system meets security and observability requirements.

## Acceptance Criteria
  * Given a valid JWT signed with the active signing key and exp claim in the future, When the token verification endpoint or middleware processes the token, Then the verification succeeds, the returned principal includes expected claims (sub, exp, iat, aud, scope), and the request is allowed to proceed.
  * Given a JWT where exp is in the past, When the token verification endpoint or middleware processes the token, Then verification fails with HTTP 401 (Unauthorized) and a machine-parseable error indicating token_expired; a server-side audit log entry is created with non-sensitive context and rejection reason.
  * Given a JWT with iat set to a time in the future beyond configured clock skew tolerance, When the token verification logic processes the token, Then verification fails with HTTP 401 and error token_invalid_iat; logs include the token id or kid (if present) and configured skew used for the decision.
  * Given a JWT with an invalid signature, unknown kid, or malformed structure, When verification is attempted, Then verification fails with HTTP 401 and error token_invalid_signature (or token_malformed for parse errors); the response body is generic (no sensitive diagnostic info), and the verification failure increments observability metrics (auth.token.verification.failures).
  * Given a token signed with a recently rotated (previous) key and the key rotation policy allows a grace period, When verification is attempted, Then the system validates using the appropriate trusted key set (current + allowed previous keys) and either accepts or rejects according to policy; the event is logged with rotation key id and metric auth.token.verification.rotation_used.

## Edge Cases
   * What happens when system clock skew exists between issuer and verifier?
     - Handling: Apply configurable clock skew tolerance (default 120 seconds). Use skew both when evaluating exp (treat token as expired if exp < now - skew) and iat (reject if iat > now + skew). Expose skew as runtime config and include it in debug logs (without sensitive token contents) for reproducibility.
   * How does system handle key rotation or missing signing keys?
     - Handling: Use kid header to select key from a local JWKS cache or vault-backed key store. If kid missing, try default key(s). Support validation against a small set of trusted previous keys for a configurable grace period; after grace period, old keys are removed. If no key validates, return 401 token_invalid_signature and emit an alert and metric. Ensure sensitive key material is never logged.
   * What happens when token claims are missing or malformed (e.g., missing exp or non-integer exp)?
     - Handling: Treat token as invalid; return 401 token_malformed_claims. Record structured log with claim parsing error type and non-sensitive context. Ensure tests cover JSON and numeric encoding edge cases (large integers, stringified numbers).
