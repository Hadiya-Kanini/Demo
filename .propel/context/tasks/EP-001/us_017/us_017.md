## Story ID

   * Story ID: us_017

## Story Title
   * Signing Key Rotation & Backwards Compatibility

## Description
  * As a backend/system engineer, I want a signing key rotation strategy that supports multiple active verification keys and automated rotation with minimal downtime, so that issued tokens remain verifiable across rotations and the system maintains security and availability.

## Acceptance Criteria
  * Given the auth service has a set of active signing keys (including one primary), When the system issues a new token, Then the token MUST be signed with the current primary key and include a 'kid' header that identifies that key.

  * Given a token signed by any active (non-revoked) key and presented to a protected API, When the token is valid (signature and claims valid, exp/iat within configured tolerance), Then verification MUST succeed using the corresponding active key and the request MUST be authorized.

  * Given a scheduled automatic rotation process with a configured retention (grace) period, When rotation completes, Then the system MUST: (a) promote the newly provisioned key to primary for signing new tokens, (b) retain previous keys for verification until the retention window elapses (configurable), and (c) ensure tokens issued before rotation remain verifiable until expiry or revocation.

  * Given the secrets vault is unavailable during an attempted rotation, When rotation fails, Then the system MUST continue to verify existing tokens using currently-active keys, schedule retries with exponential backoff and jitter, emit an alert to monitoring/incident channels, and prevent issuance of new tokens only if no signing key is locally available (configurable fallback policy must be documented).

  * Given a multi-instance deployment, When rotation is initiated, Then the promotion of a new primary key MUST be coordinated using an atomic metadata change or distributed lock so that verification endpoints do not reject valid tokens during the transition; acceptance test must demonstrate zero verification failures attributable to rotation.

## Edge Cases
   * What happens when a token arrives with an unknown or missing 'kid'?
     - The system MUST attempt to verify the signature by iterating active keys (for backward compatibility). If none verify, it MUST return HTTP 401 Unauthorized with a machine-readable error code "ERR_INVALID_SIGNING_KEY" and include an audit log entry with the presented kid (if present) and request metadata for forensic analysis.

   * How does the system handle a partial/premature key removal or mid-rotation failure?
     - Rotation must follow an atomic publish approach: prepare (provision new key), distribute (ensure other instances can load new key), and commit (flip primary pointer). If any step fails, the system MUST abort commit and roll back to the previous stable state. No verification key may be removed until the retention window ends or keys are explicitly revoked.

   * How are compromised keys handled?
     - Provide an immediate revocation path: mark key as 'revoked' in key metadata, propagate revocation to all instances, and reject any tokens signed by revoked keys (return 401 with error code "ERR_REVOKED_SIGNING_KEY"). Revocation events MUST be auditable and trigger security notifications.

Notes:
- Implementation guidance (non-prescriptive): tokens SHOULD include a standard 'kid' header; key metadata MUST include id, public key/material, creation_ts, status (active/revoked), and retention_until. Rotation MUST be configurable (rotation schedule, retention/grace window, max active keys) and support manual override. Systems MUST integrate with secrets vault (see dependency us_006) and include a safe dev fallback for local testing only. Acceptance tests (us_009) must include rotation scenarios, unknown-kid, revoked-key, vault-unavailable, and multi-instance coordination verification.
