## Story ID
* ID: US_002

## Story Title
* Login Field-level Input Validation & 400 Error Contract

## Description
* As a client of the Authentication API, I want /auth/login requests to be validated at field-level and to receive standardized, machine-readable 400 responses with error codes for validation failures, so that clients can surface precise field errors to users and reliably handle error conditions programmatically.

## Acceptance Criteria
1. Given a POST /auth/login request with required fields present, syntactically valid JSON, correct Content-Type: application/json, and values that meet configured constraints, When the request is received, Then the validation layer does not return HTTP 400 and the request is forwarded to the authentication processing flow; validation success is recorded in logs/metrics.

2. Given a POST /auth/login request missing required fields (e.g., missing "email" or "password"), When the request is received, Then the service returns HTTP 400 with a standardized JSON error body containing:
   - top-level "error.code": "ERR_MISSING_FIELD"
   - "error.message": brief human-friendly summary
   - "error.request_id": uuid for tracing
   - "error.fields": array of field error objects each with "name", "code" (e.g., "ERR_MISSING_EMAIL"), and "message".
   The response must not include sensitive request data.

3. Given a POST /auth/login request with invalid field formats (e.g., malformed email, password too short), When the request is received, Then the service returns HTTP 400 with "error.code": "ERR_INVALID_FIELD", includes per-field entries such as "ERR_INVALID_EMAIL" or "ERR_INVALID_PASSWORD" in "error.fields", and provides human-friendly messages and request_id for correlation.

4. Given a POST /auth/login request with malformed JSON, When the request is received, Then the service returns HTTP 400 with "error.code": "ERR_MALFORMED_JSON" and a generic message (do not echo the malformed payload), and includes a request_id.

5. Given a POST /auth/login request with an unsupported Content-Type (not application/json), When the request is received, Then the service returns HTTP 415 with "error.code": "ERR_UNSUPPORTED_MEDIA_TYPE" and guidance to use application/json.

6. Given a POST /auth/login request that triggers multiple validation failures, When the request is received, Then the service returns a single aggregated HTTP 400 response that lists all field-level failures in "error.fields"; the service may truncate the list to a configured maximum and indicate truncation in metadata.

7. Given excessive repeated invalid validation attempts from the same client/IP, When thresholds configured by platform are exceeded, Then the platform may respond with rate-limiting (HTTP 429) per policy; such throttling must not reveal credential state and must preserve the validation error contract for subsequent allowed requests.

## Edge Cases
* What happens when the request payload is extremely large (exceeds configured maximum size)?
  - Handling: Early reject with HTTP 413 (Payload Too Large) and "ERR_PAYLOAD_TOO_LARGE". Log size and request_id; do not attempt full validation.

* How does the system handle extra/unexpected fields in the JSON body?
  - Handling: Default behavior is to ignore unknown properties and proceed with validation of expected fields. Optionally include warnings (code "ERR_UNKNOWN_FIELD") if API policy enables warnings.

* What happens with international/unicode input?
  - Handling: Apply normalization (NFC) where applicable for format checks (e.g., email validation). Password bytes must be preserved exactly for authentication; only apply format validation based on configured rules.

* How are aggregated errors limited or ordered?
  - Handling: Return up to a configured maximum number of field errors (e.g., 20). If truncated, include "truncated": true in the error metadata.

* How to handle conflicts between validation and authentication outcomes?
  - Handling: Validation-only failures return 400 with validation codes. If validation passes but authentication fails, return authentication-specific responses (e.g., 401 with "ERR_AUTH_FAILED") as defined in the authentication story to avoid leaking existence of accounts.

* What if the validation subsystem fails catastrophically?
  - Handling: Return HTTP 500 with a generic error and request_id, emit operational alert; do not proceed to authentication until validation is healthy.
