## Story ID
* ID: US_002

## Story Title
* Build Login UI

## Description
* As a returning user, I want an accessible, validated client-side login form so that I can securely sign in to my account with clear feedback and support for keyboard and screen-reader users.

## Acceptance Criteria
1. Given I am on the login page, When I enter a valid email and password and click "Sign in" (or press Enter), Then the client:
   - Performs client-side validation,
   - Disables the submit control and shows a visible loading indicator,
   - Sends a POST to /auth/login with the email and password (over HTTPS),
   - On success receives tokens and redirects according to role (server-driven),
   - On success ensures focus is moved to the primary content of the landing page.

2. Given I focus the first input on page load, When I navigate using Tab/Shift+Tab and keyboard controls, Then:
   - All interactive elements are reachable and operable via keyboard,
   - Focus order is email -> password -> show/hide password -> "Remember me" (if present) -> submit,
   - Visible focus indicators are present and meet contrast requirements.

3. Given invalid client-side input (empty or malformed email; empty password), When I attempt to submit, Then:
   - Submission is prevented,
   - An inline error message appears adjacent to each invalid field,
   - Each inline error is connected to the input via aria-describedby and is announced by screen readers (aria-live polite),
   - Submit remains disabled until errors are resolved.

4. Given the server responds with authentication failure (e.g., 401 invalid credentials), When the response is received, Then:
   - The UI displays a non-revealing error summary ("Invalid email or password") in an aria-live assertive region,
   - The error summary is keyboard-focusable and receives programmatic focus,
   - Inputs are shown in an error visual state (border + aria-invalid),
   - No detailed server error details are exposed to the user.

5. Given a network error or server 5xx, When submission fails, Then:
   - The UI displays an accessible error banner with a retry action,
   - The form remains editable (submit re-enabled) after retry cancellation,
   - An instrumentation event is emitted for monitoring/tracing.

## Edge Cases
* What happens when the server is slow or times out?
  - After 2 seconds of pending submission show a secondary loading message ("Still working..."), keep submit disabled and provide a "Cancel" control. If the user cancels, abort the request and return focus to the email field. Preserve no sensitive data in logs.

* How does the system handle password-manager autofill and browser autofill?
  - Autofill must populate fields and trigger validation: implement onpage/readiness checks to detect autofill and run the same validation/aria announcements as manual input. Labels must be visible (not placeholder-only) to avoid loss of context. Ensure compatibility with common password managers and preserve accessibility hooks.

* What happens after multiple failed login attempts?
  - After 5 failed attempts within 15 minutes, surface a generic message that the account may be temporarily locked and provide "Forgot password" help. Do not confirm account existence or divulge timing details. The UI should continue to accept submissions but display appropriate messaging; further lockout enforcement is handled server-side.
