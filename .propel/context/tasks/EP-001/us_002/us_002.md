## Story ID: us_002

## Story Title: Implement POST /auth/login

## Description: As a registered user, I want to authenticate by posting my credentials to /auth/login, so that I can receive a secure access token and access protected resources in the application.

## Acceptance Criteria (Given/When/Then format, 3-5 criteria)
1. Given a valid email and password for an active account, When I POST /auth/login with JSON { "email": "<email>", "password": "<password>" }, Then the API returns 200 OK with a JSON payload containing an access token (JWT), token_type ("Bearer"), expires_in (seconds), and minimal user metadata (id, email, roles); no password or secret data is returned; access token expiry matches configured TTL.

2. Given invalid credentials (wrong email or password), When I POST /auth/login, Then the API returns 401 Unauthorized with a generic error message ("Invalid credentials"), issues no tokens, and does not indicate whether the email or password was incorrect.

3. Given malformed or missing input (e.g., invalid email format, password too short, missing fields), When I POST /auth/login, Then the API returns 400 Bad Request with structured validation errors that indicate which fields failed validation (field name and constraint), and no authentication attempt is processed.

4. Given multiple failed login attempts exceeding configured thresholds (e.g., 5 failures per account or IP in 15 minutes), When further POST /auth/login requests occur, Then the API enforces rate limiting/account lockout behavior (429 Too Many Requests or 403 Locked with a non-revealing message), logs the event, and requires the configured cooldown or unlock flow before allowing further attempts.
