## Story Title
   * Refresh Token Flow with Rotation

## Description
  * As a system architect/engineer responsible for authentication, I want a secure refresh token flow with rotation so that clients can renew sessions without re-entering credentials while minimizing risk from stolen refresh tokens and enabling detection/response to token theft.

## Acceptance Criteria
  * Given a valid-but-expired access token and a valid refresh token that has not been previously rotated or revoked, When the client calls POST /auth/refresh with the refresh token, Then the server:
    - Validates the token signature and token state in the refresh-token store,
    - Issues a new access token (signed, containing claims sub, roles, iat, exp) and a new refresh token,
    - Persists the new refresh token state, marks the previous refresh token record as rotated/inactive (with timestamp and previous_jti reference),
    - Returns HTTP 200 with JSON: { "access_token": "...", "refresh_token": "...", "token_type": "Bearer", "expires_at": "<ISO-8601 UTC>" }.

  * Given the same refresh token is submitted concurrently across multiple requests, When those requests are processed, Then only one request succeeds and returns new tokens (HTTP 200); all other concurrent attempts are rejected with HTTP 401 and a consistent response body containing error_code: REFRESH_TOKEN_REPLAY and message describing token reuse, and only one persisted rotated/new-token pair exists.

  * Given a request contains a refresh token that has already been rotated and therefore reused (replay), When the server detects reuse, Then the server:
    - Marks the user's session as compromised and revokes all refresh tokens for that user/session (per revocation policy),
    - Optionally revokes active access tokens if revocation store supports it,
    - Emits an audit log event with user_id, jti(s), request metadata (IP, user-agent), and timestamp,
    - Returns HTTP 401 with error_code: REFRESH_TOKEN_REUSE_DETECTED and instructs client to require full re-authentication.

  * Given a refresh token is expired, malformed, missing, signed with unknown key, or otherwise invalid, When the client attempts refresh, Then the server returns HTTP 401 with error_code: INVALID_REFRESH_TOKEN (or MALFORMED_REFRESH_TOKEN / UNKNOWN_KEY as appropriate), includes no new tokens, and logs the validation failure for diagnostics.

  * Given configurable settings for refresh-token lifetime and clock-skew tolerance, When tokens are validated in automated tests and in production, Then the system uses the configured values (documented in API docs), supports at least a default refresh lifetime and adjustable clock skew, and unit/integration tests verify behavior with clock offsets (+/-) per config.

## Edge Cases
   * What happens when the refresh token store (persistent store/redis) is unavailable?
     - Handling: The system must fail closed. Do not issue new tokens. Return HTTP 503 (or HTTP 401 if security policy dictates) with error_code: REFRESH_STORE_UNAVAILABLE, log detailed error (correlation id), and increment monitoring counter/alert. Short-term retries are allowed only if implemented with idempotent transaction semantics; otherwise surface the outage.

   * How does the system handle concurrent requests using the same refresh token (race conditions)?
     - Handling: Implement atomic rotation using a DB-level transaction, optimistic locking (CAS), or a single-writer pattern to ensure only one request can rotate a token. All other concurrent attempts must be detected and return HTTP 401 with error_code: REFRESH_TOKEN_REPLAY. Add tests that simulate concurrency and verify only one active rotation succeeds.

   * What happens when a rotated refresh token is replayed (indicative of token theft)?
     - Handling: Detect replay via token state store (refresh token marked rotated/inactive). On detection, revoke all active refresh tokens for that user/session, optionally revoke access tokens if possible, emit an audit/security event, notify monitoring/incident pipeline, and require user to re-authenticate. Consider additional controls (e.g., forced password reset) as part of security operations runbook.
