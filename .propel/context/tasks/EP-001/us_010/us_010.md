## Story ID
* ID: us_010

## Story Title
* Create Integration Tests and CI Checks for Auth Flows

## Description
* As a QA/DevOps engineer, I want automated end-to-end and error-path integration tests for login, token expiry, token revocation, and RBAC enforcement and CI pipeline checks that run these tests, so that authentication regressions are detected early and production access remains secure and auditable.

## Acceptance Criteria
1. Given a test environment with the authentication services deployed and deterministic test fixtures (test users, roles, token TTL), When the auth integration test suite runs (local CI or pipeline), Then the happy-path end-to-end flow (POST /auth/login -> receive token -> GET /protected with token -> POST /auth/logout -> token invalidation) passes and the CI job reports success with captured request/response logs and test artifacts.

2. Given a token issued in a controlled test, When the test advances simulated time or issues an expiration event beyond token TTL, Then subsequent requests using the expired token result in 401 Unauthorized with the standardized expired-token error payload and the test asserts both HTTP status and error payload structure.

3. Given a logged-in session and the presence of an implemented token revocation endpoint, When a revoke (logout) request is executed, Then the revoked token is marked/removed in the token store and any replayed requests using the same token are rejected with 401 Unauthorized; the test verifies token store state change and enforcement on protected endpoints.

4. Given role-based accounts provisioned in test fixtures (admin, editor, reader), When each role attempts actions across protected endpoints (read, write, admin actions), Then the authorization layer enforces RBAC rules (200 for allowed, 403 for forbidden) and tests assert HTTP responses and that an audit event or log entry is generated for denied attempts.

5. Given the Git repository CI configuration and branch protection rules, When a PR is created or commits are pushed, Then the CI pipeline runs the auth integration test job on a dedicated runner (with ephemeral infra), enforces pass/fail gating (fails PR on test failure), generates a coverage report for auth-related code with threshold enforcement (auth-critical >= 80%), and uploads test reports and coverage artifacts to the CI system for review.

## Edge Cases
* What happens when there is clock skew between services (token issued on one host, validated on another)? 
  - Handling: Tests include simulation of +/- 2 minute clock skew; the expected system behavior is to accept within configured clock tolerance (e.g., 2 minutes). Outside tolerance, validation fails with an expired/invalid token error and the test asserts the failure and presence of diagnostic log entries.

* How does the system behave when token-store (Redis/DB) is unavailable during revocation attempts?
  - Handling: Integration tests simulate token-store outage. Expected behavior: logout endpoint returns 503 Service Unavailable (non-revealing), a fallback persistence queue or retry is enlisted where applicable, and CI asserts the 503 response and that an alert/failure metric is emitted to observability mocks.

* What happens when CI rate limits or third-party dependencies (e.g., email, 3rd-party OAuth) cause flakiness?
  - Handling: CI jobs must use test doubles or local emulators for third-party services; tests detect real external-rate-limit responses and mark them as infra-flaky (fail the job if persistent); retry/backoff is implemented for transient responses and test stability threshold is configured (max 2 automatic retries).
