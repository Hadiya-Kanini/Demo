A retry policy (max 1 automatic re-run) is applied before marking as failed/inconclusive.

     - Handling: Each scenario must include an explicit warm-up phase (configurable; default 2 minutes) before recording baseline metrics. Results must be reported separately for warm-up vs steady-state.

   * How does the test account for token-signing key rotation or transient invalid tokens during long runs?
     - Handling: Tests must support long-lived client sessions that can automatically refresh tokens when rotation occurs, or test both token-refresh and token-expiry paths in isolation.      - Handling: Performance tests for the login path must stub or emulate third-party dependencies (local emulators or service-virtualization) in CI performance runs. ## Story ID
* ID: us_010

## Story Title
* Performance & Load Tests for Authentication Endpoint

## Description
* As a Performance/QA engineer, I want repeatable performance and load tests for the login path that produce baseline latency and throughput metrics and validate results against the defined SLAs, so that the authentication service meets reliability and latency targets and performance regressions are detected early.

## Acceptance Criteria
1. Given a dedicated test environment provisioned with the authentication service and deterministic test fixtures (test users, roles, token TTL), When the performance test harness executes a parameterized login-path scenario (with explicit warm-up, ramp-up, steady-state and ramp-down phases), Then the run produces artifacts including p50/p95/p99 latencies, throughput (req/s), error rates, and correlated host/resource metrics (CPU, memory, DB/Redis latency), and these artifacts are uploaded to CI artifact storage and a dashboard is generated.

2. Given the project's SLA configuration (referenced from EP-001 performance doc), When a performance test runs at the SLA load profile, Then the test asserts SLA compliance (e.g., p95 <= configured_p95_ms, p99 <= configured_p99_ms, error rate <= configured_error_rate) and the test fails if any SLA metric is violated; failure artifacts include top traces and resource correlations.

3. Given a committed baseline run, When a scheduled CI performance job executes subsequently, Then the job compares new metrics to the baseline, reports regressions when key metrics degrade beyond the allowed delta (e.g., p95 increases >10% or error rate increases beyond threshold), and stores historical trend graphs and regression summaries as artifacts.

4. Given limited resources for CI and risk of heavy runs blocking CI, When a pull request is opened, Then a lightweight performance smoke test runs on the PR (short ramp, sanity assertions) and full load tests run on scheduled nightly or on-demand jobs on dedicated runners; reports for both are available in CI.

5. Given the harness and scenarios are versioned in the repository, When engineers need to reproduce a failing performance run, Then they can run the same parameterized scenario locally or on CI using documented commands, with seed data and environment configuration producing reproducible results.

6. Given observability integrations exist, When a test detects failures or SLA violations, Then the post-run report includes correlated APM/DB/Redis metrics and identifies the top contributing resource or external dependency.

## Edge Cases
* Noisy test infra: pre-checks mark run inconclusive if host/network variance exceeds allowed thresholds; automatic single re-run allowed.
* Warm-up vs steady-state: warm-up phase excluded from steady-state metrics; both reported.
* Token rotation mid-run: harness supports token refresh or explicit rotation scenarios; rotation-caused failures produce detailed diagnostics.
* Third-party dependency throttling: use emulators/mocks in CI; if real third-parties used, classify failures as infra-flaky with detailed trace.
* Rate-limiting interference: use test-only rate-limits or document and classify expected rate-limit responses.
* Statistical variance: include confidence intervals/sample sizes; regression detection uses configured deltas or statistical significance when available.
* Soak-test resource accumulation: run against ephemeral infra or include cleanup; monitor storage growth and fail on threshold breaches.
