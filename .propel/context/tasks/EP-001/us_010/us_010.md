## Story ID
   * ID: us_010

## Story Title
   * Integration & E2E Tests for Auth Flows

## Description
  * As a QA engineer and developer, I want automated integration and end-to-end tests covering successful login, invalid credentials, token expiry, RBAC and logout scenarios, so that authentication flows are validated end-to-end, regressions are prevented, and release confidence is increased.

## Acceptance Criteria
1. Given a seeded test user with valid credentials and active account, When the frontend posts valid credentials to POST /auth/login, Then the system returns HTTP 200, issues a valid access token (JWT) with user_id and roles and correct exp, sets cookies/session per security config (HttpOnly, Secure, SameSite), and the frontend redirects to the correct role-specific dashboard within the NFR latency target (median auth < 1s). The test asserts token signature validity, claims, and session behavior.

2. Given a seeded test user and incorrect password, When the frontend posts invalid credentials to POST /auth/login, Then the system returns HTTP 401 with a generic message "Invalid email or password", does not issue tokens, increments the failed-login counter for that account/IP, and the UI displays a non-revealing error. The integration test asserts no token is returned and the failure is logged.

3. Given an access token and valid refresh token have been issued, When the access token expires and the client posts to POST /auth/refresh with a valid refresh token, Then the system returns HTTP 200 with a new access token and updated exp claim; when the refresh token is expired or revoked, POST /auth/refresh returns HTTP 401 and requires full re-authentication. Tests must simulate time progression (or short TTL test tokens) to validate both successful refresh and refresh failure flows.

4. Given a user authenticated with role 'viewer' and a user authenticated with role 'admin', When each requests an admin-only API endpoint or navigates to an admin-only UI route, Then the viewer's API requests return HTTP 403 and the UI prevents navigation (shows "Access denied"), while the admin receives HTTP 200 and the UI allows access. Tests assert enforcement at both API and UI layers and verify minimal, non-revealing error messages per UXR mappings.

5. Given an authenticated session, When the user triggers logout via POST /auth/logout or UI logout, Then the system revokes/invalidate tokens and session state, returns HTTP 200, the frontend clears session data and redirects to the login screen, and any subsequent request using the previous token returns HTTP 401. Tests verify revocation via token blacklist/DB and confirm no access after logout.

## Edge Cases
* What happens when a logout and token refresh occur concurrently (race condition)?
  - Handling: Tests will simulate concurrent refresh + logout requests. The expected behavior is deterministic: token revocation must make any subsequently presented token invalid. The test asserts that after logout completes, any access using pre-logout tokens or refresh flows fails with HTTP 401. Implementation should use atomic revocation (e.g., token versioning or revocation store) to pass the test.

* How does the system handle account lockout due to repeated failed logins?
  - Handling: Tests will attempt >5 failed logins within a 5-minute window and assert the system enforces rate limiting/lockout (returns HTTP 429 or HTTP 423 per policy), the UI shows a generic message, no tokens are issued, and lockout duration or admin unlock behavior follows spec. Tests verify counters reset after configured cooldown.

* How does the system behave if the identity store / token store times out or is unavailable?
  - Handling: Integration tests simulate DB/identity-service timeouts and expect the auth endpoints to return HTTP 503 (Service Unavailable) with a safe, generic message; no tokens are issued. Tests also assert that transient failures are surfaced to monitoring and that retries/backoff policies (if configured) are respected.
