## Story ID
   * ID Format: US_<unique seq num>
   * Story ID: us_014

## Story Title
   * Integrate Signing Key Usage in Auth

## Description
  * "As a backend engineer responsible for the authentication service, I want the auth service to retrieve signing keys from the secrets vault and use them to sign and verify tokens in both production and test modes, so that tokens are cryptographically secure, support key rotation, and tests remain deterministic without depending on the vault."

## Acceptance Criteria
  * Given the auth service is running in production with vault connectivity and the vault contains an active signing key (kid=K-active) and previously rotated keys, When a token is issued for an authenticated session (POST /auth/login), Then the service retrieves the active signing key from the vault (or from a valid cache), signs the token using the configured algorithm (e.g., RS256), sets the token header 'kid' to K-active, includes required claims (sub, roles, iat, exp), and returns the signed token and expires_at timestamp in the login response.
  * Given a bearer token presented for verification and the token includes a 'kid' header matching either the active key or a retained previous key, When the auth service validates the token, Then it fetches (or uses cached) the corresponding public key and successfully verifies the signature and standard claims (iss, aud, exp, nbf) with configured clock-skew tolerance and returns a valid authentication decision to callers.
  * Given a bearer token that does not include a 'kid' header, When the auth service attempts verification, Then it will attempt verification deterministically against retained public keys in configured order, log the ambiguity and attempts, and accept if any retained key verifies successfully within the retention window; otherwise return auth failure (401) and log the incident.
  * Given the auth service is operating in production and the vault is temporarily unreachable, When the service attempts to issue a token and there is no valid cached active private key available, Then the service fails token issuance with an HTTP 503 (Service Unavailable) and logs an auditable error; when verifying incoming tokens, the service may use cached public keys that are within their configured cache TTL and accept verification if they validate successfully.
  * Given the system is running in test mode (explicit environment/config flag), When tests or local development request token signing or verification, Then the auth service does not call the vault, instead uses an in-memory or config-provisioned deterministic test signing key, signs tokens with that key, and allows verification using the corresponding test public key, ensuring tests do not depend on vault availability.
  * Given a key rotation scenario where a new active key is published and previous keys are retained for verification, When tokens signed with a retained previous key are presented, Then the service successfully verifies those tokens for the configured retention/verification window, and issuance uses the newly active key going forward; all key retrievals include key ID (kid) and algorithm assertions and are logged for audit.
  * Given misconfiguration or unexpected key metadata (e.g., algorithm mismatch), When the service fetches the key, Then the service rejects usage of that key for signing, logs a clear auditable error (without private key material), retries per config, and fails token issuance with HTTP 502/503 if unresolved.
  * Given operational observability requirements, When signing keys are retrieved or used, Then the service emits metrics for vault latency, vault errors, cache hit/miss, issued tokens count, failed verifications due to missing or invalid keys, and logs key retrieval events (kid, algorithm, source=cached|vault) without exposing private key material.
  * Given secure memory handling requirements, When private keys are loaded into memory for signing, Then the service keeps private key lifetime minimal, avoids persisting private keys in logs or diagnostics, and zeroes memory where the runtime/language supports secure zeroing after use.

## Edge Cases
   * What happens when the vault returns a malformed or unsupported key format?
     - The system must log the error with operation context (operation, requested kid if any), emit a metric/alert, retry retrieval with exponential backoff (configurable, default 3 attempts), and fail token issuance with HTTP 502/503 if still unsuccessful. Verification attempts using that key must fail and be flagged for investigation.
   * How does system handle key rotation where tokens exist that were signed by a recently rotated (now previous) key?
     - Retain previous public keys for a configurable verification retention window. Use the token 'kid' to select the correct key; if 'kid' is missing, attempt verification against retained keys in deterministic order and log ambiguity. Do not issue new tokens with an old key.
   * What happens when vault is unreachable and there is no cached key (cold-start or cache eviction)?
     - In production: fail-closed for token issuance (HTTP 503) and emit incident metrics/alerts. For verification: return HTTP 503 to callers indicating dependency outage (or 401 depending on API contract), but prefer 503 to indicate dependency issue. In test mode: use in-memory test key and do not call the vault.
   * What happens when the vault returns a key whose algorithm does not match the configured signing algorithm?
     - The service must reject the key for signing, log the mismatch with kid and algorithm (no private key data), increment an alert/metric, and await reconciliation before using the key.
   * What happens on verification failures due to clock skew greater than tolerance?
     - Treat token as invalid (401); log the failure reason (clock skew). Clock-skew tolerance is configurable and test harnesses must be able to override it.
   * What happens if a key is suspected or confirmed compromised?
     - Follow operational playbook: mark key revoked in vault, purge private key from caches, publish new active key, disallow further verification/issuance for revoked key outside retention window, and surface telemetry for incident response.
   * What happens if private key material is accidentally logged or exposed?
     - Automated scans/alerts and code review guardrails should detect exposures. If exposure occurs, rotate keys immediately and follow incident response procedures.
