## Story ID
   * ID Format: US_<unique seq num>
   * Story ID: us_014

## Story Title
   * Integrate Signing Key Usage in Auth

## Description
  * "As a backend engineer responsible for the authentication service, I want the auth service to retrieve signing keys from the secrets vault and use them to sign and verify tokens in both production and test modes, so that tokens are cryptographically secure, support key rotation, and tests remain deterministic without depending on the vault."

## Acceptance Criteria
  * Given the auth service is running in production with vault connectivity and the vault contains an active signing key (kid=K-active) and previously rotated keys, When a token is issued for an authenticated session, Then the service retrieves the active signing key from the vault (or from a valid cache), signs the token using the configured algorithm (e.g., RS256), sets the token header 'kid' to K-active, and returns the signed token and expires_at timestamp in the login response.
  * Given a bearer token presented for verification and the token includes a 'kid' header matching either the active key or a retained previous key, When the auth service validates the token, Then it fetches (or uses cached) the corresponding public key and successfully verifies the signature and standard claims (iss, aud, exp, nbf) and returns a valid authentication decision.
  * Given the auth service is operating in production and the vault is temporarily unreachable, When the service attempts to issue a token and there is no valid cached active private key available, Then the service fails token issuance with an HTTP 503 (Service Unavailable) and logs an auditable error; however, when verifying incoming tokens, the service may use cached public keys that are within their configured cache TTL and accept verification if they validate successfully.
  * Given the system is running in test mode (explicit environment/config flag), When tests or local development request token signing or verification, Then the auth service does not call the vault, instead uses an in-memory or config-provisioned test signing key (deterministic fixture), signs tokens with that key, and allows verification using the corresponding test public key, ensuring tests do not depend on vault availability.
  * Given a key rotation scenario where a new active key is published and previous keys are retained for verification, When tokens signed with a retained previous key are presented, Then the service successfully verifies those tokens for the configured retention/verification window, and issuance uses the newly active key going forward; all key retrievals include key ID (kid) and algorithm assertions and are logged for audit.

## Edge Cases
   * What happens when the vault returns a malformed or unsupported key format?
     - The system must log the error with operation context, emit a metric/alert, retry retrieval with exponential backoff (configurable, default 3 attempts) and then fail token issuance with HTTP 502/503 if still unsuccessful. Verification attempts using that key must fail and be flagged for investigation.
   * How does system handle key rotation where tokens exist that were signed by a recently rotated (now previous) key?
     - Retain previous public keys for a configurable verification retention window. Use token 'kid' to select the correct key; if 'kid' is missing, attempt verification against retained keys in deterministic order and log ambiguity. Do not issue new tokens with an old key.
   * What happens when vault is unreachable and there is no cached key (cold-start or cache eviction)?
     - In production: fail-closed for token issuance (HTTP 503) and emit incident metrics/alerts. For verification: return HTTP 503 to callers indicating dependency outage. In test mode: use in-memory test key and do not call the vault.

Notes:
- Implement caching with a configurable TTL for keys (default e.g., 5 minutes) and a retention window for previous keys to support verification of recently issued tokens after rotation.
- Ensure all operations are instrumented: metrics for vault latency/errors, cache hit/miss, issued tokens count, and failed verifications due to missing keys.
- Follow secure handling of private keys in memory (minimal lifetime, secure zeroing where language supports), restrict logs from containing private key material, and ensure the service reads vault credentials from the environment or platform secret provider (per us_infra_001).
- Story scope conforms to INVEST: independent (vault integration dependency noted), negotiable, valuable (secure token issuance), estimable, small (<=5 points), testable (ACs provide verifiable conditions).
