## Story ID
   * ID Format: us_021

## Story Title
   * CI End-to-End Integration Tests with Vault

## Description
  * As a CI engineer, I want the CI pipeline to run end-to-end authentication tests against a test Vault instance that supplies signing keys, so that we validate signing and verification behavior (including token claims, expiry, iat, clock-skew tolerance, and key rotation) end-to-end before changes reach staging/production.

## Acceptance Criteria
  * Given a CI environment with a provisioned test Vault instance reachable at a configured address and valid test credentials, When the CI pipeline executes the "e2e-auth-integration" stage, Then the stage:
    - retrieves signing keys from the test Vault instance using the configured Vault auth method,
    - calls the POST /auth/login endpoint using test credentials,
    - obtains an issued token and verifies the token signature and claims (sub, roles, iat, exp) against the public/verification key fetched from Vault,
    - asserts that the /auth/login response includes expires_at and status 200,
    - and the CI job completes with exit code 0 when all assertions pass.

  * Given a token issued by the system with an expiry in the past (expired token scenario), When the test attempts verification of that token, Then the system rejects verification and the test asserts the API returns HTTP 401 with a documented error code (e.g., "token_expired") and a descriptive error message, and the verification library raises/returns the expected expiry error.

  * Given the system is configured with a clock-skew tolerance of X seconds (configurable via env var), When the test issues a token whose exp is within the configured skew boundary (e.g., exp = now + (tolerance - 1)), Then token verification succeeds; and when exp is outside the tolerance (e.g., exp = now - (tolerance + 1)), verification fails with the token_expired error — both cases asserted by the CI tests.

  * Given a signing-key rotation scenario where Vault has an active key K_new and a previously active key K_old still present in Vault's key set, When the test:
    - issues a token signed with K_old (simulate by signing with the old key or by manipulating the issuer to use K_old), and
    - attempts verification using the keys fetched from Vault,
    Then verification succeeds while K_old remains discoverable by Vault, and the test asserts that verification fails after K_old is removed (if removal is simulated in test), producing a clear failure that surfaces rotation policy issues.

  * Given Vault authentication or retrieval returns transient errors (timeouts, 5xx), When the CI stage runs, Then the test harness:
    - retries Vault reads up to a configurable retry count with exponential backoff (configurable in CI),
    - emits structured logs and artifacts (raw Vault responses, request IDs) for debugging upon failure,
    - fails the CI job if retries are exhausted, and the failure includes a clear message: "Vault unavailable or access denied — e2e-auth-integration aborted."

## Edge Cases
   * What happens when Vault is unavailable or credentials are invalid?
     - Handling: CI stage must attempt configurable retries (e.g., 3 attempts with exponential backoff). If still unavailable, the job fails fast and produces an artifact bundle: test logs, Vault request/response snippets (sanitized), and explicit failure message. Optionally the pipeline can be re-run manually after Vault is restored. For local developer iteration, tests support a flag (e.g., --use-local-keys) to use a locally-stored dev keyfile to allow offline runs; this flag must NOT be used in the authoritative CI run.

   * How does the system handle clock-skew boundary conditions?
     - Handling: Tests include cases for within-boundary acceptance and outside-boundary rejection using a controllable test clock or token generation with manipulated iat/exp. The CI asserts behavior matches configured clock-skew tolerance. Configuration values for clock skew and token TTL must be captured in the test artifacts.

   * What happens when signing keys rotate and the previous key has been fully removed before tests run?
     - Handling: Tests explicitly verify key rotation policy by:
       - ensuring rotation leaves previous key accessible during an overlap window (if policy mandates),
       - if policy does not retain previous key, tests expect older tokens to fail verification; tests will fail if product policy requires overlap but Vault/ops removed the key prematurely. Tests must surface actionable diagnostics: key IDs available in Vault at test time, token kid header values, and verification failure stack traces.
