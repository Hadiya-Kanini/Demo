## Story ID
   * ID: us_008

## Story Title
   * Standardize Non-Revealing Error Messages

## Description
  * As a user, I want authentication error messages (client and server) to be non-revealing, consistent, and accessible so that attackers cannot enumerate account state while legitimate users receive clear, actionable guidance.

## Acceptance Criteria
1. Given an authentication attempt with invalid credentials, When the client receives the response, Then the client displays a single generic message "Invalid credentials. Please try again or reset your password." and the server returns HTTP 401 with no indication whether the username or password was incorrect; detailed cause is recorded only in server logs and associated with a non-sensitive correlation ID included in the response header.

2. Given an authentication attempt that exceeds configured retry thresholds, When the server enforces rate limiting or a temporary lockout, Then the client displays "Access temporarily limited. Try again later or reset your password." (without revealing account existence) and the server returns HTTP 429 (or 423 if account locked) with consistent messaging across web and API, while the server logs include the lockout reason and timestamp.

3. Given an internal server error during authentication (validation, DB outage, third-party failure), When the client receives the response, Then the client displays "An error occurred. Please try again." and the server returns HTTP 500 without stack traces or internal diagnostics; server logs capture full details and link to the correlation ID returned to the client for support escalation.

4. Given any authentication-related message shown in the UI, When it is rendered, Then the message is provided through an ARIA live region (assertive), meets WCAG color contrast requirements, has localized text with fallback to English, and matches the centralized message catalogue so copy is consistent across login, reset, and identity flows.

5. Given API consumers (non-UI clients) receive authentication failures, When they inspect the response, Then they receive the same non-revealing error string in the response body and a machine-readable error code defined in the API spec (e.g., AUTH_INVALID_CREDENTIALS, AUTH_RATE_LIMITED), with no additional user-existence details, and the correlation ID in headers for support.

## Edge Cases
* What happens when a user requests rapid repeated password resets or login attempts? (Handling: enforce rate limits for reset and login endpoints; the UI and API return the same non-revealing rate-limit message and the server records per-IP and per-account metrics for security review.)
* How does the system handle missing localization entries for the standardized messages? (Handling: fall back to a default English non-revealing message; emit telemetry for missing translations so localization owners can fix.)
* What happens when client-side JavaScript is disabled or a low-bandwidth client is used? (Handling: server-rendered pages and API responses must include the same standardized message text in HTML and JSON payloads respectively; ensure messages appear in accessible places without requiring JS.)
