## Story ID
* ID: US_008

## Story Title
* UI Login Page & Client-side Validation

## Description
* As an end user, I want a responsive and accessible login UI with client-side validation so that I can sign in quickly, receive clear guidance on input errors, and have my credentials handled securely when the client calls /auth/login.

## Acceptance Criteria
1. Given the login screen is displayed, When the user types into the email and password fields, Then the client performs real-time validation (email format, required fields, password min/max length, trim whitespace rules), displays inline, accessible errors for each invalid field, and disables the submit button until obvious client-side validation passes.

2. Given the user submits syntactically valid credentials, When the client issues POST /auth/login, Then the request is sent over HTTPS, the UI shows a loading/disabled submit state to avoid duplicate submissions, and the client does not log or persist raw password contents.

3. Given the server responds with 200 and provides an authentication payload (token and expires_at) or sets an HttpOnly secure cookie, When the response is received, Then the client executes the onAuthSuccess flow: store token per security policy (prefer server-set HttpOnly cookie; if token returned to client, store in memory and use secure storage only per policy), clear the password input, navigate to the intended destination, and record a minimal non-sensitive audit event.

4. Given the server responds 400 for validation errors, When the client receives the response, Then the UI displays a single non-revealing error ("Invalid input") or maps server-provided field hints to inline messages (while preserving non-revealing wording) and maintains accessibility semantics (aria-live, focus management).

5. Given the server responds 401 for bad credentials, When the client receives the response, Then the UI displays a generic error ("Invalid email or password"), offers a "Forgot password" action, and does not reveal which credential failed or any timing information.

6. Given the server responds 429 (rate-limited), 423 (account locked), or returns Retry-After, When the client receives the response, Then the UI displays a non-revealing message ("Too many attempts; try again later"), disables submission until Retry-After or shows a countdown, and surfaces recovery options (password reset, contact support) and accessible challenge alternatives if the server requests a CAPTCHA.

7. Given transient network issues or 5xx errors from the server, When the user submits the login form, Then the UI surfaces a retryable error ("Service temporarily unavailable; please try again"), allows the user to retry, and avoids automatic retries that could worsen rate limits.

8. Given accessibility requirements, When the user navigates via keyboard or uses assistive technology, Then all form elements are correctly labeled, focus order is logical, error messages are announced via aria-live/role=alert, and any server-provided challenge (CAPTCHA) has an accessible alternative path (audio, support contact).

9. Given the API evolves validation hints, When the server includes structured validation metadata in responses, Then the client uses those hints to show precise, non-revealing inline messages; if unavailable, the client falls back to its built-in validation rules.

## Edge Cases
* What happens when a legitimate user is behind a shared NAT and causes many requests from a single IP?
  - Handling: If the server returns 429, present a clear, non-revealing message with Retry-After and offer CAPTCHA or recovery flows. Prioritize per-account guidance and include support/escalation links. Avoid exposing server mitigation mechanics.

* How does the UI behave on distributed low-and-slow credential stuffing?
  - Handling: The UI relies on server verdicts (429/423). If the server requests CAPTCHA or secondary verification, present it accessibly. Maintain user input (except password when policy requires clearing) so a legitimate user can recover without retyping everything.

* How does the client handle detection of out-of-spec or suspicious input (very long fields, invalid encoding)?
  - Handling: Client blocks obvious invalid submissions and shows "Invalid input". Do not reveal sanitization rules or internals. If submission proceeds and the server responds 400, map to non-revealing error and clear sensitive fields per policy.

* What happens if the network is offline or very slow?
  - Handling: Show offline indicator, allow the user to cancel the attempt, and retry manually. Do not retry automatically. Provide clear guidance and preserve security (do not write passwords to persistent storage).

* What happens if the API error contract changes unexpectedly?
  - Handling: Client uses a resilient fallback: recognize known status codes (200, 400, 401, 429, 423, 503) and for unknown responses show a generic error and log the response to telemetry (without credentials) for engineering investigation.
