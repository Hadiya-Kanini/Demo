## Story ID
* ID: US_008

## Story Title
* Implement Input Validation, Rate Limiting, and Lockout

## Description
* As a security-conscious product owner, I want consistent client- and server-side input validation, IP/account rate limiting, and account lockout policies so that brute-force and automated attack attempts are mitigated while minimizing false positives for legitimate users.

## Acceptance Criteria
1. Given a login request with malformed or out-of-spec fields, When the request is submitted, Then client-side validation prevents submission where possible and server-side validation returns HTTP 400 with a single non-revealing error message ("Invalid input") and logs sanitized details for investigation.

2. Given normal authentication traffic, When users submit credentials, Then the server validates input against the canonical schema (field lengths, allowed characters, encoding) and enforces the same validation rules as the client; valid inputs proceed to authentication flow, invalid inputs are rejected.

3. Given repeated failed authentication attempts, When the configured per-account failed-attempt threshold (e.g., 5 attempts within 15 minutes) is exceeded, Then the account is locked for a configurable TTL (default 15 minutes), the failed-attempt counter resets after successful reset/unlock, and the system returns a generic throttling response ("Too many attempts; try again later") while emitting an audit event.

4. Given repeated requests from the same IP, When the per-IP rate threshold (e.g., 100 requests/minute to auth endpoints) is exceeded, Then rate limiting returns HTTP 429 with Retry-After header and increments the IP risk score; for sustained abuse, the system applies escalations (temporary block or CAPTCHA) and records events to monitoring.

5. Given lockout or rate-limiting events, When an administrator performs an authorized unlock or the user completes an approved recovery (password reset with verified email/2FA), Then the system clears the lockout state and failed-attempt counters immediately and logs the unlock action with actor and timestamp.

## Edge Cases
* What happens when a legitimate user is behind a shared NAT and causes many requests from a single IP?
  - Handling: Prioritize per-account counters over per-IP enforcement; if per-IP enforcement would block many accounts, present CAPTCHA challenge or progressive delayed responses instead of full lockout. Provide actionable UI guidance and a support escalation path for customers.

* How does the system handle distributed low-and-slow credential stuffing (attacker rotates IPs to stay under per-IP limits)?
  - Handling: Use combined signals (per-account failed attempts, velocity of distinct usernames targeted, user-agent anomalies) to detect distributed attacks. Trigger per-account lockouts, global thresholds, or WAF rule updates. Feed signals to SIEM/IDS for automated blocking and investigation.

* How does the system behave if the rate-limiter datastore (e.g., Redis) becomes unavailable?
  - Handling: Fail-safe: apply conservative in-process throttling with lower thresholds and surface a clear operational alert. Return 503 for auth endpoints if integrity cannot be ensured, and ensure all incoming attempts are logged for post-mortem. Avoid silent failure that would allow unlimited attempts.
