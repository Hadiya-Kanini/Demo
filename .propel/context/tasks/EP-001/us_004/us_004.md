## Story ID: us_004

## Story Title: Token Issuance and TTL Enforcement

## Description: As an authenticated user, I want the system to issue a time-limited token on successful login and have the verification middleware enforce the token TTL, so that access is secure, auditable, and expired tokens are rejected automatically.

## Acceptance Criteria (Given/When/Then format, 3-5 criteria)
1. Given valid user credentials and POST /auth/login, When authentication succeeds, Then the response returns an access token string and the system stores token metadata containing:
   - issued_at (UTC ISO 8601),
   - expires_at (UTC ISO 8601 = issued_at + configured TTL),
   - token_id or token_hash,
   and the response body includes issued_at and expires_at fields.

2. Given a protected endpoint and a request carrying a valid, unexpired token, When the verification middleware executes, Then the middleware:
   - verifies token existence and metadata in the token store,
   - confirms current time < expires_at (accounting for configurable clock skew),
   - attaches token metadata (user_id, issued_at, expires_at, token_id) to the request context,
   - allows downstream handlers to proceed (200/expected behavior).

3. Given a request with an expired token (current time >= expires_at), When the verification middleware executes, Then the middleware rejects the request with HTTP 401 Unauthorized and a clear error body: { "error": "token_expired" }, and the token is marked/recorded as expired in the token store (or removed per config).
