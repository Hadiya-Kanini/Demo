## Story ID
* ID: us_004

## Story Title
* Implement Token Storage and Revocation Mechanism

## Description
* As a platform security engineer, I want server-side token persistence (or blacklist) support and a revocation API so that access and refresh tokens can be revoked immediately and logout or administrative revocation is enforced across all protected endpoints.

## Acceptance Criteria
1. Given a freshly issued access and refresh token pair, When the tokens are persisted to the token store, Then the token store records token identifier, token type (access/refresh), user id, client id, issued_at, expires_at, and status (active) and returns success (201) within normal DB latency.

2. Given a user calls the logout endpoint with a valid refresh or access token, When the endpoint receives the request, Then the service marks the associated token(s) as revoked in the token store, emits an audit log event with actor, token id, and timestamp, and responds with 200 OK.

3. Given a protected API receives a bearer token, When authorization middleware validates the token, Then it performs a revocation lookup against the token store; if token is revoked or absent/invalid, it returns 401 Unauthorized with error "token_revoked" (or "invalid_token" if invalid), otherwise allows the request; the middleware must add no more than 50ms median latency for the lookup under normal conditions.

4. Given a refresh token rotation flow, When a client exchanges a refresh token for new tokens, Then the system atomically revokes the old refresh token and persists the new refresh token to the token store; any concurrent reuse of the old refresh token is rejected and logged as a security event.

5. Given expired or long-revoked tokens accumulate, When the scheduled cleanup job runs (configurable cadence, e.g., daily), Then it purges tokens older than the configured retention window (default: 30 days), exposes metrics (purged_count, duration, errors), and records a summary audit event.

## Edge Cases
* What happens when the token store (DB/Redis) is unavailable?
  - Behavior: For protected endpoints, fail closed (reject requests with 503 Service Unavailable or 401 depending on policy) and surface a clear monitoring/alerting event. For logout/revoke operations, return 503 with retry hints. If an explicit degraded mode is permitted via config, use a short-lived in-memory cache as read-only fallback for active tokens only, with telemetry and strict limits; degraded mode is disabled by default.

* How does the system handle refresh token reuse or concurrent use (race)?
  - Handling: Use atomic transaction semantics or compare-and-set on token record (e.g., row version / unique used flag). On successful exchange, mark old refresh token revoked and persist new token in same transaction. If reuse detected, reject with 401 and "invalid_grant - token reuse", increment security counter, and create an audit/security incident record.

* What if clock skew causes tokens to appear expired/unexpired?
  - Handling: Allow a small configurable clock-skew tolerance (e.g., 60 seconds) on validation checks; audit mismatches and recommend NTP sync in operations runbook. Never extend token validity beyond configured expires_at; skew only loosens validation on the boundary to prevent false rejections.
