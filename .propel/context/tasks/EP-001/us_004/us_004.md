## Story ID
* ID: us_004

## Story Title
* Token Issuance: Signed Token with Claims

## Description
* As a platform security engineer, I want the authentication service to issue signed tokens (JWT or an agreed secure format) containing standard claims (sub, roles, iat, exp, jti) with configurable expiry and include response.expires_at so that clients can manage sessions and protected endpoints can validate token authenticity and lifetime reliably.

## Acceptance Criteria
1. Given valid user credentials and client context, When the authentication service issues an access (and optional refresh) token, Then the returned token is signed with the configured signing key/algorithm, includes claims:
   - sub: user identifier
   - roles: array of role identifiers (may be empty)
   - iat: issued-at (epoch seconds)
   - exp: expiry (epoch seconds)
   - jti: unique token identifier
   and the HTTP response body contains expires_at (ISO-8601 UTC) that corresponds to exp.

2. Given configurable token lifetime settings, When a token is requested/issued, Then the token’s exp claim and response.expires_at reflect configured lifetimes for access vs refresh tokens; if a requested TTL exceeds policy, the service rejects the request with 400 Bad Request and error "invalid_request - ttl_exceeds_limit".

3. Given signing keys are stored in the secrets vault, When issuing a token, Then the service retrieves the signing key from the vault (EP-006) and signs using the configured algorithm (e.g., RS256/ES256). If the vault is unreachable and no explicit dev fallback is enabled, the service returns 503 Service Unavailable and emits an alert and metric.

4. Given a token validator verifies a token, When validating signature and claims, Then signature verification succeeds using the corresponding verification material, iat/exp are validated with configurable clock-skew tolerance (default 60s), and invalid tokens are rejected with 401 Unauthorized and standardized error codes ("invalid_token - expired", "invalid_token - not_yet_valid", "invalid_token - missing_claim", or "invalid_token - invalid_signature").

5. Given high-load issuance scenarios under normal operating conditions, When issuing tokens, Then median token creation latency remains within acceptable platform NFRs (target: median ≤100ms for token creation path), and token issuance failures and latencies are exposed as metrics and audit logs.

6. Given tokens are returned to clients, When inspecting token contents, Then no sensitive data (passwords, secrets, PII beyond required identifiers) is present in claims; only minimal authorization claims are included.

## Edge Cases
* What happens when the secrets vault (signing key) is unavailable?
  - Behavior: Fail-safe — reject issuance with 503 Service Unavailable, create an alert/incident metric. A developer-only fallback signing key may be allowed only when explicitly enabled via configuration (dev mode) and must be auditable; production default: no fallback.

* How does the system handle requested TTLs beyond configured max?
  - Handling: Return 400 Bad Request with error "invalid_request - ttl_exceeds_limit" and do not issue token.

* What if system clock skew causes tokens to be seen as not-yet-valid or expired?
  - Handling: Use configurable clock-skew tolerance (default 60 seconds) in validation. iat is set using server UTC time. Document requirement for NTP sync in operations. Never extend exp beyond configured TTL.

* What if required claims (sub, roles) are missing or malformed during issuance or validation?
  - Handling: Issuance must not produce tokens missing required claims — treat as internal error (500) and log/audit. On consumption, validation fails with 401 Unauthorized and "invalid_token - missing_claim".

* What if signing algorithm configuration is invalid or unsupported?
  - Handling: Fail issuance with 500 Internal Server Error; emit configuration alert and guidance in logs. Validate algorithm config at startup to minimize runtime failures.

* How to mitigate token replay if revocation is required?
  - Handling: Include jti claim. Revocation/replay prevention (persistent jti tracking, blacklist) is out of scope for this issuance story and must be implemented by token revocation stories (e.g., token storage/revocation story). Document integration points (jti presence) for those stories.
