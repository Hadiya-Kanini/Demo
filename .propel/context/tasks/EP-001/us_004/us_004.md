## Story ID
* ID: us_004

## Story Title
* Issue Authentication Tokens

## Description
* As an authenticated user (or client), I want the system to issue authentication tokens (JWT or opaque) on successful login so that protected endpoints can verify my identity and role-based permissions and tokens automatically expire to reduce session risk.

## Acceptance Criteria
1. Given valid user credentials and a POST /auth/login request, When authentication succeeds, Then the response is 200 OK and contains an access token (JWT or opaque) and token metadata (token type, TTL in seconds), the token includes a unique identifier (jti or token id), user id, and role claims; for JWT tokens the response includes a signed JWT with standard claims (iss, sub, aud, exp, iat, jti) and for opaque tokens the token is stored server-side with TTL applied.

2. Given a protected API endpoint and a request containing Authorization: Bearer <token>, When the endpoint validates the token, Then the token is accepted if signature/validation succeeds (for JWT) or token exists and is unexpired (for opaque) and the role claims satisfy the endpoint's RBAC check; authorized requests return 200 and unauthorised role access returns 403.

3. Given any request presenting an expired token, When the token validation is performed, Then the system returns 401 Unauthorized with a non-revealing message ("Authentication required" or "Invalid or expired credentials"), and the response includes the WWW-Authenticate header as per spec (e.g., Bearer error="invalid_token", error_description="The access token expired").

4. Given a user submits a valid token to the logout/revoke endpoint, When the revocation completes, Then the token is invalidated immediately (opaque token removed or JWT jti recorded in revocation store), and subsequent uses of that token return 401 Unauthorized.

## Edge Cases
* What happens when system clocks differ (clock skew) between client and server?
  - Handling: The token validation logic allows a configurable leeway (default 60 seconds; configurable up to 120s). Servers must be NTP-synchronized; any near-expiry tokens within the tolerance are accepted but flagged in logs for audit.

* How does the system handle malformed or tampered tokens (bad signature / invalid format)?
  - Handling: Immediately reject with 401 Unauthorized, do not leak verification specifics to client, increment security anomaly metrics, rate-limit repeated malformed token attempts, and record detailed diagnostic info in secure logs (not sent to client).

* What happens when a token is revoked while in-use across concurrent requests?
  - Handling: Tokens are uniquely identified. Revocation marks the token id as invalid; token validation checks consult a primary store/cache. Revocation is considered effective immediately for new validations; for in-flight requests, behavior is best-effort — any subsequent requests will fail. Design must ensure cache invalidation or short cache TTL to prevent long-lived acceptance of revoked tokens.
