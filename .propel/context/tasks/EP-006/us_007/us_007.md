## Story Title
   * Enforce TLS on Auth Endpoints

## Description
  * As a Platform/Security Engineer, I want the authentication endpoints to accept only TLS-encrypted connections and have automated validation checks in CI and monitoring, so that authentication traffic is protected in transit, credentials cannot be sent over insecure channels, and misconfigurations or regressions are detected before reaching production.

## Acceptance Criteria
  * Given an incoming HTTP (non-TLS) request to any /auth/* endpoint, When the request is received by the edge/load-balancer or application, Then the request MUST be rejected with HTTP 426 Upgrade Required, the request MUST NOT be processed or logged with sensitive payloads, a minimal non-enumerating response body MUST be returned, and a sanitized audit event MUST be emitted.
  * Given a client negotiates TLS < 1.2 to any /auth/* endpoint, When the handshake occurs, Then the TLS termination layer (LB/ingress) MUST refuse the connection; no application-layer processing of the request shall occur.
  * Given a valid HTTPS request to an /auth/* endpoint, When the server responds, Then responses MUST include HSTS header per policy (min 30 days max-age and includeSubDomains where applicable) and must not include sensitive data in redirect query params or logs.
  * Given the CI/CD pipeline runs integration tests for authentication, When the automated TLS-enforcement test stage runs, Then the pipeline MUST validate: (1) HTTP to /auth/* is rejected; (2) HTTPS to /auth/* succeeds with a valid certificate chain; (3) negotiated TLS version is >= 1.2; if any check fails, the pipeline MUST block promotion.
  * Given scheduled production checks run, When a certificate is found to expire within 30 days, or an endpoint responds to HTTP, or a weak-cipher/old-protocol is detected, Then alerts MUST be raised to the SRE/security on-call channel and create a ticket in the incident queue.

## Edge Cases
   * What happens when a trusted internal proxy terminates TLS and forwards requests over HTTP?
     - The system will only trust TLS provenance headers (e.g., X-Forwarded-Proto, Forwarded) from configured trusted proxy IP ranges. If the source IP is trusted and the forwarded proto indicates "https", the request is treated as TLS-originated. If the source is not in the trusted list or header is missing/mismatched, the request is treated as non-TLS and rejected per policy.
   * How does the system handle certificate rotation or temporary certificate unavailability?
     - Implement staged, zero-downtime rotation (dual cert serving during rollout) and automated rollback. If no valid certificate is available, the service should fail-safe: reject authentication attempts (503) and immediately alert SRE with actionable diagnostics (which certs failed, rotation step). Ensure runbook exists for manual emergency rollback.
   * What happens when a legacy internal client requires non-TLS access?
     - Default policy: deny. Exceptions require an auditable approval with expiry and must use an alternative secure mechanism (private network, mutual-TLS gateway) documented in infra policies. Any ad-hoc allowance must be temporary and tracked; otherwise requests are rejected.

Notes (non-functional / implementation guidance):
- Minimum TLS version allowed: TLS 1.2 (recommend TLS 1.3 preferred). Cipher list must follow organizational security guidance.
- Do not rely on user-agent behavior for redirects on authentication POSTs. Reject non-TLS POSTs rather than silently redirecting.
- Ensure the CI automated tests include both positive (HTTPS success) and negative (HTTP rejected, weak-cipher rejected) checks, and that they run in gated pipelines before production promotion.
- Logging: never persist credentials; log only sanitized metadata (source IP, endpoint, method, rejection reason). Ensure logs for rejected insecure requests are marked for security audit review.
- IAM/operational: document trusted proxy ranges and the approved exception process for legacy callers; include runbooks for cert rotation and emergency rollback.
