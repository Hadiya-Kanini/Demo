## Story ID

## Story Title
   * Integrate Secrets Vault for Signing Keys

## Description
  * As a platform engineer, I want the auth service to securely store and retrieve signing keys and email provider credentials from a centralized secrets vault so that signing keys and provider credentials are never stored in plaintext on service hosts, key rotation can be performed without redeploying services, and access is auditable and least-privilege enforced.

## Acceptance Criteria
  * Given the secrets vault is provisioned and the auth-service has RBAC read access to the configured vault paths, When the auth service requests the signing private key for token issuance, Then the vault returns the secret (or an encrypted reference) and the auth service signs tokens in-memory without persisting the private key to disk and emits an audit log entry for the access (including key_id, timestamp, and requester identity but not the secret value).
  * Given email provider credentials (SMTP/API keys) exist in the vault at the configured path, When the auth service needs to send an email (verification/password reset), Then it retrieves the credentials at runtime, uses them to send the email, does not log the raw credential values to any logs, and records a masked audit event indicating successful retrieval and use.
  * Given a running system that supports multiple active signing keys (key-set), When a new signing key is promoted to primary in the vault as part of rotation, Then the auth service begins using the new primary key for signing new tokens immediately, continues to validate tokens signed by previous keys until they expire, and logs the rotation event (key_id_old -> key_id_new) without exposing private key material.
  * Given the auth service experiences transient vault connectivity errors during a secret retrieval operation, When the operation is retried, Then the service performs exponential backoff retries, returns 503 Service Unavailable to callers requiring the secret with a Retry-After header, increments monitoring/alerting metrics, and does not fall back to persisting secrets locally.
  * Given the vault returns a secret in an invalid structure or the auth service receives an authorization error (403) for the secret path, When parsing/using the secret, Then the auth service rejects the secret, logs redacted error details, emits an alert to the on-call channel, and prohibits token issuance or email sends that depend on the invalid secret until remediation.

## Edge Cases
   * What happens when the vault becomes unavailable during startup?
     - Handling: By default the service will fail fast at startup with a clear error and critical alert (config flag 'start-fail-on-missing-secrets' = true). If ops explicitly allow degraded mode, the service will start in read-only mode (no token issuance or email sending) and poll the vault with exponential backoff; all attempts to perform operations that require secrets will return 503.
   * How does the system handle an unexpected or malformed secret payload from the vault?
     - Handling: The secret is treated as invalid. The service must not attempt to use it, must redact sensitive data in logs, emit an audit/error event with diagnostic metadata (path, request id, non-sensitive schema validation errors), and notify on-call. The system will return 503 for dependent operations until a valid secret is present.
   * What happens if permissions for the auth-service principal are revoked during runtime (403 Forbidden)?
     - Handling: The service invalidates any in-memory cached secret, logs a redacted security event, emits an alert to SRE, switches to a safe operating mode (no new token issuance or outbound email), and returns 503 for affected endpoints until permissions are restored and verified.

Notes / Implementation Guidance (not part of acceptance criteria but required for developers and reviewers):
- Use a vault provider client that supports encrypted transit and Key/Value versioned secrets (KV v2 preferred) and server-side audit logging. Prefer existing platform-standard vault implementation (INFRA_VAULT_PROVISION).
- Secrets must be cached in-memory with a configurable TTL and cache must be invalidated on vault-notify events or TTL expiry to support rotation.
- Never log secret values; logs must use redaction and include only non-sensitive metadata (key_id, path, timestamps, request_id).
- Implement and document IAM role/policy required for the auth-service (least privilege: read/list on specific paths only).
- Add integration tests: happy-path retrieval, rotation scenario, vault 5xx simulation, malformed secret handling, 403 permission scenario.
- Provide runbook for encryption key rotation, vault failover, and emergency key-swap.
- Estimate: 3 story points (vertical slice: implement retrieval, in-memory use, caching, basic rotation handling, tests).
