## Story Title
   * Implement CI/CD and IaC for Auth Service

## Description
  * As a Platform/DevOps Engineer, I want infrastructure-as-code (IaC) and CI/CD pipelines to deploy the auth service in a highly-available pattern with automated smoke tests, so that deployments are repeatable, auditable, self-validating, and minimize manual intervention and downtime.

## Acceptance Criteria
  1. Given a developer opens a pull request with changes to auth-service code or IaC modules, When CI runs on the PR, Then the pipeline executes unit tests, lints, terraform fmt/validate/plan (for non-prod sandbox), and posts the terraform plan and test results to the PR as artifacts for reviewer inspection; the pipeline must fail the PR check if any critical checks fail.

  2. Given a merge to the protected main branch with required approvals, When the CD pipeline triggers, Then the pipeline applies the IaC to the target environment (staging or production) to provision resources in a highly-available pattern (multi-AZ deployment, autoscaling, appropriate load balancers, health checks), deploys the selected container image from the approved registry, and awaits successful deployment health checks before running smoke tests.

  3. Given deployment has completed, When the automated smoke test suite runs, Then all smoke tests must pass; if they pass, the pipeline marks the deployment as successful and continues promotion policies (if configured). If any smoke test fails, the pipeline must automatically rollback to the last known-good state (or execute the defined rollback strategy), mark the deployment as failed, publish logs and artifacts, and raise alerts/tickets to the configured on-call channel.

  4. Given the pipeline needs runtime secrets (signing keys, DB credentials, TLS certs), When the pipeline runs, Then it must fetch required secrets from Vault (or corporate secrets manager) using a least-privilege service principal; failure to retrieve required secrets causes the pipeline to abort with an auditable error and no plaintext secret exposure in logs.

  5. Given multiple successive pipeline runs and IaC executions, When IaC apply completes, Then the infrastructure must be idempotent (subsequent terraform plan shows no diffs unless changes were made), and post-deploy compliance checks (terraform validate, policy-as-code checks, and configured SAST gating) must complete successfully; otherwise the environment remains flagged unhealthy and the pipeline escalates.

## Edge Cases
   * What happens when secrets vault is unavailable?
     - Handling: The pipeline must detect the inability to authenticate or retrieve secrets, abort the run before any resource creation, log the failure to secure artifact storage, trigger an alert to the platform on-call channel, and create an incident/ticket. No fallback to insecure secret storage is allowed.

   * How does the system handle provider API rate-limits, intermittent failures, or partial apply (state drift)?
     - Handling: The pipeline must run terraform plan first and only proceed to apply if plan is approved. If an apply fails mid-way, the pipeline must run a state refresh and attempt a safe rollback or targeted destroy of ephemeral resources where feasible, produce a detailed resource-level error report, mark the deployment as failed, and create a remediation ticket. Manual intervention is required for destructive/ambiguous rollbacks.

   * What happens if smoke tests are flaky or environment-dependent?
     - Handling: The pipeline will retry the smoke test suite up to a configured retry limit (default 1 retry). If tests remain failing, the pipeline marks the deployment as failed, attaches the test artifacts and logs, and requires manual triage. A flaky-test flagging process will add the test to a flakiness review queue (UXR/QA) to either fix tests or improve environmental stability.
