## Story Title
   * Implement DB Migrations and Rollback

## Description
  * As a Database/Platform Engineer, I want database migration scripts and safe rollback procedures to add required authentication fields and supporting indexes to the production schema, so that the authentication service can dependably store new auth metadata (password_hash, hash_algorithm, failed_attempts, locked_at, reset_token_hash, reset_token_expires_at) and query by email efficiently without risking data loss or prolonged downtime.

## Acceptance Criteria
  * Given a production-like database snapshot, When the migration is executed in a staging environment, Then the schema must be updated to include the new columns (password_hash, hash_algorithm, salt if required, failed_attempts, locked_at, reset_token_hash, reset_token_expires_at) and the specified indexes (unique index on lower(email), index on reset_token_hash, index on locked_at) without data corruption and the migration must complete within the maintenance window defined in the runbook.
  * Given the migration is idempotent, When the same migration script is applied a second time, Then it must detect that changes already exist and exit cleanly without creating duplicate columns/indexes, failing, or altering existing data.
  * Given rollback is triggered (automated test or manual via CI/CD), When rollback script is executed, Then the database must be restored to the previous schema and data state for schema-level changes performed by the migration and a verification step (checksums or sample row comparisons) must confirm restoration for sampled records.
  * Given large data volume and production traffic, When running the migration in a production-like environment, Then the migration must use online, non-blocking techniques (e.g., CREATE INDEX CONCURRENTLY or equivalent), perform backfill in bounded batches, and demonstrate that typical login queries' p95 latency remains within the accepted SLA; the migration must include a documented, tested runbook (including preflight checks, backout steps, and monitoring signals).
  * Given rows with legacy hash formats or malformed values, When the migration runs, Then it must not drop or irreversibly rewrite those values; instead the migration must log affected rows to a report for follow-up and ensure the application handles such rows via migration-on-login, and any destructive actions must require manual approval before execution.

## Edge Cases
   * What happens when the migration partially fails midway (e.g., process killed, network partition)?
     - Handling: The migration must write progress checkpoints to a migrations_state table. On detection of partial run, automated rollback for completed atomic steps must be available or the operator must be able to resume safely from the last consistent checkpoint via documented runbook. Automated health checks must prevent marking deployment as successful when state is partial.
   * How does system handle very large tables / long-running DDL?
     - Handling: Use online/non-blocking schema change techniques supported by the DB (e.g., CREATE INDEX CONCURRENTLY for Postgres), perform backfill updates in small batches with checkpointing, and include metrics/ETA estimation. Provide a throttling mechanism and an abort path that leaves data consistent.
   * How does the system handle legacy/corrupt password hash formats found during migration?
     - Handling: Do not modify or delete legacy hash bytes. Record IDs and details in a migration report for manual review, and rely on application-level "migration-on-login" to re-hash on successful login. Ensure rollback preserves original values for forensic analysis.

Notes/Implementation Guidance (non-mandatory for acceptance, but required for handoff):
- Provide migration scripts as both SQL (idempotent statements) and a migration runner (Flyway/ Liquibase / custom) with explicit preflight checks.
- Include a migrations_state table to record step start/completion, timestamps, and operator notes.
- Provide a production runbook with ordered steps: preflight checks, run in a canary DB cluster, staged rollout, monitoring signals (DB locks, index build progress, query latency), and rollback steps with required approvals.
- Include automated integration tests that run migration -> data verification -> rollback on CI against a production-sized sample dataset.
