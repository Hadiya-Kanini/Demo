## Story ID: us_004

## Story Title: Implement Migration-on-Login Rehash

## Description
  * As a backend engineer maintaining authentication, I want the login flow to detect legacy password hashes, verify them safely, and rehash passwords to the new algorithm on successful authentication, so that user credentials are migrated incrementally without forcing password resets and without compromising security or availability.

## Acceptance Criteria
  * Given a user account whose stored password uses the legacy hashing algorithm, When the user successfully authenticates with their plaintext password, Then the system verifies the legacy hash, rehashes the plaintext with the configured new algorithm and parameters, atomically persists the new hash and algorithm metadata to the user record, and proceeds with normal token issuance and successful response.
  * Given a user account whose stored password already uses the new hashing algorithm, When the user successfully authenticates, Then no migration or write to the password fields occurs and the login completes normally.
  * Given an authentication attempt with an incorrect password for a legacy-hash account, When verification fails, Then the system does not attempt any rehash, increments the failed-attempt counter per existing protection policies, returns the standard authentication failure response (non-enumerating), and records a metric for failed verification without exposing hash details.
  * Given a persistence error (DB write failure or optimistic concurrency conflict) occurs while saving the rehashed credentials after successful verification, When such error arises, Then the system preserves the existing stored credentials (no partial overwrite), allows login to succeed if verification had passed, emits structured logs/metrics for "migration_persistence_failure" including account id and failure reason (no sensitive data), and flags the account for manual or automated reconciliation.
  * Given the feature is implemented, When continuous integration runs, Then unit tests and integration tests cover detection of legacy formats, verification success and failure, atomic update behavior, concurrent-login race conditions, rollback/persistence-error handling, and CI will fail the build if any test in the migration-on-login suite fails.

## Edge Cases
   * What happens when a legacy hash is malformed or uses an unsupported format?
     - Handling: Treat as unverifiable. Do not attempt rehash. Increment failed-attempt counters, return a standard authentication failure message, emit a structured event/metric "legacy_hash_unverifiable" with non-sensitive diagnostic info, and include the account in a reconciliation report for ops to remediate (e.g., prompt user to reset password).
   * How does the system handle concurrent login attempts that both attempt to rehash the same account?
     - Handling: Implement optimistic concurrency (row version or updated_at) or use a short DB row-level lock within the authentication transaction so only one rehash can succeed. Clients that lose the race will detect that the algorithm field changed on retry and skip rehash. Ensure idempotency and avoid double-writes. Log a "migration_race_detected" metric when conflicts occur.
   * What if database persistence of the new hash fails after successful verification (partial failure scenario)?
     - Handling: Do not modify or delete the original stored hash. Allow the authentication to succeed (so user is not blocked) if verification succeeded. Record a high-severity metric and audit log for "migration_persistence_failure". Surface the account id to reconciliation tooling (without logging sensitive data) and ensure retries/backoff and operator alerts are available.
