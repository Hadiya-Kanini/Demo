## Story Title
   * Implement Reset Token Storage

## Description
  * As a backend engineer, I want the database to store password-reset tokens as secure hashed values with TTL and supporting indexes, so that the system can validate single-use reset tokens atomically, minimize risk of token theft, enable efficient expiry/purge, and support integrity and concurrency tests.

## Acceptance Criteria
  * Given a request to create a password-reset token for user X, When the service generates the token, Then the system stores only a secure hash of the token (reset_token_hash), a per-token salt or HMAC metadata (reset_token_salt or pepper reference), created_at timestamp, expires_at timestamp (TTL), and reset_token_used_at as NULL; the raw token value is returned to the caller but never stored in plaintext in logs or DB.
  * Given a valid, unexpired, unused reset token, When the token is presented to the reset endpoint, Then the system validates by hashing the presented token with the same salt/pepper, locates the matching record using the index, and atomically marks the token as used (set reset_token_used_at) in the same transaction that authorizes the password reset so the token cannot be re-used.
  * Given an expired or already-used token, When a client attempts to use it, Then the system rejects the request with a generic error response (no account enumeration), does not change user credentials, and writes an audit/event log entry with masked token reference and reason (expired/used).
  * Given a high-concurrency race (multiple simultaneous attempts to redeem the same token), When concurrent validations occur, Then only one validation succeeds because the DB update is conditional (e.g., UPDATE ... WHERE id = ? AND reset_token_used_at IS NULL AND expires_at > NOW()) and integration tests verify rows-affected = 1 for the successful attempt and 0 for others.
  * Given the schema change is applied, When migrations run, Then the migration creates efficient indexes: a non-unique index on reset_token_hash (or partial index WHERE reset_token_used_at IS NULL) for direct lookup, and a composite index on (user_id, expires_at) to support TTL purge jobs; the migration includes rollback and passes data integrity tests.

## Edge Cases
   * What happens when token hash collision occurs?  
     - Handling: On rare detection of multiple unused rows with identical reset_token_hash, the system will treat this as a suspected collision: invalidate all matching unused tokens, emit a high-severity metric/alert and an audit log entry, and require issuing a new reset token. Tests will include a simulated collision scenario to verify the invalidation and alert behavior.
   * How does system handle concurrent redemption attempts (race condition)?  
     - Handling: Use DB atomic conditional update or row-level locking ensuring single-use (UPDATE ... WHERE id = ? AND reset_token_used_at IS NULL AND expires_at > NOW()). If the update affects 1 row, operation succeeds; if 0, return rejection. Integration test must simulate concurrent redemption to assert only one success.
   * What happens when DB or secret manager (pepper) is unavailable during create/validate?  
     - Handling: On DB/unavailability or secret manager error, the create or validate request should return 503 (or configured transient error) and no plaintext token is logged or persisted. Implement retry/backoff for the caller and capture metrics/alerts. Ensure tests cover transient failures and confirm no tokens were persisted in plaintext.

Notes / Implementation Guidance (non-normative)
- Schema suggested fields (example):
  - reset_token_id UUID PRIMARY KEY
  - user_id UUID (FK -> users) NOT NULL
  - reset_token_hash BYTEA / VARCHAR NOT NULL
  - reset_token_salt BYTEA / VARCHAR NULL (or store HMAC metadata)
  - created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
  - expires_at TIMESTAMP WITH TIME ZONE NOT NULL
  - reset_token_used_at TIMESTAMP WITH TIME ZONE NULL
  - purpose ENUM('password_reset') or reason column (optional)
- Index recommendations:
  - CREATE INDEX idx_reset_token_hash_not_used ON password_reset_tokens (reset_token_hash) WHERE reset_token_used_at IS NULL;
  - CREATE INDEX idx_reset_user_expires ON password_reset_tokens (user_id, expires_at);
  - Consider partial/ttl indexes per DB capability to speed purge jobs.
- Security recommendations:
  - Use a salted hash or HMAC with server-side pepper from vault (dependency: us_infra_001). Do not store raw token or pepper in logs.
  - Limit token TTL to project-defined policy (e.g., 1 hour) and configurable via environment.
- Testing:
  - Unit tests for hashing and verification logic.
  - Integration test for create->validate->single-use behavior.
  - Concurrency test simulating N parallel validation attempts asserting only one success.
  - Migration tests to verify rollback and index creation.
- Rollout:
  - Apply migration in maintenance window for heavy tables; include backfill script if existing tokens must be transformed.
- Observability:
  - Emit metrics for token creation, validation success/failure, expirations, collisions, and DB errors. Log audit events with masked token identifiers only.

The story follows INVEST: Independent (DB-level and service logic scoped), Negotiable (implementation details like hashing algorithm and TTL are configurable), Valuable (enables secure password recovery), Estimable (expected ~2-5 points depending on migration complexity), Small (focused on token storage & validation), Testable (clear acceptance criteria and tests described).
