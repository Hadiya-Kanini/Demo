## Story Title
   * Provide Seed Data and Integrity Tests

## Description
  * As a developer/test engineer, I want seeded test fixtures and automated data-integrity tests for authentication scenarios (legacy password hashes, locked accounts, reset tokens), so that other stories and CI pipelines have reliable, idempotent test data and early automated detection of schema or data format regressions.

## Acceptance Criteria
  * Given a clean development or CI test database, When the seed script is executed, Then it creates the required fixtures: 
    - legacy-hash user with algorithm metadata and legacy_hash field matching the expected legacy format regex,
    - locked account with locked_at timestamp and failed_attempts > 0,
    - password-reset user with a stored hashed reset token and expiry timestamp,
    - admin user and a standard active user; all created users must have unique emails and required fields populated.

  * Given the repository CI pipeline, When the data-integrity test suite runs, Then it verifies:
    - uniqueness of email addresses (DB constraint and test-level check),
    - allowed values for password_hash_algorithm field,
    - reset tokens are stored hashed and expiry timestamps are valid UTC ISO8601 datetimes,
    - failed_attempts is a non-negative integer,
    - locked_at is present only when account is locked,
    - legacy_hash fixture conforms to the expected legacy format regex used by migration tests.

  * Given the seed script is executed multiple times, When run consecutively, Then it behaves idempotently (no duplicate records), reports a clear summary of upserts/skips, and exits with success if final state matches expected seeded baseline.

  * Given a CI or local teardown operation, When teardown is triggered, Then seeded data is removed or reset to a deterministic baseline so subsequent test runs start from a known state, and tests validate teardown completed successfully.

  * Given a seeded legacy-hash fixture and a mocked successful login event in integration tests, When migration-on-login tests run (integration harness), Then the integrity tests confirm required legacy-hash fields exist and match expected formats so downstream migration logic has valid input (note: this story provides the fixtures and checks; actual migration behavior is validated in migration story).

## Edge Cases
   * What happens when a seeded legacy hash has a corrupted format?
     - Handling: Integrity tests fail with a clear diagnostic pointing to the malformed fixture. Seed scripts will not introduce corrupted legacy hashes by default; an intentionally-malformed fixture may be included only behind a TEST_MALFORMED_LEGACY toggle and must be documented. CI must treat unintended malformed fixtures as a blocking failure.

   * How does the system handle reset-token hash collisions or duplicates?
     - Handling: Seed generator hashes tokens using the same algorithm as production, checks for collisions, and regenerates until unique. Integrity tests assert uniqueness of stored reset token hashes. If a collision persists beyond a safe retry limit, seeding fails and returns an actionable error.

   * What happens when multiple CI jobs or developers run the seed concurrently?
     - Handling: Seed script must perform idempotent upserts within transactions and rely on DB unique constraints. On transient concurrency conflicts, the script retries up to a configurable number of attempts; persistent conflicts abort with a deterministic error. Tests verify idempotency and that upsert semantics maintain invariant state.
