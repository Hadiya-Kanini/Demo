## Story Title
   * Implement Password Reset Completion

## Description
  * As a user who requested a password reset, I want to submit a valid reset token and a new password to POST /auth/reset, so that my account password is securely updated, account protections (failed attempts / locked_at) are cleared, and all active sessions are revoked to prevent unauthorized access.

## Acceptance Criteria
  1. Given a valid, unexpired, single-use reset token stored as a hashed token reference and associated with a user, When a POST /auth/reset request is received with the token and a new password that satisfies the current password policy, Then:
     - The user's password stored in the database is replaced atomically with a new hash using the current configured hashing algorithm and parameters (salt, iterations).
     - The stored reset token is invalidated/removed (single-use).
     - failed_attempts is reset to 0 and locked_at is cleared (null).
     - All active sessions/tokens for the user are revoked (e.g., increment password_version or session_revoked marker).
     - The API returns HTTP 200 with a non-enumerative success response and does not leak whether the token was tied to a specific account.
     - An audit log event and security telemetry record is emitted (no plaintext secrets).

  2. Given a POST /auth/reset where the proposed new password violates the password policy (minimum length, required character classes, reused recent password, or present in breach-list), When validated, Then:
     - The API returns HTTP 400 with a standardized error code (INVALID_PASSWORD_POLICY) and a machine-readable message describing the policy violation.
     - No changes are made to the stored password hash, reset token, failed_attempts, locked_at, or sessions.
     - The attempt is logged to security telemetry (rate-limited to prevent info leaks).

  3. Given a POST /auth/reset request containing an invalid, expired, or already-used token, When processed, Then:
     - The API returns HTTP 400 with a standardized error code (INVALID_OR_EXPIRED_TOKEN) and a non-enumerative message.
     - No changes are made to the user's password, failed_attempts, locked_at, or session state.
     - The failed attempt increments any relevant protection counters (subject to rate-limit/anti-enumeration rules) and is recorded in telemetry.

  4. Given repeated reset attempts that exceed configured rate limits (per-IP or per-account), When limits are hit, Then:
     - The API returns HTTP 429 with a Retry-After header and a non-enumerative message.
     - The event is recorded in rate-limit telemetry and security logs.
     - The system preserves tokens and existing account state and does not reveal account existence.

  5. Given underlying system failure during processing (e.g., hashing library throws, DB write fails), When the failure occurs, Then:
     - The request is rolled back (no partial state changes), the reset token remains valid, and the API returns HTTP 500 with a generic error.
     - An operational alert is emitted and detailed error is recorded in secure logs for investigation.

## Edge Cases
   * What happens when two concurrent requests attempt to use the same reset token?
     - Handling: Implement single-use enforcement via an atomic DB operation (e.g., UPDATE ... WHERE reset_token_hash = ? AND expiry > now AND token_used = false RETURNING ...). Only the winning transaction proceeds to change the password and mark token used; the other(s) fail validation and receive INVALID_OR_EXPIRED_TOKEN. Ensure transactions and row-level locks prevent race conditions.

   * How does system handle cryptographic or DB write failures during the reset?
     - Handling: Wrap hashing and DB updates in a single transaction when possible; on any failure, perform a rollback so no state (password, token, failed_attempts, sessions) is mutated. Return HTTP 500 with a generic message, log detailed failure in secure logs, and trigger an ops alert for investigation.

   * What if the new password equals the previous password or is in a breach database?
     - Handling: Enforce policy to prevent immediate reuse of last N passwords and check breach lists. If violated, return HTTP 400 INVALID_PASSWORD_POLICY and do not modify account state.

Notes for implementation (non-normative, for devs/testers)
- Use a hashed/stored representation of reset tokens (do not store plain token). Compare using constant-time comparison against hash of provided token.
- Implement server-side password policy checks centrally (reuse existing password policy service if available) and ensure error messages are machine-readable but non-enumerative.
- Revoke sessions by the agreed mechanism (session table expiration, token blacklist, incrementing password_version claim), and ensure revocation is eventually consistent and testable.
- Emit audit events with: event_type=password_reset, user_id (or internal id), initiating_ip (subject to privacy rules), outcome, and timestamp. Do not include plaintext passwords or tokens in logs.
- Ensure story size remains within agreed sprint sizing limits; split further if DB transaction or session revocation design is large.
