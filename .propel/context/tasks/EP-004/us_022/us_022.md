## Story ID
   * ID Format: US_<unique seq num>
   * ID: us_022

## Story Title
   * Implement Secure Reset Token Generation

## Description
  * As a backend developer, I want the system to generate cryptographically-random, single-use password-reset tokens, store only a hashed representation with a TTL, and ensure collision resistance so that account recovery is secure, non-enumerable, and resilient to replay or token-guessing attacks.

## Acceptance Criteria
  * Given a valid password-reset request for an existing account, When the system generates a reset token, Then:
    - The token must be generated using a CSPRNG with at least 256 bits (32 bytes) of entropy and encoded in a URL-safe format for inclusion in email/SMS links.
    - Only a server-side keyed hash (e.g., HMAC-SHA256 using a secret retrieved from the secrets vault) of the token is written to persistent storage along with user_id, created_at, expires_at (TTL), and used=false.
    - The plaintext token is returned only to the delivery subsystem for outbound delivery and MUST NOT be stored in logs, traces, or persistent telemetry.

  * Given the system must be collision resistant, When a generated token's hashed value collides with an existing token hash (global or per-user index), Then:
    - The generator must retry generation up to 5 times; on the first non-colliding hash the record is persisted and issued.
    - If all retries collide, the issuance aborts, returns 500, increments a collision_failure metric, and writes only non-sensitive diagnostic info to logs.

  * Given a user submits a plaintext reset token to reset their password, When the system validates the token, Then:
    - The system computes the keyed hash of the provided plaintext token and searches for a matching, unexpired, unused token record.
    - If found, the system atomically marks the token as used (or deletes it) at the same time as setting the new password to guarantee single-use semantics and returns success.
    - If not found, expired, or already used, the system returns a generic failure response (no disclosure of existence), and emits an audit event indicating token_validation_failed with non-sensitive context.

  * Given operational/config requirements, When the service starts or secret rotation occurs, Then:
    - Secrets used for keyed hashing are loaded from the secrets vault and rotation is supported without invalidating existing tokens unexpectedly (documented rotation strategy).
    - The default TTL for reset tokens is 1 hour and must be configurable; expired tokens must be considered invalid even if present in the store.
    - Issuance and consumption must emit non-sensitive audit events (user_id, event_type, timestamp) and metrics (tokens_issued, tokens_consumed, token_collision_retries, token_expired, token_failed_consume) while never including plaintext tokens.

## Edge Cases
   * What happens when the CSPRNG fails or returns an error?
     - The system must detect the failure, not issue a token, return HTTP 500 to the caller, increment an alerting metric (csprng_failure), and log only non-sensitive diagnostics. No partial or plaintext token material may be emitted to logs.

   * How does the system handle repeated hash collisions for the same user or high token issuance volume?
     - The generator retries up to 5 times; if collisions persist, issuance fails with 500 and a collision metric is incremented. The system must also expose an operational alert (high collision rate) for investigation. Implement index on hashed_token to minimize collisions and ensure DB uniqueness constraints are handled gracefully.

   * What happens when a token is consumed concurrently or a DB write fails while marking used=true?
     - Consumption must be implemented as an atomic DB transaction or compare-and-set operation. If marking the token used fails after verification, the entire operation must roll back and return a transient error; client-side should retry the consume operation once. The system must guard against race conditions so that at most one successful consume can complete.

Notes on implementation approach and constraints (non-normative):
- Use a vetted cryptographic library for CSPRNG; prefer platform OS CSPRNG APIs.
- Use keyed hashing (HMAC-SHA256 or approved equivalent) with the HMAC key stored in secrets vault (us_infra_001) rather than unsalted/plain SHA to protect against bulk DB leaks.
- Keep token hash columns indexed and apply a unique constraint where appropriate to enable fast collision checks.
- Maintain observability metrics (counts) and alerts without ever including plaintext tokens in telemetry or logs.
- Keep the story scope small and testable: include unit tests for token generation, integration tests against DB for collision handling, and an end-to-end test for single-use consumption.
