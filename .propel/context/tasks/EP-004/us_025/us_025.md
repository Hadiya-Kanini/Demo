## Story Title
   * Test Reset Token Expiry and Single-use

## Description
  * As a QA Engineer, I want automated and integration tests that validate reset-token single-use semantics, expiry handling, concurrent consumption attempts, and non-enumeration behavior for the forgot-password flow, so that the password recovery mechanism is secure, reliable, and privacy-preserving.

## Acceptance Criteria
  * Given a valid, unexpired reset token that was issued and delivered to the user, When the user submits a password reset request with that token, Then the system resets the password, invalidates the token (single-use), returns a 200 OK success response, and creates an audit event recording token consumption.
  * Given the same previously-consumed token, When any client attempts to reuse that token to reset a password, Then the system responds with a generic failure (e.g., 400 Bad Request and body 'Invalid or expired reset token'), does not disclose whether the token was previously used, and logs the reuse attempt as a failed token-consumption event.
  * Given a reset token that has passed its expiry timestamp, When a reset attempt is made with that token, Then the system responds with the same generic failure message used for invalid tokens, does not reveal existence or expiry details, and records the attempt in logs with an 'expired' reason.
  * Given two concurrent requests that present the same valid reset token, When the server processes these requests, Then exactly one request succeeds in resetting the password and consuming the token while the other request fails with the generic 'Invalid or expired reset token' response; tests must assert atomic token consumption (DB-level compare-and-delete or transactional semantics) and that audit logs show one successful consumption and one failed attempt.
  * Given a request to the forgot-password endpoint for an email address that exists or does not exist, When the endpoint is invoked, Then the endpoint must always return 200 OK with an identical response body and similar timing (within an acceptable variance threshold), must not reveal whether an account exists, and tests must assert that emails are only sent when the account exists and that no side-channel via status, body, or timing leaks user enumeration.

## Edge Cases
   * What happens when a token is tampered with (modified characters or truncated)?
     - The system must fail with the same generic 'Invalid or expired reset token' response, never disclose details, and log the tampering attempt with severity for security monitoring.
   * How does the system handle a token collision during issuance (extremely low-probability)?
     - Issuance logic must detect duplicate token-hash collisions and retry generation up to a configurable limit; tests should simulate a collision (mock/hook) and verify that the issuance either succeeds after retry or fails with an internal alert while not leaking details to the user.
   * How are partial failures handled (e.g., password update succeeds but marking token as consumed fails)?
     - Tests must simulate DB failures and assert that the system uses transactions or compensating actions. If true atomicity is not possible, the system must emit an alert/audit record and implement a compensating job to reconcile token state; end-user response must remain generic and not reveal token state.

- Tests should include unit tests for token verification logic, integration tests hitting the password-reset API and the forgot-password API, and concurrency tests (multi-threaded or parallel requests) to validate atomic consumption.
