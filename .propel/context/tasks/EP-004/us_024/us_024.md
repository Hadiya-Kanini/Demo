## Story Title
   * Integrate Email Sending with Retry Behavior

## Description
  * As a backend service owner, I want to integrate the system with an external email provider using credentials stored in the secrets vault, implement reliable send retries with exponential backoff and a dead-letter path, and ensure password-reset emails are enqueued with observability, so that reset emails are delivered reliably, failures are observable and auditable, and user-facing password recovery remains non-enumerable and resilient.

## Acceptance Criteria
  * Given the application has valid email provider credentials retrieved from the vault and a generated password-reset token exists for a user, When the password-reset flow enqueues an email job, Then the system must persist an enqueue record with a dedupe key (token-id), increment an "emails.enqueued" metric, attach a trace/span id, and return internal enqueue success while the public /auth/forgot endpoint remains non-enumerating (returns 200).

  * Given the email provider returns a transient error (HTTP 5xx or 429 with Retry-After), When the email-sender worker processes the job, Then the worker must retry using exponential backoff (configurable base delay, max attempts), increment "emails.retries" for each retry, and on success mark job delivered and increment "emails.sent".

  * Given the email provider returns a permanent client error (HTTP 4xx such as invalid recipient), When the worker processes the job, Then the system must not retry, must move the message to a dead-letter queue with failure reason metadata, increment "emails.failed_permanent", and write an audit/log entry with masked provider response and token-id.

  * Given vault credentials are unavailable at send time, When a send attempt is required, Then the worker must use cached credentials if present; if not present, it must increment "emails.credential_errors", defer the job with exponential backoff for vault refresh, emit an operational alert, and write a secure audit log entry. Do not reveal credential errors to external callers.

  * Given a job reaches the configured maximum retry attempts without success, When max attempts is exceeded, Then the job must be moved to a dead-letter queue containing retry history (attempt timestamps, last error), increment "emails.dead_letter", and surface an audit record for manual investigation.

## Edge Cases
   * What happens when the secrets vault is unavailable or returns a permission error?
     - The system should use a securely cached credential (if not expired). If no cached credential exists, the worker must defer the job using backoff and jitter, increment a credential error metric, write a masked audit log, and emit an alert to ops. Public endpoints must not leak vault state; /auth/forgot still responds 200.

   * How does system handle provider rate-limit (429) with a Retry-After header longer than token TTL?
     - If honoring Retry-After would cause the token to expire before delivery, the worker should mark the job as dead-lettered with reason "token_expired_before_send", increment "emails.dead_letter", and create an audit entry including the Retry-After value and token expiry timestamp.

   * What happens if the token expires or is revoked before the email is delivered?
     - The worker must check token validity at send time. If token is expired/revoked, it must not send the email, move the job to dead-letter with reason "token_invalid_at_send", increment "emails.dead_letter", and record audit details for investigation.
