## Story ID: us_021

## Story Title: Implement Forgot Password Endpoint

## Description
  * As a backend developer responsible for the authentication API, I want a POST /auth/forgot endpoint that accepts an email and returns a consistent response so that clients cannot enumerate whether an account exists and, when the account exists, a password-reset email is enqueued and a secure single-use reset token is persisted for recovery.

## Acceptance Criteria
  * Given a syntactically valid JSON request { "email": "user@example.com" }, When the client calls POST /auth/forgot, Then the API responds with HTTP 200 and a generic message (for example: { "message": "If an account with that email exists, you will receive an email with instructions." }), and if an account exists the system enqueues a password-reset email job and persists a hashed single-use reset token with an expiry timestamp and audit record.
  * Given a syntactically valid JSON request with an email that does not exist in the user store, When the client calls POST /auth/forgot, Then the API responds with HTTP 200 and the identical generic message and does NOT enqueue a password-reset email for a non-existent account, while emitting an internal metric/log for the attempted request.
  * Given a request that is missing the email field or contains an invalid email format, When the client calls POST /auth/forgot, Then the API responds with HTTP 400 and a validation error payload describing the schema/format issue (no 200 response for malformed requests).
  * Given that persistence or enqueueing fails (e.g., DB write error, message queue downtime), When POST /auth/forgot is invoked with a syntactically valid request, Then the API still returns HTTP 200 to the caller (to prevent enumeration), logs the failure with sufficient context, increments error metrics, and triggers background retry/recovery according to retry policy (and creates an alert if retry policy is exhausted).

## Edge Cases
  * What happens when token generation collides with an existing token hash?
    - Handling: Token generation runs in a background transactional flow; if the raw token (after hashing) collides with an existing valid token hash (extremely unlikely), the worker regenerates the raw token up to N attempts (configurable, e.g., 3). If all attempts fail, persist an error audit event and alert on-call. The API response to the caller remains HTTP 200.
  * How does the system handle concurrent forgot-password requests for the same email (race condition)?
    - Handling: Token persistence uses an atomic transaction that creates a new token row and marks prior tokens for that account as revoked/invalid; only the most recently created token is accepted by the reset flow. Concurrent requests will result in multiple tokens but validation accepts only the latest (or tokens carry created_at and a single-use flag enforced at verification).
  * How does the system handle high-frequency abuse (rate-limiting) without enabling account enumeration?
    - Handling: Apply server-side rate limiting (per-IP and global thresholds). If a request is rate-limited, respond with HTTP 200 and the same generic message (do not include Retry-After header or any hints that could differentiate account existence). Internally record the rate-limit event and escalate based on thresholds.

Notes (implementation guidance and constraints)
- Endpoint contract: POST /auth/forgot accepts Content-Type: application/json, body: { "email": "<email>" }.
- Success body should be intentionally generic and identical whether or not an account exists.
- Tokens must be generated as cryptographically secure random values, stored as hashes (e.g., SHA-256/HMAC) with associated account_id, created_at, expires_at, and single-use flag.
- Background job responsibilities: create token record, enqueue email job with template id and minimal payload (reset_link contains raw token + secure one-time URL), and handle retry/error semantics. Raw token must only be present in the email payload and never stored in plaintext.
- Logging/monitoring: emit metrics for requests, enqueues, failures, rate-limits, and token generation errors. Ensure no logs or traces leak account existence to unprivileged stores.
- Security: use vault/secret manager for email provider credentials and ensure TLS enforced for endpoint.
- Size/Estimation: This story is intended as a small vertical slice (<= 5 points): implement API contract, validation, token creation + enqueue path, basic audit/logging, and unit/integration tests. Further stories should add hardening (rate-limiter tuning), observability, and localization of email content.
