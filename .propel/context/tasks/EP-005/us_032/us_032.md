## Story Title
   * Create Metrics, Dashboards and Alerts

## Description
  * As a Site Reliability Engineer, I want authentication events instrumented to produce aggregated metrics for successes, failures, lockouts, and password resets so that we can monitor authentication health, detect anomalous spikes quickly (potential incidents or attacks), and trigger actionable alerts with context and runbooks for rapid response.

## Acceptance Criteria
  * Given the auth service is running and instrumented, When authentication events occur (success, failure, account_lockout, password_reset_requested, password_reset_completed), Then metrics for each event type are emitted to the metrics backend with:
    - stable metric names (e.g., auth.requests.total, auth.success.total, auth.failure.total, auth.lockout.total, auth.reset.requests.total, auth.reset.completed.total),
    - tags: service, environment, region, client_ip_hash, account_id_hash, outcome,
    - counters and derived rates (per-minute, per-5m moving average),
    - no PII in metric names or tags,
    - retention and rollup configured per platform standards.
  * Given the metrics backend has collected at least 30 minutes of baseline data, When a failure-rate anomaly is detected (configurable threshold or statistical anomaly detector), Then an alert is created that:
    - triggers according to on-call routing rules,
    - includes the metric, current value, baseline, percent increase, top affected buckets (region, client_ip_hash top-N), relevant logs link, and runbook,
    - supports auto-escalation if unresolved per configured intervals.
  * Given the auth metrics exist, When an engineer opens the Auth Overview dashboard, Then it displays:
    - time-series for auth total requests, success rate, failure rate, lockouts, reset requests/completions,
    - charts for latency (p50/p95/p99) of auth endpoints,
    - top-N lists for account_id_hash and client_ip_hash by failure count,
    - filters for environment/region/time-range and drilldowns to logs and incidents.
  * Given alert suppression rules or maintenance windows have been configured, When suppression is active, Then alerts triggered by auth-metric rules are suppressed, and a suppressed-notification is recorded in the incident system with the reason and time window.
  * Given sampling and cardinality controls are enabled, When traffic volume or tag cardinality exceeds safe limits, Then sampling is applied to high-volume events and a high-cardinality alert increments so SRE can tune aggregation or adjust tagging.

## Edge Cases
   * What happens when the telemetry backend is unreachable?
     - Handling: The service records telemetry_emit_failures counter (low-cardinality) and writes metrics to a short-lived in-memory persistent buffer; a periodic background worker attempts to flush buffered metrics when connectivity is restored. Emit a "telemetry.backend.unavailable" health metric and trigger a paging alert if failures cross threshold X within Y minutes. Ensure auth request paths remain performant and do not block on telemetry writes (non-blocking, best-effort).
   * How does the system handle missing or malformed event fields (e.g., missing account_id or malformed IP)?
     - Handling: Emit metrics with a deterministic default tag value ("unknown" or "invalid") and increment auth_event_malformed_total (low-cardinality). Log one sample per-minute of the malformed event payload to a secure logs stream for investigation. Do not expose or log raw PII.
   * What happens when cardinality spikes (many unique account IDs or IPs) produce high cost or saturation?
     - Handling: Implement enforced cardinality limits and deterministic hashing of account_id and IP to fixed buckets; apply adaptive sampling to high-cardinality event types while always emitting global aggregate counters. Trigger a high-cardinality alert for SRE to investigate and either increase aggregation or modify tagging strategy.
