## Story ID

   * ID: us_030

## Story Title
   * Define Auth Event Schema and Logging

## Description
  * As a Platform Engineer, I want a standardized authentication event schema and centralized logging integration for login, failure, lockout, reset and admin actions, so that security, compliance, and operations teams can reliably monitor, alert on, audit, and investigate authentication activity across tenants and services.

## Acceptance Criteria
  * Given the auth service is running, When any authentication lifecycle event occurs (successful login, failed login, lockout, password reset request/completion, admin unlock/role change), Then the service emits a structured event that conforms to the agreed schema version and forwards the event to the centralized logging ingestion endpoint within acceptable latency (synchronous send or asynchronous queued send completed within 30s under normal load).
  * Given an operator queries logs, When they filter by event_type and user_id or masked_user_identifier, Then logs return entries including mandatory indexed fields: event_id (UUID), event_version, timestamp (UTC ISO8601), event_type (enum), outcome (success/failure), user_id (nullable), masked_user_identifier, actor_id (nullable), client_ip, source_service, request_id (correlation_id), tenant_id, reason_code, metadata (JSON), retention_days, and audit=true|false.
  * Given the centralized logging endpoint is unreachable or returns 5xx, When the auth service attempts to send an event, Then the service enqueues the event to a bounded retry queue (in-memory with disk spillover), increments a metric auth.logging.send_failure, continues to respond to the API client without blocking on logging, and retries emission with exponential backoff up to the configured attempts before moving the event to a dead-letter queue and creating an operational alert.
  * Given an administrative/authz-changing action occurs, When the event is emitted, Then the event must also be recorded as an immutable audit record (append-only) or flagged audit=true in the log with actor_id and action_details, and the record must be retrievable for compliance purposes for the configured retention period.
  * Given any event contains privacy-sensitive or secret fields, When constructing and validating the event payload, Then the logging pipeline must mask or hash those fields (no plain-text passwords, tokens, or raw PII), and the sanitizer must be enforced by schema validation tests in CI that fail if disallowed fields appear unmasked.

## Edge Cases
   * What happens when the logging endpoint is down or rate-limited?
     - The system persists events to a bounded local retry queue with disk spillover. It emits an operational metric (auth.logging.send_failure) and an alert when queue depth or dead-letter counts cross thresholds. The auth API response remains unaffected (non-blocking). After configured retry attempts events move to a dead-letter queue for manual review.
   * How does the system handle accidental inclusion of sensitive data (passwords, tokens, PII) in logs?
     - A sanitizer middleware runs before emission to remove or mask sensitive fields. CI tests validate that events do not include disallowed fields. If sanitizer detects an unmaskable secret, event emission is blocked and an error is logged to the local operations channel; a configurable redaction policy controls hashing vs masking.
   * What happens under burst/high-throughput conditions that exceed logging capacity?
     - The system prioritizes critical audit events (admin actions, lockouts, password resets) for guaranteed delivery. Non-critical or debug events may be sampled or batched. Metrics surface sampling rates and dropped event counts; configuration allows tuning of sampling and queue sizes.
